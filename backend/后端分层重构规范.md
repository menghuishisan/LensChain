# 链镜平台后端分层重构规范

> **适用范围：** `backend/` 目录及其所有子目录  
> **文档目的：** 为后续后端各层重构提供唯一执行标准，避免出现目录职责混乱、分层依赖倒置、接口与文档偏移、代码风格失控等问题  
> **优先级：** 当本规范与临时实现习惯冲突时，以本规范为准；当本规范与模块设计文档冲突时，以模块设计文档为准，并先更新本规范再重构代码

---

## 一、重构总原则

### 1.1 文档优先，代码服从文档

1. 后端重构必须以 `docs/modules/` 下对应模块文档为唯一真相源。
2. 重构可以调整目录、职责边界、代码组织方式，但**不得改变文档定义的业务语义、表结构语义、接口语义**。
3. 如发现现有代码、迁移、配置与文档不一致，必须先确认文档，再修正代码；禁止以现有代码反推“正确规范”。

### 1.2 重构目标

后端重构的目标不是“把代码写出来”，而是同时满足以下四点：

1. **目录职责清晰**：每层知道自己做什么、不做什么。
2. **依赖方向稳定**：严格单向依赖，不允许反向调用。
3. **模块边界清晰**：模块内高内聚，模块间低耦合。
4. **后续可持续迭代**：新增功能时不需要破坏原有分层。

### 1.3 基础层优先原则

1. `internal/pkg/` 与 `internal/middleware/` 属于后端基础层，必须先于上层功能完成规范化建设。
2. 基础层不是“够用就行”，而必须满足以下要求：
   - 能覆盖当前项目真实需要的公共能力
   - 能被后续各层稳定复用
   - 具备明确注释和清晰边界
   - 在安全性、错误处理、边界条件上达到可上线标准
3. 后续任意一层在实现过程中，如果需要调用某项基础能力，而 `pkg` 或 `middleware` 中尚未提供，则**必须先补齐基础层，再继续上层实现**。
4. 禁止在 `handler`、`service`、`repository` 中临时补“半公共能力”来绕过基础层建设。
5. 基础层一旦沉淀为公共能力，后续各层必须优先复用，禁止重复造轮子。

### 1.4 重构时禁止事项

1. 禁止为图省事把业务逻辑塞进 `handler`、`repository`、`middleware`。
2. 禁止跨层绕过调用，例如 `handler` 直接查数据库、`service` 直接写 SQL。
3. 禁止跨模块直接依赖具体实现。
4. 禁止把临时脚本、测试用逻辑、初始化杂项混进业务目录。
5. 禁止为了通过编译随意新增文档未定义字段、接口、状态值、错误码。

## 二、后端标准目录

后端标准目录如下：

```text
backend/
├── cmd/
│   └── server/
│       ├── main.go
│       ├── init_auth.go
│       ├── init_school.go
│       ├── init_course.go
│       └── init_experiment.go
├── internal/
│   ├── config/
│   ├── router/
│   ├── handler/
│   ├── service/
│   ├── repository/
│   ├── model/
│   │   ├── entity/
│   │   ├── dto/
│   │   └── enum/
│   ├── middleware/
│   └── pkg/
├── configs/
├── migrations/
└── pkg/
```

### 2.1 顶层目录职责

| 目录 | 职责 | 允许放置 | 禁止放置 |
|------|------|----------|----------|
| `cmd/` | 程序入口与模块装配 | `main.go`、依赖注入初始化文件 | 业务逻辑、SQL、DTO |
| `internal/config/` | 配置结构定义与加载逻辑 | 配置结构体、加载器、环境变量覆盖规则 | 业务逻辑 |
| `internal/router/` | 路由注册层 | 路由组、路径、HTTP 方法、中间件绑定 | 业务逻辑、数据库访问 |
| `internal/handler/` | HTTP 处理层 | 参数绑定、参数校验、调用 service、返回响应 | SQL、复杂业务规则 |
| `internal/service/` | 业务逻辑层 | 业务规则、事务编排、跨仓储协调 | `*gin.Context`、直接写 SQL |
| `internal/repository/` | 数据访问层 | GORM/SQL 查询、CRUD、分页查询 | 业务流程判断、跨模块编排 |
| `internal/model/entity/` | 实体层 | 表映射结构体 | 业务逻辑 |
| `internal/model/dto/` | DTO 层 | 请求/响应结构体 | 引用 repository、复杂转换逻辑 |
| `internal/model/enum/` | 枚举层 | 状态常量、文本映射 | 依赖其他层 |
| `internal/middleware/` | 中间件层 | 鉴权、租户注入、日志、限流、恢复 | 业务流程 |
| `internal/pkg/` | 内部公共基础能力 | 响应封装、错误码、校验器、工具组件 | 业务模块实现 |
| `configs/` | 配置文件 | `config.yaml`、`config.example.yaml` | Go 业务代码 |
| `migrations/` | SQL 迁移 | 按模块组织的 up/down SQL | Go 代码、脚本逻辑 |
| `pkg/` | 可导出公共包 | 给外部服务复用的稳定公共能力 | 内部业务实现 |

## 三、标准调用链

后端允许的调用关系如下：

```text
main -> router -> handler -> service -> repository -> model
                 ↓         ↓          ↓
            middleware     pkg        pkg
```

### 3.1 强约束

1. `main` 只做启动、加载配置、初始化基础设施、依赖注入。
2. `router` 只做路由装配，不做参数校验和业务处理。
3. `handler` 只处理 HTTP 输入输出，不做持久化操作。
4. `service` 只处理业务，不依赖 HTTP 框架对象。
5. `repository` 只处理数据读写，不做业务决策。
6. `model` 只表达数据结构，不依赖上层。

### 3.2 明确禁止的依赖

1. 禁止 `handler -> repository`
2. 禁止 `router -> service`
3. 禁止 `repository -> service`
4. 禁止 `service -> handler`
5. 禁止 `entity -> dto`
6. 禁止 `middleware -> repository/service` 承担业务流程

### 3.3 跨模块调用规则

1. 跨模块业务调用只能发生在 `service` 层。
2. 跨模块调用必须优先依赖接口，而不是依赖具体实现。
3. 跨模块读取必须有明确边界，禁止把对方模块当作“公共仓库”随意查询。
4. 聚合层模块可以读取基础层、业务层数据，但不得打破单向依赖。

## 四、各层详细规范

## 4.1 `cmd/server/` 入口层规范

### 职责

1. 加载配置
2. 初始化日志、数据库、缓存、对象存储、消息队列等基础设施
3. 初始化雪花 ID、定时任务、WebSocket 管理器等系统组件
4. 组装模块依赖
5. 启动 HTTP 服务并处理优雅退出

### 文件组织

1. `main.go` 只保留启动主流程。
2. 每个业务模块一个 `init_xxx.go` 文件，负责模块依赖组装。
3. 模块初始化文件只做依赖注入，不承载业务逻辑。

### 禁止事项

1. 禁止在 `main.go` 中直接写业务逻辑。
2. 禁止在 `init_xxx.go` 中写接口实现。
3. 禁止在入口层硬编码业务常量和 SQL。

---

## 4.2 `internal/config/` 配置层规范

### 职责

1. 定义后端配置结构体
2. 加载 YAML 配置
3. 处理环境变量覆盖
4. 向系统其他层提供只读配置访问入口

### 配置层必须满足的要求

1. `config.go` 中每个配置结构体、每个字段必须有中文注释。
2. `configs/config.example.yaml` 必须与 `internal/config/config.go` 的字段完全对齐。
3. `configs/config.yaml` 必须与 `config.example.yaml` 保持同结构。
4. 新增配置项时，必须同时更新：
   - `internal/config/config.go`
   - `configs/config.example.yaml`
   - `configs/config.yaml`
5. 环境变量覆盖命名必须与 `mapstructure` 一致。

### 配置项设计要求

1. 配置名使用小写下划线风格。
2. Duration 类型在 YAML 中统一写字符串，例如 `"10s"`、`"1h"`。
3. 敏感配置必须预留占位值，不在示例文件中写真实生产密钥。
4. 示例配置必须标明字段用途，避免靠猜配置。

### 禁止事项

1. 禁止把业务配置散落在 `os.Getenv` 的临时读取中。
2. 禁止出现“配置文件有字段但结构体没有”或“结构体有字段但示例配置没有”的情况。
3. 禁止在业务代码里重复拼装另一套配置结构。

---

## 4.3 `internal/router/` 路由层规范

### 职责

1. 创建路由组
2. 注册 API 路径
3. 绑定中间件
4. 绑定 handler 方法

### 规范要求

1. 路由层文件按模块拆分，例如 `auth.go`、`course.go`。
2. API 基础前缀统一为 `/api/v1`。
3. 路径命名统一使用小写连字符和名词复数风格。
4. 路由文件必须能让人一眼看清模块 API 全貌。

### 禁止事项

1. 禁止在路由层解析参数。
2. 禁止在路由层做权限判断细节。
3. 禁止在路由层调用 service。

---

## 4.4 `internal/handler/` 处理层规范

### 职责

1. 接收 HTTP 请求
2. 绑定路径参数、查询参数、请求体
3. 做输入合法性校验
4. 调用 service
5. 统一输出响应

### 规范要求

1. handler 方法命名统一采用动作语义：
   - `Create`
   - `Update`
   - `Delete`
   - `Get`
   - `List`
2. 复杂模块按功能域拆分文件，例如 `assignment.go`、`discussion.go`。
3. handler 只接收 DTO，不直接操作 entity。
4. 返回结构统一走响应封装工具。
5. 所有公共导出函数必须写中文注释。

### 允许的逻辑

1. 参数绑定失败处理
2. 参数格式校验
3. 基础分页参数归一化
4. 将请求上下文转换为 service 需要的上下文信息

### 禁止事项

1. 禁止直接调用 repository。
2. 禁止直接开启事务。
3. 禁止写复杂业务规则，例如状态流转、成绩计算、权限矩阵判断。

---

## 4.5 `internal/service/` 业务层规范

### 职责

1. 实现核心业务逻辑
2. 执行业务规则校验
3. 组织多个 repository 调用
4. 编排事务
5. 处理跨模块接口调用

### 规范要求

1. service 不得依赖 `*gin.Context`。
2. service 输入输出优先使用 DTO 或明确的领域结构，不向上暴露 GORM 细节。
3. 一个 service 文件对应一个业务域，不对应整个大模块。
4. 事务边界原则上放在 service 层。
5. 涉及多个表写入的一致性操作必须由 service 统一编排。

### 事务规范

1. 单表简单 CRUD 可直接调用 repository。
2. 多表一致性写入必须在 service 层开启事务。
3. repository 不自行决定业务事务边界。
4. 事务函数内只做和当前事务相关的写入，不混入远程调用。

### 跨模块规范

1. service 间调用使用接口抽象。
2. 只读依赖尽量收敛为查询接口。
3. 禁止出现模块 A service 直接 new 模块 B repository 的情况。

### 禁止事项

1. 禁止拼接 HTTP 响应。
2. 禁止直接依赖 Gin。
3. 禁止把 SQL 写在 service 中。

---

## 4.6 `internal/repository/` 数据访问层规范

### 职责

1. 对数据库做 CRUD 操作
2. 构建查询条件
3. 实现分页、排序、筛选
4. 返回 entity 或明确的数据投影结构

### 规范要求

1. repository 必须按模块拆目录。
2. 查询方法命名要体现语义，例如：
   - `GetByID`
   - `GetByPhone`
   - `ListByCourseID`
   - `ExistsByCode`
   - `CountByStatus`
3. 简单数据访问逻辑留在 repository，不上浮到 service。
4. 复杂查询可以拆私有方法构建条件，但不得混入业务决策。

### 允许的逻辑

1. 软删除过滤
2. 多租户 `school_id` 条件拼装
3. 索引友好的查询条件构建
4. 数据存在性判断

### 禁止事项

1. 禁止写“如果状态是草稿则允许更新”这类业务逻辑。
2. 禁止调用其他模块 service。
3. 禁止感知 HTTP 请求对象。

---

## 4.7 `internal/model/` 模型层规范

### `entity/` 规范

1. 一个文件按模块或相关表分组，不要求一表一文件。
2. 结构体字段必须与数据库字段语义一致。
3. 结构体只做映射，不承载业务逻辑。
4. 可以保留极少量简单方法，例如 `TableName()`。

### `dto/` 规范

1. DTO 只表达接口输入输出。
2. 请求 DTO 要带校验标签。
3. 响应 DTO 要与 API 文档保持一致。
4. DTO 不要直接依赖 entity。

### `enum/` 规范

1. 所有状态值、类型值、模式值统一放在 `enum/`。
2. 枚举常量必须有中文注释。
3. 可以提供 `Text()` 或 `GetXXXText()` 方法。
4. 禁止在业务层到处散落魔法数字。

---

## 4.8 `internal/middleware/` 中间件层规范

### 职责

1. JWT 鉴权
2. 角色与权限校验入口
3. 多租户上下文注入
4. 日志链路处理
5. 限流、恢复、跨域等通用 HTTP 拦截

### 规范要求

1. 中间件只做横切关注点。
2. 中间件输出的信息应通过 context 传递给后续 handler/service。
3. 与业务强绑定的校验不应塞入通用中间件。
4. 中间件作为基础层组成部分，必须优先补齐项目通用的鉴权、租户、日志、恢复、限流、跨域等能力，保证上层无需重复实现。
5. 中间件实现必须显式考虑并处理安全边界、异常恢复、空值场景、并发场景和生命周期管理。

### 禁止事项

1. 禁止在中间件中直接落业务数据。
2. 禁止在中间件中编排模块业务流程。
3. 禁止把“某个具体接口的特殊判断”写成通用中间件。

---

## 4.9 `internal/pkg/` 基础公共层规范

### 职责

1. 提供后端内部复用的基础能力
2. 包括但不限于：
   - 统一响应封装
   - 错误码
   - 雪花 ID
   - 校验器
   - 数据库、缓存、消息队列、对象存储客户端封装

### 规范要求

1. `internal/pkg/` 只能承载通用能力，不承载业务模块语义。
2. 包设计要稳定、少依赖。
3. 公共能力必须有注释，尤其是对外暴露的函数、结构体、常量。
4. `internal/pkg/` 必须优先沉淀项目后续各层会反复使用的基础能力，例如错误码、响应封装、上下文传递、校验器、ID生成、数据库/缓存/消息队列/对象存储客户端、安全工具等。
5. 只要上层实现需要某种“可复用公共能力”，且该能力具有跨模块复用价值，就必须先沉淀到 `internal/pkg/`，不能散落在单个模块里。
6. 公共能力必须以“完整可用”为标准，而不是以“当前勉强能调用”为标准；必须考虑错误处理、默认值、安全性、并发安全、资源释放和注释完整性。

### 禁止事项

1. 禁止在 `pkg` 中写课程、实验、竞赛等模块逻辑。
2. 禁止为了复用而强行抽取实际上只服务一个模块的代码。
3. 禁止上层因为 `pkg` 缺能力，就在本层临时复制一份实现。

---

## 五、迁移层规范

### 5.1 迁移组织方式

1. 迁移文件按模块组织，不按单表拆文件。
2. 当前标准为 8 个模块对应 8 组 up/down 文件。
3. 文件命名保持序号稳定，例如：
   - `001_auth.up.sql`
   - `001_auth.down.sql`

### 5.2 迁移内容要求

1. 迁移必须严格对齐各模块 `02-数据库设计.md`。
2. 字段、索引、默认值、外键、软删除字段都要与文档一致。
3. 避免使用数据库关键字作为字段名；如存在该问题，应优先修改文档和迁移设计，不使用双引号硬保留。
4. 模块间外键要按依赖顺序补齐。

### 5.3 注释要求

1. 每个迁移文件必须有模块头注释。
2. 每张表创建前建议有表级职责说明注释。
3. 复杂字段、约束或补充外键建议加中文说明。

---

## 六、注释规范

### 6.1 总体要求

1. 注释统一使用中文。
2. 公共导出函数、结构体、接口必须有注释。
3. 注释必须解释“为什么存在”或“该字段/方法负责什么”，不是机械翻译代码。

### 6.2 配置层注释

1. `config.go` 中每个配置字段都要说明用途。
2. `config.example.yaml` 中每个字段都要说明配置含义和常见取值。
3. 后续新增配置项时，代码注释和 YAML 注释必须同时补齐。

### 6.3 SQL 注释

1. 迁移文件要标明模块来源和文档依据。
2. 关键字段、特殊约束建议补中文注释。

---

## 七、命名与拆分规范

### 7.1 Go 命名

1. 文件名：小写下划线
2. 包名：全小写
3. 结构体：大驼峰
4. 接口：大驼峰，不加 `I`
5. 变量：小驼峰
6. 常量：大驼峰常量名

### 7.2 文件拆分原则

1. 单文件建议控制在 500-800 行以内。
2. 超过 800 行必须评估是否按功能域拆分。
3. 拆分依据是职责，不是为了“每个函数单独一个文件”。
4. 大模块按功能域拆分，小模块允许单文件。

### 7.3 推荐拆分方式

1. `handler/auth/login.go`、`handler/auth/user.go`
2. `service/course/course.go`、`service/course/assignment.go`
3. `repository/experiment/template_repo.go`、`repository/experiment/instance_repo.go`

---

## 八、错误处理规范

1. 所有错误必须处理，不允许无声吞错。
2. 业务错误统一使用错误码体系。
3. 初始化阶段允许 `Fatal` 或退出，业务处理阶段禁止 `panic`。
4. 下层返回原始错误时，上层要根据边界决定是否包装为业务错误。

---

## 九、响应与接口对齐规范

1. HTTP 返回统一使用项目统一响应结构。
2. JSON 中的雪花 ID 一律按字符串输出。
3. 时间统一使用 ISO 8601。
4. 分页参数统一使用 `page`、`page_size`。
5. API 处理逻辑必须与模块 `03-API接口设计.md` 完全一致。

---

## 十、重构执行顺序

后续后端重构建议按以下顺序进行：

1. `migrations/` 与数据库设计文档对齐
2. `internal/config/` 与 `configs/` 对齐
3. `internal/pkg/` 与 `internal/middleware/` 基础能力补齐并固化
4. `model/entity`、`model/dto`、`model/enum`
5. `repository`
6. `service`
7. `handler`
8. `router`
9. `cmd/server`

说明：

1. 下层先稳定，上层再接入。
2. 同一轮重构尽量不要跨多个层同时大改。
3. 每一层完成后都应做本层自检，再进入下一层。
4. 如果上层重构过程中发现基础能力缺失，必须回到第 3 步先补齐 `internal/pkg/` 或 `internal/middleware/`。

---

## 十一、重构完成判定清单

完成某一层重构时，至少满足以下检查项：

### 11.1 通用检查

1. 是否符合目录职责边界
2. 是否符合单向依赖
3. 是否与模块文档一致
4. 是否补齐中文注释
5. 是否通过 `gofmt`

### 11.2 配置层检查

1. `config.go` 与 `config.example.yaml` 是否一一对应
2. `config.yaml` 是否同步更新
3. 是否所有字段都有用途说明

### 11.3 基础层检查

1. `internal/pkg/` 是否已经覆盖当前项目所需的公共能力
2. `internal/middleware/` 是否已经覆盖当前项目所需的横切能力
3. 上层是否仍存在重复实现的公共能力
4. 公共能力是否具备注释、错误处理和安全边界
5. 是否存在基础层缺失却在上层临时实现的情况

### 11.4 模型层检查

1. entity 是否与数据库设计一致
2. DTO 是否与 API 设计一致
3. 枚举是否收敛到 `enum/`

### 11.5 仓储层检查

1. 是否只做数据访问
2. 是否没有业务判断
3. 是否没有跨层依赖

### 11.6 服务层检查

1. 是否不依赖 Gin
2. 是否统一承载业务规则
3. 是否正确管理事务边界

### 11.7 处理层检查

1. 是否只做参数绑定、校验、调用 service、返回响应
2. 是否没有直接操作数据库

---

## 十二、最终结论

后续所有后端重构都以一句话为判断标准：

> **代码是否仍然忠实表达了文档定义的业务，并且严格待在它所属的那一层。**

只要有任意一项答案是否定的，就说明这次重构还没有达标。
