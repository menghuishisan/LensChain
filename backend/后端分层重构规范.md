# 链镜平台后端分层重构规范

> **适用范围：** `backend/` 目录及其所有子目录
> **文档目的：** 作为后续后端重构、补齐、审查、修复的统一执行规范，避免目录职责混乱、分层依赖倒置、文档契约漂移、重复实现、注释失真等问题
> **执行原则：** 文档驱动开发，目录职责清晰，严格单向依赖，统一实现，不保留新旧两套并存

---

## 一、总原则

### 1.1 文档优先，代码服从文档

1. 后端开发与重构必须以 `docs/modules/` 下对应模块文档为唯一真相源。
2. 重点对齐以下文档：
   - `01-功能需求说明.md`
   - `02-数据库设计.md`
   - `03-API接口设计.md`
   - `05-验收标准.md`
3. 如果文档与现有代码冲突，以文档为准。
4. 如果文档与当前真实需求冲突，必须先改文档，再改代码，禁止反过来。
5. 不允许根据现有代码倒推“规范”，也不允许用“为了兼容旧实现”作为偏离文档的理由。

### 1.2 重构目标

后端重构的目标不是“代码能跑”，而是同时满足以下要求：

1. **目录职责清晰**：每一层只做自己该做的事。
2. **依赖方向稳定**：严格单向依赖，不允许跨层绕过。
3. **模块边界清晰**：模块内高内聚，模块间低耦合。
4. **实现完整可验收**：当前层承载的能力必须补齐到位，满足文档和验收标准。
5. **统一且可持续演进**：能统一的必须统一，不留下新旧两种表达或两套实现。

### 1.3 当前层优先原则

1. 当正在重构某一层时，只聚焦该层职责是否正确、是否完整。
2. 如果当前层需要依赖下层某项能力，必须先检查下层是否已有实现：
   - 已有则直接复用
   - 没有则先去下层按规范补齐
   - 补齐后再回到当前层接入
3. 禁止不检查下层能力就直接在当前层重复实现一套。
4. 禁止为了兼容上层旧代码而污染当前层设计。
5. 当前层必须先做到干净、统一、职责明确，后续上层再按规范接入。

### 1.4 统一实现原则

1. 不允许保留新旧两套并行实现。
2. 不允许同一语义出现两种 DTO 表达、两种状态体系、两种公共能力封装。
3. 发现旧实现、冗余 helper、过渡代码、兼容分支时，必须评估并及时清理。
4. 只要存在统一枚举、统一错误码、统一响应结构、统一基础能力，所有上层必须统一使用。
5. 禁止继续写魔法数字、魔法字符串来绕过已有统一定义。

### 1.5 禁止事项

1. 禁止为了编译通过去做不符合规范的兼容。
2. 禁止私自新增文档未定义的字段、接口、状态、行为。
3. 禁止把本该在基础层的能力散落到业务层。
4. 禁止把本该在 `service` 的业务逻辑下沉到 `handler` 或 `repository`。
5. 禁止把本该在 `handler` 的参数处理上推到 `service`。
6. 禁止跳过逐文件审查直接宣布某层完成。
7. 禁止留下一堆职责不清、仅用于绕过设计问题的辅助文件。

## 二、标准目录与职责边界

后端标准目录如下：

```text
backend/
├── cmd/
│   └── server/
│       ├── main.go
│       ├── init_auth.go
│       ├── init_school.go
│       ├── init_course.go
│       ├── init_experiment.go
│       ├── init_ctf.go
│       ├── init_grade.go
│       ├── init_notification.go
│       └── init_system.go
├── internal/
│   ├── config/
│   ├── router/
│   ├── handler/
│   ├── service/
│   ├── repository/
│   ├── model/
│   │   ├── entity/
│   │   ├── dto/
│   │   └── enum/
│   ├── middleware/
│   └── pkg/
├── configs/
├── migrations/
└── pkg/
```

### 2.1 顶层目录职责表

| 目录 | 职责 | 允许 | 禁止 |
|------|------|------|------|
| `cmd/` | 程序入口与模块装配 | 配置加载、基础设施初始化、依赖注入、服务启动 | 业务逻辑、SQL、HTTP 处理逻辑 |
| `internal/config/` | 配置结构定义与加载 | 配置结构体、加载逻辑、环境变量覆盖 | 业务逻辑 |
| `internal/router/` | 路由注册层 | 路由组、路径、中间件绑定、handler 绑定 | 参数校验、业务逻辑、数据库访问 |
| `internal/handler/` | HTTP 处理层 | 参数绑定、基础校验、调用 service、返回响应 | 直接调 repository、复杂业务逻辑、SQL |
| `internal/service/` | 业务逻辑层 | 业务规则、事务编排、跨仓储协调、跨模块接口调用 | `*gin.Context`、HTTP 响应拼装、直接写 SQL |
| `internal/repository/` | 数据访问层 | CRUD、查询构造、分页、排序、投影查询 | 业务流程判断、事务编排、调用 service |
| `internal/model/entity/` | 数据库映射层 | 表映射结构体、标签、极少量简单辅助 | 业务逻辑、复杂方法 |
| `internal/model/dto/` | 数据传输对象层 | 请求/响应结构、校验标签、分页 DTO | 引用 entity、业务逻辑 |
| `internal/model/enum/` | 枚举常量层 | 状态、类型、级别、文本映射 | 引用其他层、业务实现 |
| `internal/middleware/` | HTTP 中间件层 | 鉴权、租户注入、日志、限流、恢复、RBAC | 模块业务逻辑 |
| `internal/pkg/` | 内部公共基础层 | 错误码、响应封装、校验器、ID、时间、安全、客户端封装等 | 具体业务模块实现 |
| `configs/` | 配置文件目录 | YAML 配置与示例配置 | Go 业务代码 |
| `migrations/` | SQL 迁移目录 | 按模块组织的 up/down SQL | Go 代码、临时脚本 |
| `pkg/` | 可导出公共包 | 提供给其他服务复用的稳定能力 | 内部业务实现 |

## 三、标准调用链与依赖规则

后端允许的调用关系如下：

```text
main -> router -> handler -> service -> repository -> model
                 ↓         ↓          ↓
            middleware     pkg        pkg
```

### 3.1 强约束

1. `main` 只做启动、装配、初始化。
2. `router` 只做路径注册和中间件绑定。
3. `handler` 只做 HTTP 输入输出处理。
4. `service` 只做业务逻辑与事务编排。
5. `repository` 只做数据访问。
6. `model` 只表达数据结构。

### 3.2 明确禁止的依赖

1. 禁止 `handler -> repository`
2. 禁止 `router -> service`
3. 禁止 `repository -> service`
4. 禁止 `service -> handler`
5. 禁止 `model -> handler/service/repository`
6. 禁止 `middleware` 承担具体模块业务流程

### 3.3 跨模块调用规则

1. 跨模块业务调用只能发生在 `service` 层。
2. 跨模块调用必须通过接口解耦，不直接依赖具体实现。
3. 跨模块读取也必须有明确边界，不能把其他模块当成“公共仓库”。
4. 聚合层模块可以读取下层数据，但不得打破单向依赖。
5. 不允许因为临时方便就在本模块里直接 new 其他模块 repository 或复制其逻辑。

### 3.4 复用规则

1. 如果 `internal/pkg/` 已有通用能力，必须复用，不能在业务层再写一套。
2. 如果 `repository` 已有对应查询或写入能力，必须优先复用，不能在 `service` 重复实现。
3. 如果 `enum` 已有状态常量，必须统一使用，不能继续写魔法数字。
4. 如果 `errcode` 已有错误码，必须统一使用，不能自己手写错误文本替代。

## 四、各层详细规范

## 4.1 `cmd/server/` 入口层规范

### 职责

1. 加载配置
2. 初始化数据库、缓存、对象存储、消息队列等基础设施
3. 初始化日志、ID 生成器、全局组件
4. 组装各模块依赖
5. 启动 HTTP 服务并处理优雅退出

### 文件组织要求

1. `main.go` 只保留主启动流程。
2. 每个业务模块一个 `init_xxx.go` 文件，用于依赖注入与模块装配。
3. 模块初始化文件只做装配，不写业务实现。

### 禁止事项

1. 禁止在入口层写业务逻辑。
2. 禁止在入口层写 SQL。
3. 禁止在入口层硬编码业务规则、状态常量和模块判断。

---

## 4.2 `internal/config/` 配置层规范

### 职责

1. 定义配置结构体
2. 加载 YAML 配置
3. 处理环境变量覆盖
4. 向系统其他层提供只读配置

### 必须满足的要求

1. `config.go` 中每个配置结构体和每个字段都必须有中文注释。
2. `configs/config.example.yaml` 必须与配置结构体字段完全对齐。
3. 示例配置中的每个字段都要有中文说明，明确用途和常见取值。
4. 新增配置项时，必须同步更新：
   - `internal/config/`
   - `configs/config.example.yaml`
   - `configs/config.yaml`（如果存在）
5. 配置项必须完整覆盖后端运行所需，不允许上层代码再零散读取 `os.Getenv` 拼另一套配置。

### 禁止事项

1. 禁止出现“结构体有字段但示例配置没有”的情况。
2. 禁止出现“示例配置有字段但代码结构体没有”的情况。
3. 禁止把模块配置分散写进业务代码。

---

## 4.3 `internal/router/` 路由层规范

### 职责

1. 创建路由组
2. 注册 API 路径
3. 绑定中间件
4. 绑定 handler 方法

### 规范要求

1. 路由层文件按模块拆分，例如 `auth.go`、`course.go`。
2. 路径命名统一为小写连字符、名词复数风格。
3. 路由层必须清楚体现接口全貌、权限绑定和模块边界。
4. 路由层可以绑定通用权限中间件，但不处理具体业务判断。

### 禁止事项

1. 禁止在路由层解析参数。
2. 禁止在路由层做参数校验。
3. 禁止在路由层写业务逻辑。
4. 禁止在路由层直接调用 service 或 repository。

---

## 4.4 `internal/handler/` 处理层规范

### 职责

1. 接收 HTTP 请求
2. 绑定路径参数、查询参数、请求体
3. 做输入合法性校验
4. 调用 service
5. 返回统一响应

### 规范要求

1. handler 可以使用 `*gin.Context`，但只能停留在 HTTP 层。
2. handler 输入输出以 DTO 为主，不直接操作 entity。
3. handler 方法命名应统一、清晰、语义明确。
4. handler 只做参数处理、service 调用和响应输出，不下沉业务规则。
5. 分页、绑定错误、路径参数转换等通用能力要优先复用基础层能力。

### 允许的逻辑

1. 参数绑定失败处理
2. 基础格式校验
3. 分页参数归一化
4. 将请求上下文信息转换为 service 上下文

### 禁止事项

1. 禁止直接调用 repository。
2. 禁止开启事务。
3. 禁止写复杂业务规则，例如状态流转、成绩计算、流程编排、跨表一致性逻辑。
4. 禁止直接拼装 SQL 或访问数据库。

---

## 4.5 `internal/service/` 业务层规范

### 职责

1. 实现核心业务逻辑
2. 执行业务规则校验
3. 组织多个 repository 调用
4. 编排事务
5. 通过接口完成跨模块协作

### 规范要求

1. `service` 不得依赖 `*gin.Context` 或任何 HTTP 类型。
2. `service` 必须成为业务规则的唯一收敛层。
3. 一个 service 文件应对应一个功能域，而不是整个模块所有功能混在一起。
4. 多表写入一致性必须在 `service` 层编排事务。
5. `service` 需要调用其他模块能力时，必须通过接口解耦。

### 允许的逻辑

1. 状态流转判断
2. 权限补充校验
3. 业务参数归一
4. 事务编排
5. 聚合多个 repository 的结果生成响应 DTO

### 禁止事项

1. 禁止依赖 Gin。
2. 禁止直接写 SQL。
3. 禁止为了省事越过 repository 直接操作数据库。
4. 禁止直接依赖其他模块 service 的具体结构体实现。

---

## 4.6 `internal/repository/` 数据访问层规范

### 职责

1. 实现 CRUD
2. 构造查询条件
3. 处理分页、排序、筛选
4. 返回 entity 或明确的只读投影结构

### 规范要求

1. repository 必须按模块拆目录。
2. 方法命名必须体现清晰语义，如 `GetByID`、`ListByCourseID`、`ExistsByCode`、`CountByStatus`。
3. repository 负责数据访问，不负责业务决策。
4. 若项目既有风格是“一个功能一个明确实现”，则必须统一保持，不引入另一套泛型抽象风格。

### 允许的逻辑

1. 软删除过滤
2. 租户条件拼装
3. 查询条件构造
4. 数据存在性判断
5. 批量更新与批量写入

### 禁止事项

1. 禁止写业务规则，如“课程已归档不允许修改”。
2. 禁止调用 service。
3. 禁止处理 HTTP 请求对象。
4. 禁止把跨表事务编排塞进 repository。

---

## 4.7 `internal/model/entity/` 实体层规范

### 职责

1. 表映射结构体定义
2. 字段与标签定义
3. 极少量简单辅助，如 `TableName()`

### 规范要求

1. 字段、类型、可空性、默认值语义必须与迁移文件和数据库设计文档一致。
2. 不允许私自增减文档未定义字段。
3. 同一个模块相关表可以按功能聚合在一个文件中，不要求一表一文件。
4. entity 不承载复杂业务逻辑。

### 禁止事项

1. 禁止写复杂方法。
2. 禁止依赖其他层。
3. 禁止为了兼容上层旧实现修改数据库语义。

---

## 4.8 `internal/model/dto/` DTO 层规范

### 职责

1. 请求 DTO
2. 响应 DTO
3. 列表项 DTO
4. 分页 DTO

### 规范要求

1. DTO 必须服务于 API 文档契约。
2. 请求字段、响应结构、分页结构必须与 API 文档一致。
3. 请求 DTO 带必要校验标签。
4. DTO 必须保持纯数据结构，不承载业务逻辑。
5. 同一语义的字段和结构必须统一表达，不能一会儿 `list` 一会儿 `items`。

### 禁止事项

1. 禁止引用 entity。
2. 禁止夹带 service/repository 逻辑。
3. 禁止保留两套等价 DTO 结构长期并存。

---

## 4.9 `internal/model/enum/` 枚举层规范

### 职责

1. 定义状态、类型、级别、角色等枚举常量
2. 提供枚举文本映射方法

### 规范要求

1. 所有需要统一表达的状态、类型值必须收敛到 `enum/`。
2. 枚举定义要有清晰中文说明。
3. 可以提供 `Text()`、`GetXXXText()` 等辅助方法。
4. 一旦某个语义已有统一枚举，所有上层必须统一使用。

### 禁止事项

1. 禁止引用其他层。
2. 禁止保留两套等价常量体系。
3. 禁止在业务层到处散落魔法数字。
4. 文档没有的枚举不要私自添加。

---

## 4.10 `internal/middleware/` 中间件层规范

### 职责

1. JWT 鉴权
2. 角色权限与 RBAC 入口
3. 多租户上下文注入
4. 请求日志、链路追踪
5. 限流、恢复、跨域等横切能力

### 规范要求

1. 中间件只做横切关注点，不做模块业务流程。
2. 中间件输出的信息应通过上下文传递给后续层。
3. 中间件实现必须考虑安全边界、异常恢复、空值场景和并发安全。
4. 中间件作为基础层的一部分，必须优先补齐项目通用能力，避免上层重复实现。

### 禁止事项

1. 禁止在中间件中直接落业务数据。
2. 禁止在中间件中编排模块业务流程。
3. 禁止把某个具体接口的特殊业务判断写成通用中间件。

---

## 4.11 `internal/pkg/` 内部公共基础层规范

### 职责

1. 提供项目内部复用的基础能力
2. 包括但不限于：
   - 统一响应封装
   - 错误码
   - 雪花 ID
   - 校验器
   - 时间工具
   - 安全工具
   - 内容清洗
   - 上下文工具
   - 数据库、缓存、消息队列、对象存储客户端封装
   - 异步、短信、邮件等通用基础设施封装

### 规范要求

1. `internal/pkg/` 只能承载通用能力，不承载业务模块语义。
2. 只要某项能力具有跨模块复用价值，就应优先沉淀到 `internal/pkg/`。
3. 公共能力必须以“完整可用”为标准，不能只满足某一个调用点勉强能跑。
4. 公共能力必须考虑错误处理、安全性、默认值、资源释放、并发安全和注释完整性。
5. 上层一旦需要该能力，必须直接复用，不得再复制一份实现。

### 禁止事项

1. 禁止在 `pkg` 中写具体业务模块逻辑。
2. 禁止为了复用而强行抽取仅服务一个模块且没有共性价值的代码。
3. 禁止因为 `pkg` 缺能力，就在上层随便复制一套临时实现。

---

## 4.12 `migrations/` 迁移层规范

### 职责

1. 管理数据库结构迁移
2. 按模块维护 up/down SQL

### 规范要求

1. 迁移必须严格对齐模块数据库设计文档。
2. 迁移文件按模块组织，不按单表碎片化组织。
3. 每个模块可以拆为固定数量的迁移文件，但必须结构清晰、上下对应。
4. 字段、索引、约束、软删除字段、默认值必须与文档一致。
5. 遇到 PostgreSQL 关键字冲突，优先修正文档与迁移设计，不通过保留关键字硬顶。

### 注释要求

1. 每个迁移文件必须有模块头注释。
2. 复杂表、字段、约束应有中文说明。
3. 注释应说明本文件负责的模块范围和文档依据。

---

## 五、工作方法规范

### 5.1 开始某层前的必做步骤

开始处理某一层或某个模块前，必须按以下顺序执行：

1. 阅读该模块相关文档
2. 审查当前目录现有文件
3. 标出与文档、职责、统一性冲突的问题
4. 再开始修复、补齐或重建

### 5.2 逐文件审查原则

1. 必须逐文件审查，不能只抽查一部分文件。
2. 要确认当前目录下每个文件是否：
   - 符合目录职责
   - 与文档契约一致
   - 存在冗余或旧实现
   - 存在命名混乱或职责漂移
   - 缺失必要注释
3. 如果某一层已经严重偏离规范，可以直接删除后按正确方式重建，但重建后必须更统一、更干净。

### 5.3 完成一轮后的必查项

1. 是否符合目录职责边界
2. 是否符合单向依赖
3. 是否仍有重复实现
4. 是否仍有新旧两套实现并存
5. 是否仍有魔法数字、魔法字符串未统一
6. 是否仍有文档契约漂移
7. 注释是否达标
8. 当前层职责内需要的能力是否已经完整实现

---

## 六、注释规范

### 6.1 总体要求

1. 注释统一使用中文。
2. 公共导出函数、结构体、接口必须有注释。
3. 注释不能是废话，不能只是机械重复函数名。
4. 注释必须帮助后续开发人员理解项目结构、职责和业务约束。

### 6.2 文件头注释要求

每个重要文件开头应说明：

1. 文件属于哪个模块
2. 当前文件负责哪个功能域
3. 当前文件属于哪一层
4. 需要时标明主要对照的文档

例如：

1. 模块03 课程与教学：作业管理业务逻辑
2. 模块05 CTF竞赛：竞赛与题目数据访问层
3. 模块07 通知与消息：通知模板与发送偏好 DTO

### 6.3 实现注释要求

以下场景必须补充有意义的实现注释：

1. 复杂业务规则判断
2. 状态流转
3. 权限限制
4. 事务编排关键点
5. 与文档强绑定的特殊约束
6. 容易被误解的实现

注释要解释：

1. 为什么这样做
2. 这一段规则是什么
3. 它与文档或验收标准的关系

### 6.4 明确禁止的无效注释

以下注释视为不达标：

1. `Create 创建数据`
2. `Update 更新信息`
3. `Handle 处理请求`

这类注释没有提供理解价值，必须改成说明职责、场景和约束的注释。

### 6.5 模型与配置注释要求

1. entity 的复杂字段、特殊字段、文档契约特殊字段要有注释。
2. DTO 的特殊字段、权限差异字段、仅教师可见字段应有注释。
3. enum 的业务含义要清楚。
4. 配置结构体与示例配置中的每个字段都要有中文说明。

---

## 七、命名与拆分规范

### 7.1 Go 命名

1. 文件名：小写下划线
2. 包名：全小写
3. 结构体：大驼峰
4. 接口：大驼峰，不加 `I`
5. 变量：小驼峰
6. 枚举和常量：统一收敛到 `enum/`

### 7.2 文件拆分原则

1. 单文件建议控制在 500-800 行以内。
2. 超过 800 行必须评估是否按功能域拆分。
3. 拆分依据是职责和内聚性，不是为了机械拆文件。
4. 大模块按功能域拆分，小模块允许单文件。
5. helper 文件只有在职责明确、确实有存在价值时才允许保留。
6. 如果 helper 只是为绕过设计混乱而产生，应收敛或删除。

---

## 八、错误处理与接口契约规范

### 8.1 错误处理

1. 所有错误必须处理，不允许吞错。
2. 业务错误统一使用错误码体系。
3. 初始化阶段允许退出，业务处理阶段禁止 `panic`。
4. 上层根据边界决定是否将下层错误包装成业务错误。

### 8.2 接口契约

1. HTTP 返回统一使用项目统一响应结构。
2. JSON 中的雪花 ID 一律按字符串传输。
3. 时间统一使用 ISO 8601。
4. 分页参数统一为 `page`、`page_size`。
5. API 请求体、响应体、分页结构必须与 API 设计文档完全一致。

---

## 九、执行顺序建议

后续后端重构建议按以下顺序进行：

1. `migrations/`
2. `internal/config/` 与 `configs/`
3. `internal/pkg/` 与 `internal/middleware/`
4. `model/entity`
5. `model/enum`
6. `model/dto`
7. `repository`
8. `service`
9. `handler`
10. `router`
11. `cmd/server`

说明：

1. 下层先稳定，上层再接入。
2. 每一层完成后都要先做本层自检。
3. 如果上层重构过程中发现基础能力缺失，必须先回到 `pkg` 或 `middleware` 补齐。

---

## 十、完成判定清单

完成某一层重构时，至少满足以下检查项：

### 10.1 通用检查

1. 是否符合目录职责边界
2. 是否符合单向依赖
3. 是否与模块文档一致
4. 是否完成必要中文注释
5. 是否完成统一实现并清理旧实现
6. 是否通过格式化与基本自检

### 10.2 基础层检查

1. `internal/pkg/` 是否已覆盖当前项目需要的公共能力
2. `internal/middleware/` 是否已覆盖当前项目需要的横切能力
3. 上层是否仍存在重复实现的基础能力
4. 基础能力是否具备注释、错误处理和安全边界

### 10.3 模型层检查

1. entity 是否与数据库设计一致
2. enum 是否统一收敛并被上层正确使用
3. DTO 是否与 API 契约一致

### 10.4 仓储层检查

1. 是否只做数据访问
2. 是否没有业务判断
3. 是否没有跨层依赖

### 10.5 服务层检查

1. 是否不依赖 Gin
2. 是否统一承载业务规则
3. 是否正确管理事务边界
4. 是否通过接口完成跨模块依赖

### 10.6 处理层检查

1. 是否只做参数绑定、基础校验、service 调用、响应返回
2. 是否没有直接操作数据库
3. 是否没有承载复杂业务逻辑

---

## 十一、最终判断标准

后续所有后端重构，都以这句话作为最终判断标准：

> **代码是否忠实表达了文档定义的业务，并且严格待在它所属的那一层，同时没有留下重复实现、旧实现和不达标注释。**

只要其中任意一项答案是否定的，就说明这一层或这一轮重构还没有达标。
