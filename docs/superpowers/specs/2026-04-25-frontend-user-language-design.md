# 前端用户语义文案改造设计

## 背景

当前前端大量使用研发与设计阶段术语，例如“模块”“模板”“实例”“租户”“配置”“治理”“监控”“统计”等。  
这些表达虽然有助于内部沟通，但会直接向最终用户暴露系统设计方式，使页面更像开发后台或文档索引，而不是面向教师、学生、学校管理员和超级管理员的产品界面。

本次改造只处理用户可见文案，不调整路由、接口、类型命名、权限模型和页面结构。

## 目标

将前端所有直接面向用户的可见文案，从研发语义统一收敛为产品语义与用户语义。

改造后应满足以下要求：

- 文案从“系统结构导向”转为“用户任务导向”。
- 页面不暴露内部模块划分、对象建模和实现细节。
- 同一角色端的表达风格保持一致。
- 不修改不会暴露给用户的内部命名。

## 范围

本次改造包含：

- `frontend/AGENTS.md` 的前端表达规范补充
- 侧边栏导航文案
- 顶部栏角色文案、主标题与提示文案
- 页面主标题、副标题
- 卡片标题与说明
- 按钮、表格头、空状态、错误提示、成功提示中的高风险研发术语

本次改造不包含：

- 路由路径
- API 路径和 service 命名
- TypeScript 类型命名
- hooks、stores、内部变量名
- 后端接口或权限逻辑
- 页面信息架构重组

## 设计原则

### 1. 面向用户目标，而不是面向系统结构

用户界面应描述“用户要完成什么”，而不是“系统内部如何组织”。

示例：

- “课程教学”优于“课程模块”
- “实验运行”优于“实例治理”
- “平台运行”优于“系统运维模块”

### 2. 弱化内部对象名

当术语主要服务于研发建模而不服务于用户理解时，应替换为更自然的产品语言。

示例：

- “租户”优先替换为“学校”
- “实例”优先替换为“实验”或“运行记录”
- “模块”不应出现在用户界面

### 3. 保留必要专业词，但避免裸露实现感

部分场景中用户已能理解专业术语，例如教师维护“实验模板”、平台管理员查看“告警规则”。  
此类词汇可以保留或弱化，但不能直接拼接出内部设计语境，例如“模板治理”“实例生命周期”“模块08统计面板”。

### 4. 角色视角一致

- 学生端聚焦学习、实验、作业、竞赛、成绩、通知
- 教师端聚焦教学、内容准备、批改、实验组织、竞赛组织
- 学校管理员端聚焦学校服务、账号管理、资源安排、教学保障
- 超级管理员端聚焦平台服务、学校支持、平台安全、运行保障

## 高风险词汇处理规则

以下词汇默认视为高风险研发语义，除非处于平台级专业页面且用户本身就是管理员：

- 模块
- 租户
- 实例
- 生命周期
- 治理
- 运行时
- 配置项
- 内部接口
- 监控面板

推荐替换方向：

- 模块 -> 不展示，改为具体业务名
- 租户 -> 学校 / 学校空间
- 实例 -> 实验 / 实验记录 / 当前实验
- 生命周期 -> 运行过程 / 状态变化
- 治理 -> 管理 / 处理 / 运行支持
- 监控面板 -> 运行看板 / 运行概览
- 配置项 -> 设置内容 / 规则 / 平台设置

## 实施策略

### 第一层：统一入口

优先修改统一入口配置，收敛以下内容：

- 角色展示文本
- 侧边栏导航标题与说明
- 顶部栏主标题与辅助提示

### 第二层：高频业务组件

优先处理高频页面与跨角色复用组件中的用户可见文案，例如：

- 系统页总布局
- 用户管理面板
- 学校管理面板
- 通知面板
- 课程面板
- 实验面板
- 竞赛面板
- 成绩面板

### 第三层：状态文案

清理明显暴露设计实现的空状态、错误态和说明文案，例如：

- “该路由将在对应模块开发阶段按文档接入”
- “链镜正在读取 config_change_logs 历史”
- “当前实例尚未分配 SimEngine 会话”

上述文案应改为用户可理解的产品表达。

## 验收标准

- 用户界面不出现“模块01/02/03”等编号或模块表达。
- 导航说明不直接暴露内部对象建模。
- 页面标题不再像研发后台或设计文档索引。
- 空状态与错误文案不再引用内部表名、字段名、日志名或协议实现。
- 改造不影响路由、接口与现有页面功能。
