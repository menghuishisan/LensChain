## 一、模块概述

CTF竞赛模块是链镜平台的独立子系统，为区块链安全教育提供专业的竞赛平台。本模块支持两种竞赛模式：**解题赛（Jeopardy）** 和 **攻防对抗赛（Attack-Defense）**，聚焦区块链与智能合约安全领域，与平台实验环境模块（模块04）共享环境编排服务层。

### 核心设计理念

- **独立子系统**：竞赛独立于课程体系，不与课程关联，不回传成绩
- **两种赛制**：解题赛（个人/团队解题夺旗）和攻防对抗赛（链上战场实时攻防）
- **真实漏洞驱动**：题目基于真实智能合约漏洞，通过结构化漏洞描述 + 断言验证系统确保可解可验
- **链上验证**：智能合约安全题目通过链上状态断言验证，非简单字符串匹配
- **共享基础设施**：复用模块04的环境编排服务层（镜像库、K8s编排、容器管理、网络隔离）

### 系统定位

│              链镜平台                     │

│  ┌──────────┐  ┌──────────┐  ┌────────┐ │

│  └────┬─────┘  └────┬─────┘  └───┬────┘ │
│       │              │             │      │

│       │         │                         │
│       │    ┌────┴────────────┐            │

## 三、用户故事

### US-01：超级管理员创建平台竞赛

```
作为超级管理员，我希望创建平台级CTF竞赛，
设置竞赛类型、时间、参赛规则、题目配置，
以便全平台学生参与区块链安全竞赛。
```

**验收条件**：
- 可选择解题赛或攻防对抗赛模式
- 可设置竞赛时间、参赛人数上限、团队规模
- 可从题库选题或邀请教师出题
- 可配置计分规则、排行榜策略
- 竞赛创建后为草稿状态，发布后进入报名阶段

### US-02：教师创建竞赛题目

```
作为教师，我希望创建区块链安全竞赛题目，
通过漏洞转化工具将真实漏洞转化为CTF题目，
经过预验证确保题目可解后提交审核。
```

**验收条件**：
- 支持6种题目类型：Web安全、密码学、智能合约安全、链上分析、逆向、杂项
- 智能合约安全题目支持三条漏洞获取路径
- 题目必须通过6步预验证方可提交审核
- 审核通过后题目进入平台题库

### US-03：学生参加解题赛

```
作为学生，我希望报名参加CTF解题赛，
查看题目列表，启动独立链环境，
通过提交Flag或攻击交易来解题得分。
```

**验收条件**：
- 竞赛开始后所有题目全量可见
- 每道题目可启动独立环境（智能合约题目获得独立链）
- 支持三种Flag验证：静态Flag、动态Flag、链上状态验证
- 实时查看得分和排行榜
- 动态计分：解题人数越多分值越低

### US-04：学生参加攻防对抗赛

```
作为学生，我希望参加攻防对抗赛，
在回合制的链上战场中攻击对手、防守自身，
通过Token余额争夺最终排名。
```

**验收条件**：
- 每支队伍拥有独立队伍链和初始Token
- 回合制：攻击阶段 → 防守阶段 → 自动结算
- 攻击阶段可向其他队伍链提交攻击交易
- 防守阶段可提交补丁合约修复漏洞
- Token余额为核心排名依据
- 实时排行榜展示各队Token变化

### US-05：教师转化真实漏洞为题目

```
作为教师，我希望将真实的智能合约漏洞转化为CTF题目，
通过SWC导入、模板库或自定义方式创建题目，
并通过预验证确保题目的正确性。
```

**验收条件**：
- SWC Registry导入：选择SWC编号，自动生成合约和断言
- 参数化模板库：选择模板，调整参数生成题目变体
- 完全自定义：手动编写合约、初始化交易、验证断言
- 6步预验证：部署 → 提交PoC → 正向验证 → 反向验证
- 验证通过后方可提交审核

## 四、竞赛类型详解

### 4.1 解题赛（Jeopardy）

#### 基本规则

- 竞赛开始后所有题目全量发布，选手自由选择解题顺序
- 每道题目有基础分值，采用动态计分（解题人数越多分值越低）
- 支持个人赛和团队赛
- First Blood（首个解出）额外加分 +10%
- 无提示系统

#### 动态计分公式

```
score = max(min_score, base_score × decay_factor^(solve_count))
```

- `base_score`：题目基础分值（由难度决定）
- `min_score`：最低分值（默认为 base_score 的 20%）
- `decay_factor`：衰减因子（默认 0.95）
- `solve_count`：当前解题队伍数

#### 五级难度体系

| 难度 | 编码 | 基础分值范围 | 说明 |
|------|------|-------------|------|
| Warmup | 1 | 100-200 | 入门热身 |
| Easy | 2 | 200-400 | 基础题目 |
| Medium | 3 | 400-600 | 中等难度 |
| Hard | 4 | 600-800 | 高难度 |
| Insane | 5 | 800-1000 | 极限挑战 |

#### 六种题目类型

| 类型 | 编码 | 说明 | 环境需求 |
|------|------|------|----------|
| Web安全 | web | Web应用安全漏洞 | Web容器环境 |
| 密码学 | crypto | 密码学算法分析与破解 | 计算环境（可选） |
| 智能合约安全 | contract | 智能合约漏洞利用 | 独立链环境 |
| 链上分析 | blockchain | 链上数据分析与追踪 | 链节点 + 浏览器 |
| 逆向 | reverse | 二进制/字节码逆向分析 | 逆向工具环境 |
| 杂项 | misc | 综合类题目 | 按需配置 |

#### Flag验证三种方式

| 方式 | 适用场景 | 说明 |
|------|----------|------|
| 静态Flag | Web/Crypto/Reverse/Misc | 固定字符串，格式 `flag{...}` |
| 动态Flag | 需要防作弊的题目 | `flag{HMAC-SHA256(secret, team_id)}`，每队不同 |
| 链上状态验证 | 智能合约安全/链上分析 | 执行 verification_assertions 断言数组判定 |

#### 解题赛环境隔离

每支队伍的每道题目获得独立K8s Namespace：

```
Namespace: ctf-{competition_id}-{team_id}-{challenge_id}
├── 题目容器（链节点/Web服务/工具等）
└── 选手工具容器（按需）
```

#### 反作弊提交限流

- 每支队伍每道题每分钟最多提交 5 次
- 连续失败 10 次后触发 5 分钟冷却期
- 冷却期内该队伍该题目不可提交

### 4.2 攻防对抗赛（Attack-Defense）

#### 链上战场概念

攻防对抗赛采用"链上战场"模式，每支队伍拥有独立的区块链环境，通过攻击对手链上的漏洞合约窃取Token，同时防守自身合约不被攻破。

#### 两层链架构

```
┌─────────────────────────────────────┐
│           裁判链（Judge Chain）        │
│  部署：裁判合约、Token总账、结算逻辑    │
│  权限：仅裁判系统可写，选手只读         │
│  用途：Token转账结算、攻击记录、排名    │
├─────────────────────────────────────┤
│         队伍链（Team Chains）         │
│  Team A Chain: 漏洞合约 + Token副本   │
│  Team B Chain: 漏洞合约 + Token副本   │
│  Team C Chain: 漏洞合约 + Token副本   │
│  权限：本队可部署补丁，其他队可提交攻击  │
└─────────────────────────────────────┘
```

- **裁判链**：每个竞赛分组共享一条裁判链，部署裁判合约负责Token结算和攻击记录
- **队伍链**：每支队伍独立一条链，部署漏洞合约和Token副本

#### 回合制机制

每轮回合包含三个阶段：

| 阶段 | 时长（可配置） | 说明 |
|------|--------------|------|
| 攻击阶段 | 5-15 分钟 | 向其他队伍链提交攻击交易 |
| 防守阶段 | 5-15 分钟 | 提交补丁合约修复自身漏洞 |
| 结算阶段 | 自动（~30s） | 裁判系统自动结算Token转账 |

- 总回合数可配置：5-30 轮
- 攻击阶段和防守阶段时长独立配置

#### 攻击机制

- 攻击方向目标队伍链提交攻击交易
- 平台执行 verification_assertions 判定攻击是否成功
- 攻击成功：从目标队伍Token中扣除奖励，转入攻击方
- 漏洞衰减：同一漏洞被多队利用，后续攻击奖励递减

**漏洞衰减公式**：
```
attack_reward = base_reward × decay_factor^(exploit_count - 1)
```
- `base_reward`：漏洞基础奖励
- `decay_factor`：衰减因子（默认 0.8）
- `exploit_count`：该漏洞已被利用的次数

#### 防守机制

- 防守方提交补丁合约（修复后的合约代码）
- 平台验证补丁：
  1. **功能完整性检查**：原有功能测试用例全部通过
  2. **漏洞修复验证**：官方PoC不再成功（断言不再通过）
- 补丁验证通过后替换队伍链上的漏洞合约

#### Token经济与计分

| 计分项 | 规则 |
|--------|------|
| 初始Token | 每队初始 10000 Token |
| 攻击得分 | 窃取对手Token（经漏洞衰减计算） |
| 攻击奖励 | 额外获得窃取Token的 5% 作为奖励 |
| 防守奖励 | 每轮未被攻破 +50 Token |
| 首补丁奖励 | 首个修复某漏洞的队伍 +200 Token |
| First Blood | 首个成功攻击某漏洞 +10% 额外奖励 |
| 最终排名 | 以Token余额排序 |

#### 攻防赛环境架构

```
竞赛分组 Namespace: ctf-ad-{competition_id}-{group_id}
├── judge-chain（裁判链节点）
├── judge-service（裁判结算服务）
│
├── team-{team_id_1}-chain（队伍1链节点）
├── team-{team_id_1}-tools（队伍1工具容器）
│
├── team-{team_id_2}-chain（队伍2链节点）
├── team-{team_id_2}-tools（队伍2工具容器）
│
└── ...
```

## 五、真实漏洞转化系统

### 5.1 漏洞数据结构

每道智能合约安全题目的核心是结构化的漏洞描述：

```json
{
  "chain_config": {
    "chain_type": "evm",
    "chain_version": "london",
    "block_number": 0
  },
  "contracts": [
    {
      "name": "VulnerableBank",
      "source_code": "// Solidity source...",
      "abi": [],
      "bytecode": "0x...",
      "constructor_args": [],
      "deploy_order": 1
    }
  ],
  "setup_transactions": [
    {
      "from": "deployer",
      "to": "VulnerableBank",
      "function": "deposit",
      "args": [],
      "value": "10 ether"
    }
  ],
  "verification_assertions": [
    {
      "type": "balance_check",
      "target": "VulnerableBank",
      "operator": "lt",
      "value": "1 ether",
      "description": "合约ETH余额被掏空"
    }
  ]
}
```

### 5.2 七种断言类型

| 类型 | 说明 | 典型场景 |
|------|------|----------|
| balance_check | ETH余额检查 | 重入攻击掏空合约 |
| token_balance_check | ERC20代币余额检查 | 闪电贷攻击获利 |
| storage_check | 存储槽值检查 | 存储碰撞攻击 |
| owner_check | 合约owner变更检查 | 权限提升攻击 |
| event_check | 事件日志检查 | 特定事件是否触发 |
| code_check | 合约代码检查（extcodesize） | 自毁攻击后代码为空 |
| custom_script | 自定义脚本验证 | 复杂多步骤攻击 |

### 5.3 三条漏洞获取路径

#### 路径一：SWC Registry 自动导入

- 从 SWC（Smart Contract Weakness Classification）注册表导入标准漏洞分类
- 每个 SWC 条目自带示例代码，平台自动生成合约 + 断言
- 适合快速生成基础题库
- 导入流程：选择 SWC 编号 → 平台获取示例代码 → 自动生成 contracts[] + verification_assertions[] → 教师调整参数 → 预验证

#### 路径二：参数化模板库（~20个模板）

- 基于 Rekt/SlowMist 等真实安全事件的元数据，结合参数化模板
- 每个模板通过调整参数（资金量、合约逻辑变体、攻击路径）可生成多道题
- ~20个模板可生成50-60道不同题目
- 模板示例：
  - 重入攻击模板（单函数/跨函数/跨合约变体）
  - 闪电贷模板（不同DEX协议变体）
  - 整数溢出模板（加法/乘法/类型转换变体）
  - 权限控制模板（tx.origin/未初始化代理/delegatecall变体）
  - 预言机操纵模板（价格操纵/TWAP绕过变体）

#### 路径三：完全自定义

- 出题者手动编写合约源码、初始化交易、验证断言
- 适合高难度原创题目或复现特定真实事件（如 The DAO、Parity Wallet、Ronin Bridge 等）
- 完全自由度，不受模板限制

### 5.4 预验证流程（6步强制验证）

题目发布前必须通过完整预验证，确保题目可解且断言正确：

```
Step 1: 部署测试环境
  → 平台根据 chain_config 启动临时链（ganache/hardhat-node）
  → 部署 contracts[]（按 deploy_order 顺序）
  → 执行 setup_transactions[]

Step 2: 出题者提交官方 PoC
  → 出题者提交能利用漏洞的攻击交易/脚本

Step 3: 正向验证
  → 在测试环境中执行 PoC
  → 检查所有 verification_assertions[]
  → 全部断言通过 = 正向验证成功

Step 4: 反向验证
  → 重新部署干净环境（不执行 PoC）
  → 检查所有 verification_assertions[]
  → 全部断言失败 = 反向验证成功
  （确保断言不会在未攻击时误判为通过）

Step 5: 验证通过 → 题目可提交审核
Step 6: 验证失败 → 返回错误详情，出题者修改后重新验证
```

### 5.5 在两种赛制中的应用

| 应用场景 | 解题赛 | 攻防对抗赛 |
|----------|--------|-----------|
| 环境部署 | 每队每题独立链 | 每队独立队伍链 |
| 合约部署 | 部署漏洞合约 + 执行初始化交易 | 部署漏洞合约 + Token副本 |
| 攻击验证 | 选手提交攻击交易 → 断言验证 → 判定Flag | 攻击方提交交易 → 断言验证 → Token结算 |
| 防守验证 | 不适用 | 补丁合约 → 功能测试 + PoC失败验证 |
| 环境重置 | 选手可重置自己的环境 | 每轮结算后自动重置（保留补丁） |

## 六、竞赛生命周期

### 6.1 竞赛状态流转

```
创建竞赛
    │
    ▼
草稿(draft) ── 编辑/配置题目/设置规则
    │
    ▼ 发布
报名中(registration) ── 选手报名/组队
    │
    ▼ 竞赛开始时间到达
进行中(running)
    │
    ├── 解题赛：选手自由解题
    │
    ├── 攻防赛：回合制攻防
    │       ├── 攻击阶段(attacking)
    │       ├── 防守阶段(defending)
    │       └── 结算阶段(settling)
    │
    ▼ 竞赛结束时间到达
已结束(ended) ── 最终排名确认/环境回收
    │
    ▼ 归档
已归档(archived) ── 历史记录保留
```

### 6.2 竞赛时间线

| 阶段 | 说明 |
|------|------|
| 草稿期 | 管理员配置竞赛参数、选题、设置规则 |
| 报名期 | 选手浏览竞赛信息、报名、组队 |
| 竞赛期 | 选手参赛，实时计分和排行 |
| 结束期 | 最终排名确认，排行榜解冻，环境资源回收 |
| 归档期 | 竞赛数据归档，可查看历史记录 |

## 七、参赛与组队

### 7.1 参赛模式

| 模式 | 说明 |
|------|------|
| 个人赛 | 每人独立参赛，一人一队 |
| 自由组队 | 选手自行创建/加入团队 |
| 指定组队 | 管理员预先分配团队 |

### 7.2 团队规则

- 团队人数上限由竞赛配置决定（默认1-5人）
- 自由组队模式下，队长创建团队并生成邀请码
- 邀请码可邀请本校或外校学生加入（外校学生必须是平台注册用户）
- 竞赛开始后团队锁定，不可变更成员
- 指定组队模式下，管理员直接分配学生到团队

### 7.3 跨校参赛

- 平台级竞赛默认允许全平台学生参赛
- 校级竞赛可通过邀请码允许外校学生参赛
- 外校学生必须是平台注册用户，通过邀请码加入
- 不存在"临时账号"或"非平台用户"概念

---

## 八、排行榜与公告

### 8.1 排行榜

- 实时排行榜通过 WebSocket 推送更新
- 解题赛：按总分排序，同分按最后解题时间排序
- 攻防赛：按Token余额排序
- **最后一小时冻结**：竞赛最后一小时排行榜冻结，不再更新（增加悬念）
- 竞赛结束后解冻，显示最终排名

### 8.2 公告系统

- 支持全局公告和单题公告
- 公告通过 WebSocket 实时推送到所有参赛选手
- 公告类型：信息通知、题目勘误、规则说明
- 管理员/出题者可发布公告

### 9.1 竞赛管理规则

## 十二、平台竞赛链

### 12.1 链节点选择

竞赛使用平台镜像库中已有的链节点镜像，不fork主网：

| 场景 | 推荐镜像 | 说明 |
|------|----------|------|
| 解题赛-智能合约题 | ganache / hardhat-node | 轻量快速，适合单题独立环境 |
| 攻防赛-队伍链 | ganache / hardhat-node | 每队独立链，快速出块 |
| 攻防赛-裁判链 | geth (dev mode) | 裁判链需要更高可靠性 |
| 需要特定EVM版本 | geth / besu | 可配置特定硬分叉版本 |

### 12.2 链配置

- 所有竞赛链使用开发模式（instant mining / auto-mine）
- 预置充足ETH的测试账户
- 禁用P2P发现（单节点模式）
- 开放所有RPC API（教学/竞赛环境）
