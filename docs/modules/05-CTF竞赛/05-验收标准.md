# CTF竞赛模块 — 验收标准

> 模块状态：✅ 已确认
> 文档版本：v1.0

---

## 一、功能验收条件

### AC-01：超级管理员创建平台竞赛

```
Given 超级管理员已登录系统
When 填写竞赛名称、类型（解题赛）、时间、参赛规则、计分配置并保存
Then 竞赛以"草稿"状态创建成功，出现在竞赛管理列表中
```

```
Given 超级管理员已登录系统
When 填写竞赛名称、类型（攻防对抗赛）、时间、回合配置、Token配置并保存
Then 竞赛以"草稿"状态创建成功，攻防赛配置完整保存
```

### AC-02：学校管理员创建校级竞赛

```
Given 学校管理员已登录系统
When 创建竞赛并选择"校级"范围
Then 竞赛关联本校school_id，仅本校学生默认可见
```

```
Given 学校管理员已登录系统
When 尝试创建"平台级"竞赛
Then 系统拒绝，提示"学校管理员只能创建校级竞赛"
```

### AC-03：竞赛发布

```
Given 竞赛处于草稿状态，已配置至少1道题目，时间设置完整
When 竞赛创建者点击"发布"
Then 竞赛状态变为"报名中"，学生可在竞赛大厅看到
```

```
Given 竞赛处于草稿状态，未配置任何题目
When 竞赛创建者点击"发布"
Then 系统拒绝发布，提示"竞赛至少需要1道题目"
```

### AC-04：竞赛状态自动流转

```
Given 竞赛状态为"报名中"，竞赛开始时间已到达
When 定时任务检测到时间条件满足
Then 竞赛状态自动变为"进行中"
And 所有已报名团队的状态锁定为"已锁定"
```

```
Given 竞赛状态为"进行中"，竞赛结束时间已到达
When 定时任务检测到时间条件满足
Then 竞赛状态自动变为"已结束"
And 排行榜解冻，显示最终排名
And 所有竞赛环境开始回收
```

### AC-05：竞赛强制终止

```
Given 竞赛状态为"进行中"
When 超级管理员点击"强制终止"并确认
Then 竞赛状态立即变为"已结束"
And 所有竞赛环境资源被回收
And 排行榜以终止时刻的状态为最终结果
```

### AC-06：竞赛进行中不可修改核心参数

```
Given 竞赛状态为"进行中"
When 管理员尝试修改竞赛时间、规则、题目配置
Then 系统拒绝修改，提示"竞赛进行中不可修改核心参数"
```

### AC-07：教师创建题目（完全自定义）

```
Given 教师已登录系统
When 填写题目名称、类型（contract）、难度、基础分值、Flag类型（链上验证），添加合约代码和断言，保存
Then 题目以"草稿"状态创建成功
And 合约和断言关联保存
```

### AC-08：教师从SWC导入生成题目

```
Given 教师在漏洞转化工具页选择"SWC Registry导入"
When 选择SWC-107（Reentrancy），填写题目名称和难度，点击"导入生成"
Then 系统自动生成题目，包含1个漏洞合约和对应断言
And 题目以"草稿"状态创建，来源标记为"SWC导入"
```

### AC-09：教师从模板生成题目

```
Given 教师在模板库选择"重入攻击模板"
When 选择"跨函数重入"变体，设置难度Medium、基础分500，点击"生成题目"
Then 系统根据模板参数生成题目，包含合约和断言
And 题目以"草稿"状态创建，来源标记为"参数化模板"
And template_params记录所选参数值

### AC-09A：外部A级源生成Fork链题目

```
Given 教师导入一个依赖历史链状态的A级真实漏洞源
When 平台识别该案例必须使用固定历史区块复现，并生成题目草稿
Then 题目 runtime_mode = "forked"
And chain_config.fork 中记录 rpc_url、chain_id、block_number
And setup_transactions 记录复现所需的前置交易
And 题目仍需通过预验证后方可提交审核
```

### AC-09B：C级源降级为链上分析题素材

```
Given 教师导入一个仅包含文章/报告/事故分析的C级外部源
When 平台评估该案例缺少完整源码、PoC 或初始化步骤
Then 系统不生成链上状态验证题
And 仅允许将其降级为 blockchain 或 misc 类静态/动态题素材
And 题目来源标记中保留外部事件引用信息
```
```

### AC-10：题目预验证（通过）

```
Given 教师创建了链上验证题目，已配置合约和断言
When 教师提交PoC脚本并发起预验证
Then 系统依次执行：部署测试环境 → 提交PoC → 正向验证 → 反向验证
And 正向验证：执行PoC后所有断言通过
And 反向验证：不执行PoC时所有断言失败（符合预期）
And 预验证状态变为"通过"

```
Given 题目 runtime_mode 为 "forked"，且配置了固定历史区块
When 教师提交PoC脚本并发起预验证
Then 系统在指定区块创建临时Fork链
And 正向验证在该Fork链上执行 PoC 后所有断言通过
And 反向验证会重置到同一Fork快照后重新执行断言
And 预验证状态变为"通过"
```
```

### AC-11：题目预验证（失败）

```
Given 教师提交的PoC无法触发漏洞
When 预验证执行到正向验证步骤
Then 断言验证失败，预验证状态变为"失败"
And 返回详细错误信息（实际值 vs 期望值）
```

```
Given 题目断言配置有误（未攻击时断言也通过）
When 预验证执行到反向验证步骤
Then 反向验证失败（断言不应在未攻击时通过）
And 预验证状态变为"失败"
```

### AC-12：题目审核

```
Given 教师提交题目审核，链上验证题目有通过的预验证记录
When 超级管理员审核通过
Then 题目状态变为"已通过"，进入平台公共题库
```

```
Given 教师提交题目审核
When 超级管理员审核拒绝并填写拒绝原因
Then 题目状态变为"已拒绝"，教师可查看拒绝原因并修改后重新提交
```

```
Given 链上验证题目无通过的预验证记录
When 超级管理员尝试审核通过
Then 系统拒绝，提示"链上验证题目必须有通过的预验证记录"
```

### AC-13：竞赛题目配置

```
Given 竞赛处于草稿状态
When 管理员从题库选择已审核通过的题目添加到竞赛
Then 题目成功关联到竞赛，可设置排序
```

```
Given 竞赛已发布（报名中或进行中）
When 管理员尝试添加或移除题目
Then 系统拒绝，提示"竞赛已发布，不可修改题目配置"
```

### AC-14：学生创建团队

```
Given 竞赛处于报名阶段，参赛模式为"自由组队"
When 学生创建团队并填写团队名称
Then 团队创建成功，学生成为队长
And 系统生成唯一邀请码
```

```
Given 竞赛参赛模式为"个人赛"
When 学生尝试创建团队
Then 系统拒绝，提示"个人赛无需创建团队"
```

```
Given 学生已在该竞赛的其他团队中
When 学生尝试创建新团队
Then 系统拒绝，提示"您已在该竞赛的其他团队中"
```

### AC-15：通过邀请码加入团队

```
Given 团队处于组建中状态，人数未满
When 学生输入正确的邀请码
Then 学生成功加入团队，角色为"队员"
```

```
Given 团队人数已达max_team_size
When 学生输入该团队邀请码
Then 系统拒绝，提示"团队已满"
```

```
Given 竞赛已开始，团队已锁定
When 学生输入邀请码
Then 系统拒绝，提示"团队已锁定"
```

### AC-16：竞赛报名

```
Given 竞赛处于报名阶段，团队人数满足最低要求
When 队长点击"报名参赛"
Then 报名成功，团队出现在竞赛报名列表中
```

```
Given 竞赛报名队伍数已达max_teams上限
When 队长尝试报名
Then 系统拒绝，提示"报名队伍数已达上限"
```

```
Given 竞赛参赛模式为"个人赛"
When 学生点击"报名参赛"
Then 系统自动创建单人团队并完成报名
```

### AC-17：解题赛 — 题目全量发布

```
Given 竞赛状态变为"进行中"
When 参赛选手进入解题赛主页
Then 所有竞赛题目同时可见
And 每道题目显示类型、难度、当前分值、解题数
And 不显示合约源码、断言详情等敏感信息
```

### AC-18：解题赛 — 启动题目环境

```
Given 参赛选手查看某道智能合约题目
When 点击"启动环境"
Then 系统创建独立K8s Namespace（ctf-{comp_id}-{team_id}-{challenge_id}）
And 部署链节点和漏洞合约
And 环境就绪后返回RPC地址
```

```
Given 选手已有该题目的运行中环境
When 再次点击"启动环境"
Then 系统返回已存在的环境信息，不重复创建
```

### AC-19：解题赛 — 提交静态Flag

```
Given 题目Flag类型为"静态Flag"
When 选手提交正确的Flag字符串
Then 提交标记为正确，获得当前动态分值
And 团队总分更新，排行榜实时刷新
```

```
Given 选手提交错误的Flag
When 提交验证完成
Then 提交标记为错误，返回"Flag不正确"
And 不扣分
```

### AC-20：解题赛 — 提交动态Flag

```
Given 题目Flag类型为"动态Flag"
When 选手提交 flag{HMAC-SHA256(secret, team_id)} 格式的正确Flag
Then 提交标记为正确
And 该Flag仅对本团队有效，其他团队提交同一字符串无效
```

### AC-21：解题赛 — 链上状态验证

```
Given 题目Flag类型为"链上状态验证"，选手已启动题目环境
When 选手提交攻击交易数据
Then 平台在选手环境中执行攻击交易
And 执行所有verification_assertions断言
And 全部断言通过 = 解题成功，获得分值
```

```
Given 选手提交的攻击交易未能触发漏洞
When 断言验证执行完成
Then 部分或全部断言失败，提交标记为错误
And 返回断言详情（实际值 vs 期望值）
```

### AC-22：解题赛 — 动态计分

```
Given 题目base_score=300，decay_factor=0.95，min_score_ratio=0.2
When 第1支队伍解出（solve_count=1）
Then 当前分值 = max(60, 300 × 0.95^1) = 285
```

```
Given 同一题目已有20支队伍解出
When 第21支队伍解出
Then 当前分值 = max(60, 300 × 0.95^21) ≈ max(60, 107) = 107
And 之前解出的队伍得分不追溯调整
```

### AC-23：解题赛 — First Blood

```
Given 某题目尚无队伍解出
When 第1支队伍成功解出
Then 该队伍获得额外10%分值奖励
And 题目标记First Blood队伍和时间
And 排行榜显示First Blood标记
```

### AC-24：解题赛 — 提交限流

```
Given 某队伍某题目本分钟已提交5次
When 该队伍再次提交该题目
Then 系统拒绝，提示"提交过于频繁，请稍后再试"
```

```
Given 某队伍某题目连续失败10次
When 触发冷却期
Then 该队伍该题目5分钟内不可提交
And 前端显示冷却倒计时
```

```
Given 某队伍某题目处于冷却期
When 冷却期结束
Then 该队伍该题目恢复正常提交
And 连续失败计数重置
```

### AC-25：解题赛 — 重复提交防护

```
Given 某队伍已成功解出某题目
When 该队伍再次提交该题目
Then 系统拒绝，提示"您的团队已解出此题"
```

### AC-26：攻防赛 — 分组与环境初始化

```
Given 攻防赛竞赛开始
When 系统初始化分组环境
Then 每个分组创建独立K8s Namespace（ctf-ad-{comp_id}-{group_id}）
And 启动裁判链节点和裁判合约
And 每支队伍创建独立队伍链，部署漏洞合约
And 每支队伍初始化10000 Token
```

### AC-27：攻防赛 — 回合自动推进

```
Given 攻防赛进行中，当前为第3轮攻击阶段
When 攻击阶段时间结束
Then 系统自动切换到防守阶段
And 通过WebSocket推送阶段切换通知
```

```
Given 当前为防守阶段
When 防守阶段时间结束
Then 系统自动进入结算阶段
And 自动执行Token转账计算
And 结算完成后自动进入下一轮攻击阶段
```

```
Given 当前为最后一轮结算阶段
When 结算完成
Then 竞赛状态变为"已结束"
And 以最终Token余额确定排名
```

### AC-28：攻防赛 — 攻击提交

```
Given 当前为攻击阶段，选手选择目标队伍和漏洞
When 提交攻击交易数据
Then 平台在目标队伍链上执行攻击交易
And 执行断言验证
And 全部断言通过 = 攻击成功
And 从目标队伍扣除Token，转入攻击方
And 攻击方额外获得窃取Token的5%作为奖励
```

```
Given 选手尝试攻击自己的队伍
When 提交攻击
Then 系统拒绝，提示"不能攻击自己的队伍"
```

```
Given 目标队伍的某漏洞已被修复（补丁通过）
When 选手尝试攻击该漏洞
Then 系统拒绝，提示"目标漏洞已被修复"
```

### AC-29：攻防赛 — 漏洞衰减

```
Given 某漏洞base_reward=400，decay_factor=0.8
When 第1支队伍攻击成功（exploit_count=1）
Then 攻击奖励 = 400 × 0.8^(1-1) = 400
```

```
Given 同一漏洞已被3支队伍利用
When 第4支队伍攻击成功（exploit_count=4）
Then 攻击奖励 = 400 × 0.8^(4-1) = 400 × 0.512 = 204.8 ≈ 205
```

### AC-30：攻防赛 — First Blood攻击奖励

```
Given 某分组某漏洞尚无队伍成功攻击
When 第1支队伍攻击成功
Then 该队伍获得额外10%的Token奖励
And 攻击记录标记is_first_blood=true
```

### AC-31：攻防赛 — 防守补丁提交

```
Given 当前为防守阶段，选手的某漏洞未修复
When 选手提交补丁合约源码
Then 平台验证补丁：
  1. 功能完整性检查：原有功能测试用例全部通过
  2. 漏洞修复验证：官方PoC不再成功（断言不再通过）
And 两项均通过 = 补丁接受，替换队伍链上的合约
```

```
Given 选手提交的补丁破坏了原有功能
When 功能完整性检查执行
Then 功能测试失败，补丁被拒绝
And 返回拒绝原因"功能完整性检查未通过"
```

```
Given 选手提交的补丁未修复漏洞
When 漏洞修复验证执行
Then 官方PoC仍可成功，补丁被拒绝
And 返回拒绝原因"漏洞修复验证失败"
```

### AC-32：攻防赛 — 首补丁奖励

```
Given 某分组某漏洞尚无队伍成功修复
When 第1支队伍的补丁验证通过
Then 该队伍获得+200 Token首补丁奖励
And 防守记录标记is_first_patch=true
```

### AC-33：攻防赛 — 防守奖励

```
Given 某轮结算阶段
When 某队伍本轮未被任何队伍成功攻击
Then 该队伍获得+50 Token防守奖励
```

### AC-34：攻防赛 — Token流水记录

```
Given 攻防赛进行中，发生Token变动
When 任何Token变动发生（攻击/防守/奖励）
Then 系统在ad_token_ledger中记录完整流水
And 记录包含：变动类型、金额、变动后余额、关联攻击/防守记录
And 选手可在Token流水页查看完整记录
```

### AC-35：排行榜实时更新

```
Given 竞赛进行中，排行榜未冻结
When 有新的正确提交（解题赛）或Token变动（攻防赛）
Then 排行榜通过WebSocket实时推送更新
And 所有在线选手同步看到排名变化
```

### AC-36：排行榜冻结

```
Given 竞赛设置freeze_at为结束前1小时
When 到达冻结时间
Then 排行榜停止公开更新
And 前端显示"已冻结"标记
And 后台仍继续记录真实排名（不公开）
```

```
Given 竞赛结束
When 排行榜解冻
Then 显示最终真实排名
And 冻结期间的排名变化一次性揭晓
```

### AC-37：排行榜排序规则

```
Given 解题赛中两支队伍总分相同
When 排行榜排序
Then 按最后解题时间升序排列（先解出的排前面）
```

```
Given 攻防赛中
When 排行榜排序
Then 按Token余额降序排列
```

### AC-38：公告发布与推送

```
Given 竞赛进行中
When 管理员发布全局公告
Then 公告通过WebSocket实时推送到所有参赛选手
And 选手端顶部铃铛显示未读公告数
```

```
Given 管理员发布题目勘误公告（关联某题目）
When 选手查看该题目详情
Then 题目详情页顶部显示黄色勘误横幅
```

### AC-39：题目环境重置

```
Given 选手的题目环境处于运行中
When 选手点击"重置环境"
Then 系统销毁当前环境并重新创建
And 合约和链状态恢复到初始状态
And 选手之前的攻击操作全部清除
```

### AC-40：题目环境资源配额

```
Given 竞赛资源配额已用尽
When 选手尝试启动新的题目环境
Then 系统拒绝，提示"竞赛资源配额不足"
```

### AC-41：竞赛结束环境回收

```
Given 竞赛状态变为"已结束"
When 环境回收定时任务执行
Then 系统回收该竞赛所有K8s环境资源（Namespace、Pod、PVC）
And challenge_environments状态变为"已销毁"
And ad_team_chains状态变为"已停止"
```

### AC-42：竞赛统计

```
Given 竞赛已结束
When 管理员查看竞赛统计页
Then 显示完整统计数据：
  - 参赛队伍数、总提交数、正确率
  - 每道题目的解题数、正确率、平均解题时间、First Blood
  - 分数分布图
  - 提交时间线
```

### AC-43：竞赛归档

```
Given 竞赛状态为"已结束"超过30天
When 归档定时任务执行
Then 竞赛状态变为"已归档"
And Redis缓存数据清理
And 竞赛历史记录保留可查
```

## 三、安全要求

| 要求 | 说明 |
|------|------|
| 题目环境隔离 | 每支队伍每道题独立K8s Namespace，完全网络隔离 |
| 攻防赛队伍链隔离 | 每支队伍独立链节点，攻击通过平台代理执行，不直接暴露链地址 |
| 裁判链只读 | 选手对裁判链仅有读权限，不可篡改结算逻辑 |
| Flag安全存储 | 静态Flag加密存储，动态Flag密钥不暴露给前端 |
| 合约源码保护 | 参赛选手不可查看题目合约源码和断言详情 |
| PoC安全存储 | 教师提交的PoC脚本加密存储，不暴露给选手 |
| 提交限流 | 每队每题每分钟5次，连续10次失败触发5分钟冷却 |
| 防重复提交 | 已解出的题目不可重复提交（Redis标记） |
| WebSocket认证 | WebSocket连接需携带JWT Token验证身份 |
| 容器安全策略 | 禁止特权容器、限制Linux Capabilities |
| 资源配额强制执行 | 竞赛资源配额在后端强制校验，不可绕过 |
| 跨队伍数据隔离 | 选手只能查看自己团队的提交记录和环境信息 |
| 攻防赛攻击代理 | 攻击交易通过平台裁判服务代理执行，防止直接操作目标链 |
| 补丁代码审查 | 补丁合约在沙箱环境中编译和验证，防止恶意代码 |

## 四、边界条件与异常处理

| 场景 | 处理方式 |
|------|---------|
| 竞赛无题目时发布 | 拒绝发布，提示"竞赛至少需要1道题目" |
| 竞赛时间设置不合理 | 拒绝保存，提示"报名时间必须早于竞赛开始时间" |
| 攻防赛未配置分组时开始 | 系统自动执行随机分组 |
| 题目环境创建失败 | 环境状态变为"异常"，选手可重试 |
| 链节点启动失败 | 记录错误日志，环境状态变为"异常" |
| 攻击交易执行超时 | 返回"执行超时"，攻击标记为失败 |
| 攻击交易执行revert | 返回revert原因，攻击标记为失败 |
| 补丁编译失败 | 返回编译错误信息，补丁被拒绝 |
| 回合结算中系统异常 | 结算事务回滚，重新执行结算 |
| WebSocket连接断开 | 前端自动重连（指数退避，最大间隔30秒），重连后同步最新状态 |
| 大量选手同时启动环境 | 通过资源配额限制并发，超出配额时拒绝并提示 |
| 竞赛结束时选手正在提交 | 接受竞赛结束前发起的提交，拒绝结束后的提交 |
| Token余额为负 | Token余额最低为0，不允许负数 |
| 攻防赛队伍链OOM | 自动重启链节点，从最近状态恢复 |
| 预验证环境超时未清理 | 定时任务每30分钟清理超过1小时的预验证环境 |
| 排行榜快照保存失败 | 记录错误日志，下次定时任务重试 |
| Redis缓存与数据库不一致 | 定时任务每分钟同步Token余额和动态分值到数据库 |
| 选手并发提交同一题目 | 后端加分布式锁，串行处理同一队伍同一题目的提交 |
| 攻防赛同时攻击同一目标 | 后端加分布式锁，串行处理对同一目标的攻击 |
| 竞赛归档后访问 | 可查看历史记录和最终排名，不可启动环境 |
