# 场景算法容器编排 —— 完整落地方案（生产标准）

> 本文档对应 `06-可视化仿真引擎设计.md` 中 SceneManager + 场景算法容器章节，给出**当前代码层 gap** 与**完整落地步骤**。
> 实施完成后，sim-engine 的静态 `scene.endpoints` 配置项将被彻底移除（不保留新旧两套实现）。

---

## 一、现状 gap

| 层 | 文件 | 现状 | 问题 |
|---|---|---|---|
| 数据库 | `migrations/004_experiment.up.sql` | `sim_scenarios.container_image_url` 字段已存在 | ✅ |
| proto | `sim-engine/proto/.../sim_engine.proto` `SceneConfig` | 仅 `scene_code/scenario_id/params/...` | **缺 `container_image_url`、`resource_*`** |
| backend | `instance_service.go::buildSimSessionRequest` | 仅传 `SceneCode` 等元数据 | **不读取 `scenario.ContainerImageURL`，不传给 sim-engine** |
| sim-engine | `internal/scene/grpc_client.go` `NewGRPCClientFactory` | 用静态 `endpoints map[code]"host:port"` | **违反"按需启停"，无法动态拉起 Pod** |

## 二、目标架构（与文档 §6 一致）

```
backend (CreateSession RPC)
   │   传入: scene_code + container_image_url + 资源请求
   ▼
sim-engine Core
   ├── SessionMgr     创建会话 → 调用 SceneManager.Start
   └── SceneManager
        ├── (1) 通过 K8s API 创建 Pod (image=container_image_url)
        ├── (2) 创建 Service 暴露 gRPC 端口
        ├── (3) 等待 Pod Ready (readinessProbe HTTP 200)
        ├── (4) 建立 gRPC 连接，缓存到连接池 (sceneCode → conn)
        ├── (5) 调用 SimScenarioService.Init / Step / ...
        └── (6) DestroySession 时删除 Pod + Service + 关闭连接
```

**关键约定：**
- 场景 Pod 命名空间：`sim-engine`（与 SimEngine Core 同 namespace）
- 场景 Pod 命名：`scn-{session_id_short}-{scene_code}`
- 场景 Service 命名同 Pod 名，暴露 50100 端口
- 镜像拉取策略：`IfNotPresent`（与实验环境一致，避免 :latest 强拉）
- 标签：`app.kubernetes.io/managed-by=sim-engine`, `lenschain.io/session=<session_id>`, `lenschain.io/scene-code=<code>`

// K8sOrchestrator 通过 K8s API 按需启停场景算法容器。
// 实现 ClientFactory + 销毁清理 + 空闲回收。
type K8sOrchestrator struct {
    clientset      *kubernetes.Clientset
    namespace      string
    pullSecretName string
    defaultCPU     string
    defaultMemory  string

    readyTimeout   time.Duration
    pool           sync.Map // sceneRuntimeKey -> *podConnection
}

type podConnection struct {
    podName     string
    serviceName string
    conn        *grpc.ClientConn
    client      simscenariov1.SimScenarioServiceClient
}

// NewK8sOrchestrator 通过 kubeconfig / in-cluster 配置创建编排器。
func NewK8sOrchestrator(cfg OrchestratorConfig) (*K8sOrchestrator, error) { ... }

// CreatePodForScene 创建 Pod + Service，等待 Ready，返回 gRPC 连接。
func (o *K8sOrchestrator) CreatePodForScene(ctx context.Context, sessionID, sceneCode, image, cpu, mem string) (*grpc.ClientConn, error) { ... }

// DestroySessionScenes 删除会话内所有场景 Pod。
func (o *K8sOrchestrator) DestroySessionScenes(ctx context.Context, sessionID string) error { ... }

// AsClientFactory 适配 ClientFactory 函数签名。
func (o *K8sOrchestrator) AsClientFactory() ClientFactory { ... }

## 五、验收标准

1. **数据驱动**：`UPDATE sim_scenarios SET container_image_url='registry.lianjing.com/scenarios/pow-mining:v1.1.0'` 后，下次会话自动使用新镜像，无需修改任何配置或重启 SimEngine
2. **按需启停**：`kubectl get pods -n lenschain -l app.kubernetes.io/managed-by=sim-engine` 在无活跃会话时为空；CreateSession 后秒级出现 Pod；DestroySession 后 Pod 自动消失
3. **空闲回收**：`idle_ttl=10m` 配置下，10 分钟无活动的 Pod 被自动清理（日志可观测）
4. **故障恢复**：场景 Pod 异常退出后，下一次 `Step` 调用自动重建 Pod（与文档 §6.4 "RestartScenario" 一致）
5. **本地开发**：执行 `scripts/build-scenarios.ps1` 后，43 个 `registry.lianjing.com/scenarios/*` 镜像出现在本地 docker；启动纯仿真实验时 SimEngine 自动拉起对应 Pod
6. **生产部署**：K8s 集群通过 RBAC 授予 SimEngine ServiceAccount 创建 Pod 的权限；不需要任何 ConfigMap/Secret 变更即可上线新场景

## 七、推荐落地顺序

1. Phase 1（proto）+ Phase 2（backend）—— 协议层先到位，但 sim-engine 暂不读这两个字段（向后兼容空字符串）
2. Phase 3（sim-engine 编排）+ Phase 5（RBAC）+ Phase 6（数据库种子）—— 一起合并，移除静态 endpoints
3. Phase 4（场景镜像）+ Phase 7（文档）—— 收尾

每一步独立可发布，但最终 Phase 3 合并那次 PR 必须**同时**移除所有静态 endpoints 配置，避免新旧并存。
