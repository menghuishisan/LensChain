## 一、设计概述

ChainMirror SimEngine 是链镜平台自研的可视化仿真引擎，为区块链教学提供完整的算法仿真与交互式可视化能力。引擎采用 **独立微服务 + 场景算法容器化** 架构，仿真算法完全独立于平台后端，前端只负责渲染与交互采集。

### 核心设计原则

- **后端仿真 + 前端渲染**：所有算法逻辑在后端运行，前端不包含任何算法实现
- **算法容器化**：每个仿真场景的算法实现打包为独立容器镜像，与平台后端解耦
- **领域共享渲染器**：按可视化领域划分渲染器，同一领域下所有场景共享同一个渲染器
- **平台自动匹配**：教师只选场景，平台根据场景所属领域自动加载对应渲染器
- **联动可选**：同一联动组内的场景共享状态空间，教师可选择启用或关闭联动
- **可扩展**：教师可通过 SimScenario SDK 开发自定义场景，上传镜像审核后即可使用

### 2.1 三层职责划分

| 层 | 部署形态 | 职责 |
|---|---|---|
| **前端渲染层** | React 应用（随平台前端） | 接收状态数据 → 领域渲染器渲染画面 → 采集用户交互 → 发送操作指令 |
| **SimEngine Core** | 独立微服务（长驻） | 会话管理、场景调度、联动引擎、仿真时钟、状态管理、WebSocket Hub |
| **场景算法容器** | 独立容器镜像（按需启停） | 实现具体算法逻辑（PBFT、PoW、SHA-256 等），通过 gRPC 与 Core 通信 |

### 2.2 场景算法容器 vs 实验环境容器

| | 场景算法容器 | 实验环境容器 |
|---|---|---|
| 用途 | 运行仿真算法逻辑 | 运行真实链节点/开发工具 |
| 体积 | 极小（10-50MB） | 较大（100MB-1GB+） |
| 资源 | CPU 少量 + 内存少量 | CPU/内存/存储按需分配 |
| 生命周期 | 随仿真会话启停 | 随实验实例启停 |
| 镜像来源 | 仿真场景库（sim_scenarios） | 镜像库（images，四类） |
| 通信方式 | gRPC 与 SimEngine Core | SSH/WebSocket 与前端终端 |
| 启动速度 | 秒级（极轻量） | 数秒到数十秒 |

### 2.3 三种实验类型的运行方式

```
纯仿真实验：
  SimEngine Core 启动场景算法容器 → gRPC 通信 → WebSocket 推送前端渲染
  不启动任何实验环境容器
  学生即开即用（场景容器秒级启动）

真实环境实验：
  只启动实验环境容器（Geth/Fabric/工具等）
  没有 SimEngine，学生直接操作真实链
  通过 Web 终端/IDE 交互

混合实验：
  实验环境容器运行真实链 + SimEngine 可视化
  SimEngine 通过采集模式（Collector Agent sidecar）从容器拉取实时数据
  算法逻辑在容器里的真实链上跑，SimEngine 只负责采集 + 渲染
```

**总计：8 个渲染器，覆盖 43 个场景。**

### 3.2 渲染器自动匹配机制

```
教师选择场景（如 PBFT 三阶段共识）
    │
    ▼
场景元信息：category = "consensus"
    │
    ▼
DomainRendererRegistry 查找 category → ConsensusProcessRenderer
    │
    ▼
渲染器根据 algorithm_type = "pbft-consensus" 加载对应渲染配置
    │
    ▼
渲染画面
```

教师和学生均不感知渲染器的存在，平台全自动匹配。

### 3.3 领域渲染器接口定义

```typescript
interface DomainRenderer {
  // 领域标识
  domain: string;

  // 支持的算法类型列表
  supportedAlgorithms: string[];

  // 根据算法类型获取渲染配置（布局、颜色方案、图元类型等）
  getRenderConfig(algorithmType: string): RenderConfig;

  // 渲染主方法：接收后端推送的状态，输出画面
  render(state: RenderState, canvas: CanvasContext): void;

  // 处理用户交互，转换为后端 action
  handleInteraction(event: InteractionEvent): SimAction;

  // 动画插值：在两个 tick 之间平滑过渡
  interpolate(prevState: RenderState, nextState: RenderState, t: number): RenderState;
}
```

### 3.4 跨领域场景处理

部分场景涉及两个领域的渲染能力（如 Merkle 树同时属于密码学和数据结构），处理方式：

- 场景声明一个**主领域**（决定使用哪个渲染器）和可选的**辅助领域**
- 主渲染器可以内嵌调用辅助渲染器的子组件
- 例：CRY-05 Merkle 树构建验证，主领域 cryptography，辅助领域 data_structure，CryptographyRenderer 内嵌 DataStructureRenderer 的树形渲染子组件

## 五、场景时间控制模式

不是所有场景都需要完整的时间控制器。根据场景特性分为三种模式：

### 5.1 三种时间控制模式

| 模式 | 编码 | 适用场景 | 前端控件 | 说明 |
|------|------|---------|---------|------|
| **过程化** | process | 有明确步骤/阶段的场景 | 播放/暂停/单步/回退/变速/重置 | PBFT 共识、交易生命周期等 |
| **交互响应式** | reactive | 输入即响应的场景 | 无时间控件，只有操作面板 | SHA-256 哈希、Merkle 树构建等 |
| **持续运行式** | continuous | 持续演化观察的场景 | 暂停/恢复/变速 | P2P 网络发现、Gossip 传播等 |

### 5.2 过程化模式控件规格

| 控件 | 功能 | 说明 |
|------|------|------|
| 播放 ▶ | 按当前速率自动推进 tick | 默认速率 1x |
| 暂停 ⏸ | 冻结当前 tick | — |
| 单步 ⏭ | 前进 1 个 tick | 暂停状态下可用 |
| 回退 ⏮ | 回到上一个 tick | 从快照栈恢复 |
| 变速 | 0.5x / 1x / 1.5x / 2x | 四档切换 |
| 重置 ⟲ | 回到初始状态（tick=0） | 重新初始化场景 |

### 5.3 动画节奏

- 每个场景自定义默认动画节奏，不做全局统一
- 原则：每个过程步骤动画时长 **1-3 秒**（1x 速率下），确保学生能看清但不会等太久
- 场景开发者根据教学效果调优各步骤时长
- 变速只影响 tick 推进速率，不影响单步动画的流畅度

## 六、SimEngine Core 详细设计

### 6.1 核心组件

| 组件 | 职责 |
|------|------|
| **SceneManager** | 场景生命周期管理：通过编排服务启动/停止场景算法容器，维护 gRPC 连接池 |
| **LinkEngine** | 联动引擎：管理共享状态空间，场景间状态变更自动传播 |
| **SessionMgr** | 会话管理：每个学生实验实例对应一个仿真会话，隔离状态 |
| **SimClock** | 仿真时钟：管理 tick 推进、变速、暂停/恢复，按场景时间控制模式工作 |
| **EventBus** | 事件系统：所有仿真事件按 tick 排序，支持回放 |
| **StateMgr** | 状态管理：维护每个 tick 的状态，计算 diff，只推送变更给前端 |
| **SnapshotStack** | 快照栈：支持状态回退和暂停恢复 |
| **Collector** | 数据采集器：混合实验中从容器 Collector Agent 采集实时数据注入仿真状态 |
| **WebSocket Hub** | 管理所有前端连接，路由消息到对应会话 |

### 6.2 SimEngine Core 与场景算法容器通信

```protobuf
// 场景算法容器暴露的 gRPC 服务
service SimScenario {
    // 初始化场景
    rpc Init(InitRequest) returns (SceneState);

    // 执行一个仿真步进
    rpc Step(StepRequest) returns (StepResponse);

    // 处理用户交互操作
    rpc HandleAction(ActionRequest) returns (ActionResponse);

    // 获取可渲染状态
    rpc GetRenderState(StateRequest) returns (RenderState);

    // 获取交互操作定义
    rpc GetInteractionSchema(Empty) returns (InteractionSchema);

    // 获取场景元信息
    rpc GetMeta(Empty) returns (ScenarioMeta);

    // 健康检查
    rpc HealthCheck(Empty) returns (HealthResponse);
}

message ScenarioMeta {
    string code = 1;              // 场景编码
    string name = 2;              // 场景名称
    string category = 3;          // 所属领域（决定前端用哪个渲染器）
    string algorithm_type = 4;    // 算法类型标识
    string description = 5;       // 场景描述
    string version = 6;           // 版本号
    string time_control_mode = 7; // process / reactive / continuous
    string data_source_mode = 8;  // simulation / collection / dual
    bytes default_params = 9;     // 默认参数（JSON）
}

message StepRequest {
    bytes state = 1;              // 当前状态
    int64 tick = 2;               // 当前 tick
    bytes shared_state = 3;       // 联动共享状态（如有）
}

message StepResponse {
    bytes state = 1;              // 新状态
    repeated SimEvent events = 2; // 产生的事件
    bytes shared_state_diff = 3;  // 共享状态变更（如有）
}

message ActionRequest {
    bytes state = 1;              // 当前状态
    string action_code = 2;       // 操作编码
    bytes params = 3;             // 操作参数（JSON）
}

message ActionResponse {
    bytes state = 1;              // 新状态
    repeated SimEvent events = 2; // 产生的事件
    bytes render_state = 3;       // 可渲染状态
}
```

### 6.3 状态快照与回退策略

| 策略 | 说明 |
|------|------|
| 关键帧快照 | 每 50 个 tick 自动保存完整状态 |
| 增量快照 | 每个 tick 保存 state diff |
| 回退机制 | 找到最近关键帧 → 重放 diff 到目标 tick |
| 快照栈深度 | 最多保留最近 1000 个 tick |
| 暂停/恢复 | 暂停时保存完整快照到持久化存储（对象存储） |

### 6.4 多人协作仿真状态同步

多人组网实验（拓扑模式 3）中，多个学生可能同时操作同一个仿真会话：

| 机制 | 说明 |
|------|------|
| 操作冲突解决 | 按服务端收到的时间顺序执行，乐观锁 |
| 操作广播 | A 学生的操作结果实时推送给同组所有学生 |
| 角色权限 | 不同角色可执行不同操作（如角色 A 控制节点 1-3，角色 B 控制节点 4-6） |
| 状态一致性 | 所有学生看到的仿真状态由服务端统一管理，前端只渲染 |

```json
{
```

### 7.3 后端 → 前端消息

| type | 说明 | payload 示例 |
|------|------|-------------|

## 八、联动机制设计

### 8.1 联动原理

联动不是"通知机制"，而是**共享状态空间**。同一联动组内的场景读写同一份底层状态对象，一个场景的操作自然影响其他场景的仿真结果。

```
┌─────────────┐     ┌─────────────────────┐     ┌─────────────┐
│ PoW挖矿场景  │────▶│   共享状态空间         │◀────│ 51%攻击场景  │
│ 读写: nodes, │     │ nodes: {...}         │     │ 读写: nodes, │
│ blockchain   │     │ blockchain: {...}    │     │ blockchain   │
└─────────────┘     │ network: {...}       │     └─────────────┘
                    │ globalParams: {...}  │
┌─────────────┐     │                      │     ┌─────────────┐
│ 区块链分叉场景│────▶│                      │◀────│ 网络传播场景  │
│ 读: blockchain│    └─────────────────────┘     │ 读写: network│
└─────────────┘                                  └─────────────┘
```

### 8.2 联动组定义（9 组）

| # | 联动组 | 编码 | 包含场景 | 共享状态 | 联动效果 |
|---|--------|------|----------|----------|----------|
| 1 | PoW 攻击组 | pow-attack-group | CON-01 + ATK-01 + DAT-01 + NET-05 | nodes, blockchain, network | 调整攻击者算力→挖矿竞争变化→链分叉→网络传播延迟 |
| 2 | PBFT 攻击组 | pbft-attack-group | CON-03 + ATK-03 + NET-03 | nodes, messages, view | 注入拜占庭节点→投票异常→视图切换→网络消息变化 |
| 3 | Raft 容错组 | raft-fault-group | CON-04 + NET-03 + NET-05 | nodes, terms, logs | 宕掉 Leader→重新选举→网络分区→脑裂场景 |
| 4 | 密码学验证组 | crypto-verify-group | CRY-01 + CRY-03 + CRY-05 | hashes, keys, tree | 修改数据→哈希变化→签名失效→Merkle 路径变化 |
| 5 | 区块链完整性组 | blockchain-integrity-group | DAT-01 + TXN-01 + CRY-05 + DAT-02 | blocks, transactions, merkle_root | 发起交易→打包入块→Merkle 树更新→链结构增长 |
| 6 | 合约安全组 | contract-security-group | ATK-04 + ATK-05 + SC-01 + SC-02 | contract_state, call_stack, balances | 发起攻击→调用栈展开→状态异常变化→余额被盗 |
| 7 | PoS 经济组 | pos-economy-group | CON-02 + ECO-01 + ECO-03 + ECO-02 | stakes, token_supply, proposals | 质押变化→验证者权重变化→投票权变化→治理结果变化 |
| 8 | 交易处理组 | tx-processing-group | TXN-01 + TXN-02 + TXN-03 + TXN-04 | mempool, gas, balances | 交易入池→Gas 计算→排序打包→余额变更 |
| 9 | 网络基础组 | network-base-group | NET-01 + NET-02 + NET-06 | topology, routing_table, load | 节点发现→消息传播→负载均衡 |

### 8.3 联动可选机制

- 教师在编排实验时选择仿真场景
- 如果选择了同一联动组内的多个场景，平台**自动提示**"检测到可联动场景，是否启用联动？"
- 教师可选择：**启用联动**（共享状态空间）或 **独立运行**（各场景独立状态）
- 单个场景始终可以独立运行，不强制联动
- 学生端不感知联动配置，只看到多个场景面板自然联动或独立运行

### 8.4 联动状态传播流程

```
场景A执行 Step/HandleAction
    │
    ▼
返回 shared_state_diff
    │
    ▼
LinkEngine 将 diff 合并到共享状态空间
    │
    ▼
LinkEngine 通知同组其他场景：共享状态已变更
    │
    ▼
其他场景下一次 Step 时读取到最新共享状态
    │
    ▼
各场景产生的 RenderState 推送给前端
    │
    ▼
前端各面板同步更新（学生看到联动效果）
```

## 九、混合实验数据采集

### 9.1 采集架构

混合实验中，真实链节点运行在实验环境容器中，SimEngine 通过 Collector Agent（sidecar 容器）采集实时数据进行可视化。

```
┌──────────────────────────────────────────────────┐
│              实验环境容器组                         │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌────────────────┐  │
│  │ Geth节点1 │  │ Geth节点2 │  │ Collector Agent│  │
│  │          │  │          │  │ (sidecar容器)   │  │
│  └────┬─────┘  └────┬─────┘  └───────┬────────┘  │
│       │              │                │            │
│       └──────────────┴────────────────┘            │
│              容器内部网络                            │
└──────────────────────────┬───────────────────────┘
                           │ WebSocket
┌──────────────────────────┴───────────────────────┐
│  SimEngine Core → Collector 模块                   │
│  接收采集数据 → 转换为 RenderState → 推送前端渲染    │
└──────────────────────────────────────────────────┘
```

### 9.2 Collector Agent 标准化接口

Collector Agent 是一个轻量 sidecar 容器，负责从同 Pod 内的链节点采集数据并转换为标准格式。

```json
// Collector Agent 输出的标准化数据格式
{
  "source": "ethereum",
  "timestamp": 1712500000000,
  "data_type": "block | transaction | peer | consensus | contract_event",
  "payload": {
    // 根据 data_type 不同而不同
  }
}
```

### 9.3 链生态采集适配器

| 链生态 | 采集方式 | 采集内容 |
|--------|---------|---------|
| Ethereum (Geth) | JSON-RPC + WebSocket 订阅 | 区块、交易、节点列表、合约事件 |
| Hyperledger Fabric | gRPC EventHub | 区块、交易、链码事件、通道状态 |
| ChainMaker | SDK 订阅 | 区块、交易、合约事件 |
| FISCO BCOS | JSON-RPC + AMOP | 区块、交易、共识状态 |

平台内置以上四种链生态的 Collector Agent 镜像，教师可通过 SDK 开发自定义采集适配器。

## 十四、与现有编排流程的集成

教师在实验模板创建向导（P-21）的步骤三"工具与可视化"中：

1. 选择实验类型（纯仿真 / 真实环境 / 混合）
2. 如果包含仿真：从场景库中选择场景（按领域分类浏览）
3. 配置场景参数（节点数、难度、延迟等，每个场景有自己的参数 schema）
4. 如果选了同一联动组的多个场景 → 平台自动提示"是否启用联动"
5. 配置场景面板布局（栅格拖拽排列）
6. 如果是混合实验：配置 Collector Agent（选择链生态适配器）

学生端全自动：启动实验 → 后端初始化仿真会话 + 启动场景算法容器 → 前端自动加载领域渲染器 → 开始交互。
