## 一、设计概述

ChainMirror SimEngine 是链镜平台自研的可视化仿真引擎，为区块链教学提供完整的算法仿真与交互式可视化能力。引擎采用 **独立微服务 + 场景算法容器化** 架构，仿真算法完全独立于平台后端，前端只负责渲染与交互采集。

### 核心设计原则

- **后端仿真 + 前端渲染**：所有算法逻辑在后端运行，前端不包含任何算法实现
- **算法容器化**：每个仿真场景的算法实现打包为独立容器镜像，与平台后端解耦
- **领域共享渲染器**：按可视化领域划分渲染器，同一领域下所有场景共享同一个渲染器
- **平台自动匹配**：教师只选场景，平台根据场景所属领域自动加载对应渲染器
- **联动可选**：同一联动组内的场景共享状态空间，教师可选择启用或关闭联动
- **可扩展**：教师可通过 SimScenario SDK 开发自定义场景，上传镜像审核后即可使用

### 2.1 三层职责划分

| 层 | 部署形态 | 职责 |
|---|---|---|
| **前端渲染层** | React 应用（随平台前端） | 接收状态数据 → 领域渲染器渲染画面 → 采集用户交互 → 发送操作指令 |
| **SimEngine Core** | 独立微服务（长驻） | 会话管理、场景调度、联动引擎、仿真时钟、状态管理、WebSocket Hub |
| **场景算法容器** | 独立容器镜像（按需启停） | 实现具体算法逻辑（PBFT、PoW、SHA-256 等），通过 gRPC 与 Core 通信 |

### 2.2 场景算法容器 vs 实验环境容器

| | 场景算法容器 | 实验环境容器 |
|---|---|---|
| 用途 | 运行仿真算法逻辑 | 运行真实链节点/开发工具 |
| 体积 | 极小（10-50MB） | 较大（100MB-1GB+） |
| 资源 | CPU 少量 + 内存少量 | CPU/内存/存储按需分配 |
| 生命周期 | 随仿真会话启停 | 随实验实例启停 |
| 镜像来源 | 仿真场景库（sim_scenarios） | 镜像库（images，四类） |
| 通信方式 | gRPC 与 SimEngine Core | SSH/WebSocket 与前端终端 |
| 启动速度 | 秒级（极轻量） | 数秒到数十秒 |

### 2.3 三种实验类型的运行方式

```
纯仿真实验：
  SimEngine Core 启动场景算法容器 → gRPC 通信 → WebSocket 推送前端渲染
  不启动任何实验环境容器
  学生即开即用（场景容器秒级启动）

真实环境实验：
  只启动实验环境容器（Geth/Fabric/工具等）
  没有 SimEngine，学生直接操作真实链
  通过 Web 终端/IDE 交互

混合实验：
  实验环境容器运行真实链 + SimEngine 可视化
  SimEngine 通过采集模式（Collector Agent sidecar）从容器拉取实时数据
  算法逻辑在容器里的真实链上跑，SimEngine 只负责采集 + 渲染
```

## 三、原语化渲染体系

### 3.1 范式总览

所有场景统一走 **原语组合 + 微动画调度** 渲染范式：

- **视觉单元**：47 个原语（几何 / 动效 / 布局 / 数据展示 / 状态指示 / 领域复合）
- **渲染器**：1 个 `PrimitiveBasedRenderer` 基类 + 8 个类目皮肤（仅配色 / 字号 / 图元密度差异）
- **协议**：统一 `RenderEnvelope`（primitives / micro_steps / link_triggers / container_data，详 §六）
- **教师扩展**：L1 原语组合（≥ 90% 场景） / L2 +1 原语函数 / L3 npm 自定义渲染器

47 原语 + 1 基类 + 8 类目皮肤覆盖 43 内置场景与教师扩展场景。**清单已经过 43 场景反向验证：每个原语在 ≥3 个场景中被调用，无冗余无遗漏。**

### 3.2 47 原语完整清单（含参数 schema）

按职责分 6 大类。每个原语的 `params` 字段必须按下方 schema 提供；可选参数标 `?`；参数组成的稳定 ID 由场景 `BuildRenderState` 生成（不变即为同一原语跨 tick）。

> **坐标字段统一约定**（v0.5 决议，覆盖下方所有原语 schema）：
>
> 47 原语中所有坐标字段（`x` / `y` / `center_x` / `center_y` / `radius` / `cell_w` / `cell_h` / `bounds` 等）按下列规则解读，下方 schema 表保留字面写法以表达**协议字段名**，实际使用时按本约定决定是否填写。
>
> 1. **参与父布局原语时**（如 `node` 装入 `ring_layout` / `graph_layout` / `matrix_layout` / `tree_layout` / `stack` / `horizontal_lane`）：坐标字段**必须省略**，由前端 `PrimitiveBasedRenderer` 按布局原语的算法字段（`algorithm` / `slots` / `rows` / `cols`）+ 画布响应式尺寸推导。
> 2. **不参与任何布局原语（绝对定位）**：坐标字段**必填**，单位为画布逻辑坐标 `0~1`（前端按 viewport 实际像素缩放），用于少数自由分布场景（攻击节点群、对照展示、画布角落 HUD 等）。
> 3. **布局原语自身的坐标参数**（`ring_layout.center_x/center_y/radius`、`matrix_layout.cell_w/cell_h`、`graph_layout.bounds` 等）：默认由前端皮肤决定（响应式 + 类目皮肤密度差异）；场景仅在画布角落或对照定位时给出"建议初值"，前端有最终决定权。
>
> framework / sdk Go 构造器配套提供两个变体（`sim-engine/AGENTS.md §3.7` 命名约定）：
>
> - 默认构造器（如 `PrimNode(id, label, status, role)`）输出**无坐标版本**，参与父布局——这是 90%+ 场景的正确路径
> - `*At` 后缀构造器（如 `PrimNodeAt(id, label, status, role, x, y, size)`）输出带逻辑坐标版本，仅用于绝对定位
>
> 教师 L3 自定义场景应优先使用默认构造器，仅在必要时使用 `*At` 后缀，以确保 8 类目皮肤的响应式与一致性。
>
> **判别试纸**（详 `sim-engine/AGENTS.md` 顶部）：任何不确定的字段，问"教师 L3 自定义场景作者能否在不改前端代码的情况下使用？"——能 → 该字段属于场景层；不能 → 该字段属于前端皮肤。坐标值具体像素属于"不能"一类，故场景默认不写。

#### 3.2.1 几何类（8 个）

> 静态可视化的基本图元；`Primitive.Layer` 默认 `content`。

| 原语 | 参数 schema（params 字段内） | 用途 | 复用场景示例 |
|---|---|---|---|
| `node` | `id, x, y, label, status, role, size?` | 拓扑节点、状态机节点、树节点 | 几乎所有图布局类场景 |
| `edge` | `id, from_id, to_id, style?, animation?` | 节点连线、引用关系 | 拓扑、状态机、树、调用图 |
| `bar` | `id, x, y, height, width?, color_role, label?` | 算力柱、质押权重柱、Gas 消耗柱 | PoW/PoS/Gas/Bloom 等 |
| `curve` | `id, equation?, points[], style` | 椭圆曲线、AMM 曲线、供应曲线 | ECDSA/RSA/AMM/Token 经济 |
| `polygon` | `id, vertices[], fill?, stroke?` | 区块边框、多边形分区 | 区块结构、网络分区 |
| `area` | `id, points[], gradient?` | 面积图（Gas 利用率、链高对比） | EIP-1559 / 区块同步 |
| `grid_cell` | `id, row, col, value, color_role` | 投票矩阵、位运算矩阵、Bloom 位数组 | PBFT / SHA-256 / Bloom |
| `ring` | `id, x, y, radius, total, current, label?` | 进度环、Epoch 时间轮 | SHA-256 64 轮、PoS Epoch |

#### 3.2.3 布局类（6 个）

> 决定多个原语的排列方式。原语本身不绘制内容，但定义"画在哪里"；`Primitive.Layer` 默认 `background`。

| 原语 | 参数 schema | 用途 |
|---|---|---|
| `horizontal_lane` | `id, y, label?` | 水平泳道（多矿工竞速、多链对比） |
| `stack` | `id, x, items[], direction` | 调用栈（EVM、重入攻击） |
| `ring_layout` | `id, center_x, center_y, radius, nodes[]` | 环形节点布局（PBFT、DPoS）。`nodes[]` 显式声明环上成员 ID 列表，渲染器按列表顺序从 12 点钟方向顺时针均分 N 个 slot；与 `graph_layout.nodes[]` 同义。**不再使用 `slots` 数量字段**，槽位由 `len(nodes)` 推导，避免 `slots` 与实际 node 数漂移。 |
| `tree_layout` | `id, root_id, layout_algorithm` | 自顶向下/自底向上树（Merkle、MPT、分叉树） |
| `graph_layout` | `id, algorithm: force\|circular\|grid, nodes[], edges[]` | 力导向 / 环形 / 网格图（拓扑场景） |
| `matrix_layout` | `id, rows, cols, cell_w, cell_h` | 矩阵布局（投票矩阵、位数组） |

| `code_marker` | label + glow | 操作码 / PC 指针标记（EVM） |
| `partition_zone` | polygon + glow | 网络分区区域 |
| `curve_point` | curve + node + trail | 曲线上的点（ECDSA / RSA） |

**总计：8 + 7 + 6 + 7 + 8 + 11 = 47 个原语**。

### 3.3 原语分层（Layer）

每个原语在 `Primitive.Layer` 字段声明四层之一，渲染器按层叠顺序绘制：

| Layer | 用途 | 默认承载原语 |
|---|---|---|
| `background` | 背景层（坐标轴 / 区域底色 / 布局） | `horizontal_lane` / `stack` / `ring_layout` / `tree_layout` / `graph_layout` / `matrix_layout` |
| `content` | 主内容层（节点 / 边 / 数据图 / 文本） | `node` / `edge` / `bar` / `polygon` / `vote_matrix` / `register_row` / `math_pipeline` / `code_block` / `math_formula` / `label` |
| `effect` | 动效层（粒子 / 高亮 / 抖动） | `burst` / `particle_stream` / `glow` / `trail` / `pulse` / `shake` / `shift_animation` |
| `overlay` | 覆盖层（HUD / 徽章 / 错误 / 教师标注） | `tooltip` / `annotation` / `link_indicator` / `external_event_marker` / `error_overlay` / `phase_progress` |

不使用数字 z-index，避免冲突。同层内按 `RenderEnvelope.primitives[]` 数组顺序绘制（先入底）。原语可显式覆盖默认 Layer。

### 3.4 原语交互态

每个原语可声明三个交互态字段：

| 字段 | 类型 | 用途 |
|---|---|---|
| `Clickable` | bool | 是否响应点击 |
| `HoverTooltip` | string \| null | 鼠标悬停时显示的 tooltip 文本（支持 i18n key 占位但平台不实现 i18n） |
| `ClickAction` | string \| null | 点击触发的 `action_code`（绑定 `InteractionDefinition.actions`，详 §6.3） |

hover / active 视觉反馈（亮度 +10% / 边框加粗）由渲染器基类默认实现，不通过协议下发。

### 3.7 教师扩展三层模型（L1 / L2 / L3）

| 层级 | 占比 | 适用 | 教师工作量 | 触发条件 |
|---|---|---|---|---|
| **L1 原语组合** | ≥ 90% | 视觉范式落在 47 原语内 | 0 行渲染代码（只写场景算法 + `BuildRenderState` 返回原语数组） | 默认路径 |
| **L2 +1 原语函数** | 5-9% | 缺少某种通用图元 | 平台核心团队加 ~50 行原语函数 | ≥ 3 个场景共需该原语 |
| **L3 npm 自定义渲染器** | < 1% | 视觉范式完全不属 8 类目（如量子区块链、特殊领域） | 教师开发自定义 npm 包，实现 `CustomRenderer` 接口 | `category = "generic"` 且原语库无法表达 |

**L3 npm 包路径**：上传至平台 npm 私有仓库 → 管理员审核（安全扫描 + 沙箱测试）→ 注册到 `DomainRendererRegistry` → 教师在场景 meta 的 `custom_renderer_package` 字段引用包名。

### 3.8 跨领域场景

所有原语为平等可组合单元，不区分主辅。例：CRY-05 Merkle 树直接组合 `tree_layout` + `node`（树节点）+ `edge`（父子边）+ `glow`（被篡改节点）+ `verify_path_highlight`（验证路径）+ `math_formula`（哈希公式）+ `code_block`（哈希值展示）。`sim_scenarios.category` 仅决定皮肤，不约束可用原语。

### 3.9 锚点（anchor_id）语义

> 多个动效与状态指示原语依赖 `anchor_id` 字段定位"附着到哪个图元"，必须明确语义以避免重构歧义。

**定义：**

`anchor_id` **是任意一个已存在的 `Primitive.ID`**，不是独立的"锚点"实体。被锚定的原语必须在同一 `RenderEnvelope.primitives[]` 数组中（且应在 anchor 之前出现，确保渲染时已就位）。

**使用场景：**

| 字段 | 来源 | 引用 |
|---|---|---|
| `Primitive.params.anchor_id`（动效原语 / state marker） | 同 envelope 内某 content 层 Primitive.ID | 视觉附着 |
| `LinkTrigger.source_anchor_id` | 源场景的 Primitive.ID | M8 跨画布弧线起点 |
| `LinkTrigger.target_anchor_id` | 接收方场景的 Primitive.ID | M8 跨画布弧线终点 |

**锚定规则：**

1. **存在性**：渲染器收到 envelope 时校验 `anchor_id` 必须存在于本 envelope 或上一帧 PrimitiveCache（FullSnapshot=false 增量推送时）；缺失则跳过该动效原语并记 warning（不抛错）
2. **跨场景锚点**（仅 LinkTrigger）：`source_anchor_id` 引用的 ID 必须存在于 `source_scene` 的最新 envelope；前端 MultiSceneContainer 持有所有同屏场景的 PrimitiveCache，统一查询
3. **位置推导**：渲染器从被锚定 Primitive 的 bounding box 中心点推导 (x, y)；如该 Primitive 已声明 (x, y) 则直接使用
4. **锚点消失**：被锚定的 Primitive 在下一帧消失（不在 envelope 中且 FullSnapshot=true）→ 依赖它的动效原语自动清除

**最佳实践：**

- 锚点优先选 `node` / `bar` / `grid_cell` 等有明确 (x, y) 的内容层原语
- 不要将 `anchor_id` 指向另一个动效层原语（嵌套动画难调试）
- 跨画布锚点（M8）建议用语义稳定的 ID（如 `attacker-miner-A`），避免 tick 间漂移

### 3.10 动效原语动画行为细节

> 动效类 7 个原语遵循两条原则：
>
> 1. **视觉曲线基类硬编码，时长协议可配**：动画关键帧（如"渐扩+fade"、"正弦呼吸"、"6 次摆动"）属视觉语言刚性部分由前端基类实现；所有时长参数通过 params 下发，基类只给默认值。
> 2. **速率倍率只缩放 tick 推进 / MicroStep 间隔，不压缩单个原语动画时长**（对齐 `06.2-SimEngine前端设计.md` §4.3 既定 SSOT）。这保证学生 2× 变速"快速过场景"时，单个动画（如 burst 爆炸）视觉不走样。

#### 3.10.1 动效原语参数与默认行为

| 原语 | 视觉曲线（基类固定） | 时长 params（协议可配） | 默认值（ms） | 说明 |
|---|---|---|---|---|
| `particle_stream` | 粒子从 anchor 按 direction 持续抛出，随机扰动 + 透明度渐变 | `rate?` / `lifetime_ms?` | rate=20 / lifetime=1200 | 持续运行至原语下一帧消失；direction 枚举 `out` / `broadcast` / `target:<primitive_id>` |
| `burst` | 圆环渐扩 + 中心光斑 + fade out | `duration_ms?` | 600 | `fired_at_tick` 控制触发幂等（同 tick 不重放） |
| `pulse` | 亮度 50% ↔ 100% 正弦呼吸 | `period_ms?` | 1200 | 持续运行直到原语消失 |
| `trail` | 按 points 顺序扫描发光 + 尾部 fade | `duration_ms?` / `fade_ms?` | 1500 / 500 | 扫描结束后 fade；points[] 必须按时序排序 |
| `glow` | 静态 box-shadow（不闪烁） | —（持续态，无时长） | — | `intensity` 0-1 映射到 shadow blur 半径 |
| `shake` | 6 次摆动后归位（正弦衰减） | `duration_ms` | 400 | `magnitude` px 2-10 |
| `shift_animation` | 平滑 ease-in-out 位移 | `duration_ms` | 500 | 结束后图元位置永久更新 |

**默认值来源：** 前端基类 `DEFAULT_DURATIONS` 常量。优先级：`params.duration_ms > 前端主题系数 > 基类默认`。教师课堂"慢速观察模式"可通过前端主题设 1.5× 全局系数放大所有动画。

**速率倍率不影响动画时长的副作用：** `MicroStep.duration_ms` 必须满足 **≥ 该步骤内 fire_primitives[] 最长动画时长 + 200ms 观察 buffer**，否则 2× 变速下 MicroStep 窗口（`duration_ms / 2`）可能短于内部动画，造成动画被下一步骤打断。场景 `BuildRenderState` 构造 MicroStep 时必须校验该约束；违反时后端记 warning 日志 + 自动延长 duration_ms。

#### 3.10.2 速率倍率（Speed Multiplier）作用范围

学生端时间控件的 `0.5× / 1× / 1.5× / 2×` 速率倍率（四档，对齐 06.2 §4.3 与 BR-41）：

| 作用对象 | 是否受速率影响 | 规则 |
|---|---|---|
| Tick 推进频率（后端 Step 调用间隔） | ✅ | × multiplier（2× 下 tick 频率翻倍） |
| `MicroStep.duration_ms`（步骤间等待时间） | ✅ | × (1 / multiplier) |
| 动效原语 `duration_ms` / `period_ms` / `lifetime_ms` | ❌ | **不缩放**，保持视觉节奏一致 |
| `particle_stream.rate` | ❌ | 保持粒子密度不变 |
| `cross-canvas-overlay` 弧线 fade（默认 1.5s） | ❌ | 保持联动感知清晰 |
| `burst.fired_at_tick` / `glow.intensity` / `shake.magnitude` / `trail.points[]` | ❌ | 形状参数不受时间影响 |

#### 3.10.3 MicroStep 与动效原语协作

- `MicroStep.fire_primitives[]` 列出"本步骤要触发"的动效原语 ID（典型：`burst` / `shake`）
- `MicroStep.highlight_ids[]` 列出"本步骤要高亮"的内容层原语 ID（前端基类自动叠加临时 `glow` + 边框加粗）
- 一个 MicroStep 内 fire 多个 burst 时**按数组顺序错开触发**，错开量 = `min(MicroStep.duration_ms / (N+1), 150ms)`（N = fire_primitives 数量；150ms 上限为经验值，前端主题可覆盖）；该逻辑为基类实现，不上协议
- 同一原语 ID 在连续 MicroStep 中重复 fire → 前一个未播完即打断重放

#### 3.10.4 前端动画调度器的职责边界

前端动画调度器（实现于 `sim-engine/renderers/shared/animationScheduler.ts`）仅负责：

1. 按 `MicroStep.duration_ms` 排队执行（受速率倍率影响：1× → `duration_ms`；2× → `duration_ms / 2`）
2. 步骤生效时，对 `highlight_ids[]` 列表的内容层原语叠加临时 `glow` + 边框加粗
3. 步骤生效时，对 `fire_primitives[]` 列表的动效原语调用其内置动画曲线（`burst` 渐扩 fade、`pulse` 正弦呼吸、`shake` 6 次摆动等）
4. 切换到下一 MicroStep 前，回收临时高亮（保持 `glow` 原语本身的持续高亮不动）

调度器**不做语义判断**：不询问"为什么这一步要 pulse"、不依赖 `Primitive.Type` 的业务含义、不引用场景代码 / 算法字段。所有"何时高亮、何时触发"的决策由场景在构造 `RenderEnvelope` 时通过 MicroStep 显式表达——这属于场景作者的教学决策，详 §一 三层架构。

这条边界保证：新增场景或教师 L3 自定义场景**不需要修改前端调度器代码**。

## 五、场景时间控制模式

不是所有场景都需要完整的时间控制器。根据场景特性分为三种模式：

### 5.1 三种时间控制模式

| 模式 | 编码 | 适用场景 | 前端控件 | 说明 |
|------|------|---------|---------|------|
| **过程化** | process | 有明确步骤/阶段的场景 | 播放 / 暂停 / 单步前进 / 单步回退（仅单场景）/ 变速（含 0.5× 慢速）/ 重置 | PBFT 共识、交易生命周期等 |
| **交互响应式** | reactive | 输入即响应的场景 | 无时间控件，只有操作面板 | SHA-256 哈希、Merkle 树构建等 |
| **持续运行式** | continuous | 持续演化观察的场景 | 暂停/恢复/变速 | P2P 网络发现、Gossip 传播等 |

### 5.2 过程化模式控件规格

| 控件 | 功能 | 说明 |
|------|------|------|
| 播放 ▶ | 按当前速率自动推进 tick | 默认速率 1× |
| 暂停 ⏸ | 冻结当前 tick | — |
| 单步前进 ⏭ | 前进 1 个 tick | 暂停状态下可用 |
| 单步回退 ⏮ | 回退 1 个 tick | **仅单场景 process 模式启用**（联动组 / 多场景对照 / continuous / reactive 模式下按钮置灰）；最多回退 20 步 |
| 变速 | 0.5× / 1× / 1.5× / 2× | 四档切换（0.5× 为"慢速"，便于详看动画细节） |
| 重置 ⟲ | 回到初始状态（tick=0） | 重新初始化场景 |

**单步回退的限制范围**：

| 使用场合 | 是否启用 ⏮ | 原因 |
|---|---|---|
| A 单仿真 · process | ✅ 启用 | 学生看错一步可退一步 |
| B 多场景对照 · 任一 process | ❌ 置灰 | 多画布独立回退会造成跨画布因果弧线 M8 状态不一致 |
| C 联动组 · 任一场景 | ❌ 置灰 | 联动共享状态不可部分回退；需重启请走 重置 ⟲ |
| D 混合实验 | ❌ 置灰 | 容器侧真实链不可回退，仿真侧独立回退会双边偏离 |
| reactive / continuous 模式 | ❌ 不提供 | 该两模式本就不面向 tick 推进 |

WS 推送采用"snapshot + diff"模式，首连与重连后首帧 `IsFullSnapshot=true` 全量，后续增量推送；SimEngine Core 为每个单场景 process 会话维护 N=20 的 tick 快照环形缓冲区以支持单步回退。

### 5.3 微动画节奏（`MicroSteps` 量化）

场景动画通过 `RenderEnvelope.micro_steps[]` 协议字段下发（详 §6.2），该字段为动画描述列表，渲染器按顺序调度。

**基础配速原则（1×）**：让学生看清每一步（不匆忙），又不会等太久（不无聊、不中途走神）。以一节课 45 分钟内学生能完整体验 2-3 个场景 + 多轮反复为准。

| 模式 | 微动画数量 | 单 MicroStep（1×） | 单一主流程完成（1×） | 典型场景参考 |
|---|---|---|---|---|
| `process` | 每阶段 ≥ 3 子步 | **1.2-2.0s** | 30-50s | PBFT 5 阶段走完一轮 ≈ 40s |
| `continuous` | 每轮/周期 ≥ 3 子步 | **0.6-1.2s** | 5-10s | PoW 单出块周期 ≈ 6s |
| `reactive` | 每操作响应 ≥ 3 子步 | **0.3-0.8s** | 1-3s（单次） | SHA-256 单轮压缩 ≈ 1.5s |

**阶段切换过渡**：200-300ms 渐变过渡。**联动事件插入 MicroStep**（`is_link_trigger=true`）：固定 600-1000ms，不受场景节奏影响，以保证学生看到跨场景因果弧线。

**超长场景优化**（如 SHA-256 64 轮压缩）：场景可内置"重点详看 + 中间快进"脚本（场景级逻辑，非协议层），如默认 《前 3 轮详看 + 中间 60 轮按 0.1s 快过（压缩表示） + 末 3 轮详看》，总时 ≈ 20s。

变速只影响 tick 推进速率，不影响 MicroStep 子步骤的动画流畅度（子步 duration_ms 不被变速压缩）。场景开发者可为每个 MicroStep 赋予个性化 `duration_ms` 实现节奏差异。

### 6.1 核心组件

> SceneManager 的 K8s 编排细节详见 [`06.1-场景算法容器编排设计.md`](./06.1-场景算法容器编排设计.md)。

### 6.2 渲染数据契约 `RenderEnvelope`

场景算法容器通过 gRPC 向 SimEngine Core 返回的可渲染数据，统一封装为 `RenderEnvelope` 结构。Core 通过 WebSocket 转发给前端（§七）。

```go
type RenderEnvelope struct {
    // 视觉与动画
    Primitives    []Primitive       `json:"primitives"`               // 47 原语列表，必填
    MicroSteps    []MicroStep       `json:"micro_steps,omitempty"`    // 微动画序列（详 §5.3）
    LinkTriggers  []LinkTrigger     `json:"link_triggers,omitempty"`  // 来自联动组其他场景的因果事件（详 §八）
    ContainerData []ContainerMetric `json:"container_data,omitempty"` // 混合实验容器采集数据（详 §九）

    // 推送优化
    ChangedKeys    []string `json:"changed_keys,omitempty"`   // 本 tick 变更的 primitive ID，diff 推送提示
    IsFullSnapshot bool     `json:"is_full_snapshot,omitempty"` // 首连与重连后首帧为 true

    // 侧栏指标 / 文字面板用数据（非原语参数）
    Data map[string]any `json:"data,omitempty"`
}

type Primitive struct {
    ID           string         `json:"id"`                       // 稳定 ID，用于 diff 与 Playwright 测试定位
    Type         string         `json:"type"`                     // 47 原语 type 之一（如 "node" / "vote_matrix"）
    Layer        string         `json:"layer"`                    // background | content | effect | overlay（详 §3.3）
    Params       map[string]any `json:"params"`                   // 该原语的参数 schema（详原语库）
    Clickable    bool           `json:"clickable,omitempty"`      // 是否响应点击
    HoverTooltip string         `json:"hover_tooltip,omitempty"`  // 鼠标悬停 tooltip 文本
    ClickAction  string         `json:"click_action,omitempty"`   // 点击触发的 action_code（绑定 §6.3 InteractionDefinition）
}

type MicroStep struct {
    ID            string   `json:"id"`                       // 稳定 ID
    Label         string   `json:"label"`                    // 步骤名（侧栏显示）
    DurationMs    int      `json:"duration_ms"`              // 1× 速率下时长（详 §5.3）
    HighlightIDs  []string `json:"highlight_ids,omitempty"`  // 该步骤要高亮的 primitive ID 列表
    FirePrimitives []string `json:"fire_primitives,omitempty"` // 该步骤新增 / 触发的原语 ID
    IsLinkTrigger bool     `json:"is_link_trigger,omitempty"` // 是否为联动事件触发的临时步骤
    LinkSource    string   `json:"link_source,omitempty"`    // 联动事件来源场景 code
    ParentPhase   string   `json:"parent_phase,omitempty"`   // 所属阶段标识（详 §11.2 phase_progress 事件时间线）
}

type LinkTrigger struct {
    ID             string         `json:"id"`              // 事件唯一 ID（去重用）
    SourceScene    string         `json:"source_scene"`    // 触发源场景 code
    SourceAction   string         `json:"source_action"`   // 源动作 action_code
    LinkGroup      string         `json:"link_group"`      // 所属联动组（多组归属时按组分发）
    ChangedFields  []string       `json:"changed_fields"`  // 变更的 SharedState 字段列表
    Payload        map[string]any `json:"payload"`         // 字段新值
    Timestamp      int64          `json:"ts"`              // 事件时间戳
    SourceAnchorID string         `json:"source_anchor_id,omitempty"` // 源场景锚点原语 ID（M8 跨画布弧线起点）
    TargetAnchorID string         `json:"target_anchor_id,omitempty"` // 接收方场景锚点原语 ID（M8 弧线终点）
}

type ContainerMetric struct {
    SourceContainer string         `json:"source_container"` // Pod 内容器名（如 "geth-node-1"）
    MetricKey       string         `json:"metric_key"`       // 指标键（如 "block_height" / "tx_pool_size"）
    Value           any            `json:"value"`            // 指标值
    Timestamp       int64          `json:"ts"`
    TargetPrimitive string         `json:"target_primitive,omitempty"` // 映射到的原语 ID
    TargetParam     string         `json:"target_param,omitempty"`     // 映射到的原语参数键
}
```

#### 6.2.1 `SceneState` 与 `RenderEnvelope.Data` 的可见性边界

场景算法容器内部维护**两类不同的状态结构**，可见性与职责严格分层：

| 结构 | 可见范围 | 内容 | 用途 |
|---|---|---|---|
| `SceneState` | 仅场景容器与 SimEngine Core | **完整算法状态**（如 PBFT 的 `view / sequence / votes / leader / fault_count`、SHA-256 的 `round / w[64] / a..h / digest`、PoW 的 `nonce / difficulty / chain`） | 场景容器序列化往返于 gRPC `Init / Step / HandleAction`；前端不可见，前端皮肤不应也无法依赖任何场景特定字段名 |
| `RenderEnvelope.Data` | 前端可见，通过 WebSocket 转发 | **场景主动选择暴露给学生的展示数据**（key-value 字典） | 前端通用 `SidePanel` 组件按 key-value 渲染；前端不内置 `if (sceneCode === "pbft-consensus") {...}` 分支 |

**场景作者写入 Data 的规则**：

- 用人类可读的 key（如 `"view_number"` / `"current_round"`），或同时提供 `data_labels: {"view": "视图编号"}` 中文映射，标签直接出现在侧栏 UI 上
- 仅放学生需要看到的指标（如视图号 / 当前轮 / 摘要值 / 雪崩比特数），不放算法内部所有状态
- 建议 ≤ 20 个 key，避免侧栏拥挤；超过时考虑分组（用嵌套 map 或 `data_groups` 元信息）
- 不在 Data 里放与原语 `params` 重复的字段（这些已在 `Primitives` 中表达）

**这条分层的含义**：新场景接入或教师 L3 自定义场景**无需任何前端代码改动**——场景作者只需 (a) 实现算法 + 输出 RenderEnvelope；(b) 在 Data 暴露想展示的指标。前端皮肤永远不感知任何场景特定字段。

这是 §一 三层架构（教学决策层 / 协议契约层 / 皮肤层）在数据通道层面的具体落地。

### 6.3 学生交互动作 `InteractionDefinition`

场景算法容器通过 `GetInteractionSchema` gRPC 返回 `InteractionDefinition`，描述本场景所有学生可执行 action。前端按此动态生成交互面板。

```go
type InteractionDefinition struct {
    SceneCode     string      `json:"scene_code"`      // 场景编码
    SchemaVersion string      `json:"schema_version"`  // schema 版本号
    Actions       []ActionDef `json:"actions"`
}

type ActionDef struct {
    // 基础
    ActionCode  string     `json:"action_code"`              // 全局唯一标识，全小写下划线
    Label       string     `json:"label"`                    // 学生看到的按钮文字
    Description string     `json:"description,omitempty"`    // 鼠标悬停提示
    Category    string     `json:"category"`                 // param_tune | attack_inject | primary | observe
    Trigger     string     `json:"trigger"`                  // submit | immediate | hold
    Fields      []FieldDef `json:"fields"`                   // 输入字段

    // 权限与频次
    Roles      []string `json:"roles"`                     // ["student"] / ["teacher"] / ["student","teacher"]
    CooldownMs int      `json:"cooldown_ms,omitempty"`     // 冷却时间，防滥用

    // 联动传递
    LinkOwnerFields    []string `json:"link_owner_fields,omitempty"`    // 该 action 的"owner 写入声明"（必须 ⊆ 本场景 owner 字段集，详 §8.3）
    WritesOwnedFields  []string `json:"writes_owned_fields,omitempty"` // 本动作会修改的 owner 字段路径；Core 据此自动 fan-out 联动（取代旧 TriggersLinkGroups）

    // 回退与教师干预
    Reversible    bool   `json:"reversible,omitempty"`      // 是否可参与 process 模式回退栈（container 通道或有外部副作用的动作必须 false）
    InterveneType string `json:"intervene_type,omitempty"`  // 仅 Roles 含 teacher 时必填；Core 写审计日志分类（hint / fault / attack / phase / topology / state / reset / epoch / revert / freeze）

    // 混合实验通道
    HybridChannel string `json:"hybrid_channel,omitempty"`   // sim（默认） | container（混合实验真链通道）
    ContainerCmd  string `json:"container_cmd,omitempty"`    // 容器通道命令模板，含 {{field_name}} 占位符
}

type FieldDef struct {
    Name        string `json:"name"`
    Type        string `json:"type"`                      // string | number | boolean | select | enum | range | json | multi_select
    Label       string `json:"label"`
    Required    bool   `json:"required,omitempty"`
    Default     any    `json:"default,omitempty"`
    Min         any    `json:"min,omitempty"`
    Max         any    `json:"max,omitempty"`
    Step        any    `json:"step,omitempty"`
    Options     []any  `json:"options,omitempty"`         // select / enum 选项
    OptionsFrom string `json:"options_from,omitempty"`    // 动态从 RenderState 读取的字段路径（如 "state.nodes"）
}
```

**`Category` 业务语义**：

| category | 颜色 | 含义 | 例 |
|---|---|---|---|
| `param_tune` | 蓝色 [⚙ 调参] | 改数值参数，改变仿真行为 | 调矿工算力 / 改 RSA 密钥位数 |
| `attack_inject` | 红色 [⚠ 攻击] | 注入恶意行为 | 51% 攻击 / 拜占庭节点 / 重入攻击 |
| `primary` | 绿色 [▶ 操作] | 走正常流程的主动动作 | 发起 TX / 签名 / 投票 / 部署合约 |
| `observe` | 灰色 [👁 观察] | 切视图、不改状态 | UTXO/Account 切换 / 攻防对比双视图 |

`attack_inject` 按钮额外加二次确认，防误触。

**InteractionSchema 客户端缓存**：HTTP GET `/api/v1/experiment-instances/:id/sim-scenes/:scene_code/interaction-schema`，浏览器缓存 24h；教师调试场景时通过 WS `schema_invalidated` 信号触发前端拉取最新 schema。

### 6.4 SimEngine Core 与场景算法容器 gRPC 服务

```protobuf
service SimScenario {
    rpc Init(InitRequest) returns (RenderEnvelope);                       // 初始化
    rpc Step(StepRequest) returns (StepResponse);                          // 单 tick 步进（process/continuous）
    rpc HandleAction(ActionRequest) returns (ActionResponse);              // 处理学生动作
    rpc GetInteractionSchema(Empty) returns (InteractionDefinition);       // 获取 ActionDef 列表
    rpc GetMeta(Empty) returns (ScenarioMeta);                             // 元信息
    rpc HealthCheck(Empty) returns (HealthResponse);                       // 健康检查
}

message ScenarioMeta {
    string code               = 1;
    string name               = 2;
    string category           = 3; // 9 选 1（详 §3.5）
    string algorithm_type     = 4;
    string description        = 5;
    string version            = 6;
    string time_control_mode  = 7; // process | reactive | continuous
    string data_source_mode   = 8; // simulation | collection | dual
    bytes  default_params     = 9;
    string custom_renderer_package = 10; // L3 npm 包名（仅 category=generic 时使用）
}

message StepRequest {
    bytes state        = 1;
    int64 tick         = 2;
    bytes shared_state = 3; // 联动共享状态（仅联动组场景）
}

message StepResponse {
    bytes  state              = 1;
    bytes  render_envelope    = 2; // 序列化的 RenderEnvelope（§6.2）
    bytes  shared_state_diff  = 3; // 联动共享状态变更（仅联动组场景）
}

message ActionRequest {
    bytes  state       = 1;
    string action_code = 2;
    bytes  params      = 3; // ActionDef.Fields 的字段值（JSON）
    string user_role   = 4; // student | teacher
    string actor_id    = 5; // 多 actor 会话中的 actor 标识
}

message ActionResponse {
    bytes  state              = 1;
    bytes  render_envelope    = 2; // 序列化的 RenderEnvelope（§6.2）
    bool   success            = 3; // 操作是否成功
    string error_message      = 4; // 失败时的错误描述
    bytes  shared_state_diff  = 5; // 联动共享状态变更（仅联动组场景，同 StepResponse）
}
```

### 6.5 状态推送与单步回退策略

平台不提供"任意 tick 跳转"能力。仅支持**单步回退**（限单场景 process 模式，详 §5.2）。

| 策略 | 说明 |
|---|---|
| 推送模式 | "snapshot + diff"。首连与重连后首帧 `IsFullSnapshot=true` 全量推送；后续每 tick 只发增量 `ChangedKeys` |
| 单步回退缓冲 | 单场景 process 会话维护 N=20 的 tick 环形快照缓冲（约 64KB / 场景）；超出后旧快照丢弃 |
| 重置重建 | 重置 ⟲ 时丢弃所有快照，重新调用 `Init` 并下发新 `IsFullSnapshot=true` 帧 |
| 暂停 | 不写持久化存储；暂停只是冻结 SimClock，状态仍在内存 |
| 重连 | 客户端断线重连后，Core 立即下发当前 tick 的 full snapshot |

### 6.6 多人协作仿真状态同步

多人组网实验（拓扑模式 3）中，多个学生可能同时操作同一个仿真会话：

| 机制 | 说明 |
|------|------|
| 操作冲突解决 | 按服务端收到的时间顺序执行，乐观锁 |
| 操作广播 | A 学生的操作结果实时推送给同组所有学生 |
| 角色权限 | 不同角色可执行不同操作（如角色 A 控制节点 1-3，角色 B 控制节点 4-6） |
| 状态一致性 | 所有学生看到的仿真状态由服务端统一管理，前端只渲染 |

## 七、WebSocket 通信协议

### 7.1 连接地址

```
wss://api.lianjing.com/api/v1/ws/sim-engine/:session_id?token=<jwt>
```

> `session_id` 为 SimEngine 会话 ID，在启动实验时由 SimEngine Core 分配。前端浏览器 WebSocket 连接通过 query token 完成鉴权；服务端实现时需校验该 token 与实验实例、会话归属关系。

### 7.2 消息格式

```json
{
  "type": "render | event | control_ack | schema_invalidated | action | control | step_back",
  "scene_code": "pbft-consensus",
  "tick": 42,
  "timestamp": 1712500000000,
  "payload": {}
}
```

`payload` 类型按 `type` 决定（详 §7.3 / §7.4）。

### 7.3 后端 → 前端消息

| type | 说明 | payload 类型 |
|---|---|---|
| `render` | 渲染数据下发（每 tick / Action 响应 / 重连首帧） | `RenderEnvelope`（详 §6.2，含 primitives / micro_steps / link_triggers / container_data / changed_keys / is_full_snapshot） |
| `event` | 仿真事件通知（侧栏日志 / 提示） | `{"event": string, "data": {...}}` |
| `control_ack` | 控制指令确认 | `{"command": string, "success": bool, "error"?: string}` |
| `schema_invalidated` | 教师调试场景导致 ActionDef 变更，提示前端重新拉取 InteractionDefinition | `{"reason": "teacher_published"}` |

> `render` 是唯一的渲染数据通道，统一承载全量首帧（`is_full_snapshot=true`）与增量帧。前端按 `changed_keys` 做局部 patch。

### 7.4 前端 → 后端消息

| type | 说明 | payload 类型 |
|---|---|---|
| `action` | 用户交互操作 | `{"action_code": string, "params": {...}}`（params 字段对齐 `ActionDef.Fields`） |
| `control` | 仿真控制 | `{"command": "play" \| "pause" \| "step" \| "set_speed" \| "reset" \| "resume", "value"?: number}` |
| `step_back` | 单步回退（仅单场景 process 模式） | `{}`（无 payload，每次回退 1 个 tick；服务端从环形快照缓冲恢复，缓冲耗尽则返回 `control_ack {success:false, error:"buffer_exhausted"}`） |

**control 命令约束**：

| command | 适用模式 | 说明 |
|---|---|---|
| `play` | process | 播放 |
| `pause` | process / continuous | 暂停 |
| `step` | process | 单步前进 1 tick |
| `set_speed` | process / continuous | 变速；`value ∈ {0.5, 1, 1.5, 2}` |
| `reset` | process | 重置到 tick=0，丢弃所有快照 |
| `resume` | continuous | 恢复持续运行 |

> `process` 模式用 `play` 启动播放，不用 `resume`；`continuous` 模式用 `resume` 恢复，不用 `play`。
> `step_back` 仅在单场景 process 会话有效；联动 / 多场景对照 / 混合实验 / continuous / reactive 模式下服务端直接返回 `control_ack {success:false, error:"step_back_not_allowed_in_<context>"}`，前端 ⏮ 按钮 UI 层应已置灰，此为服务端兜底校验。

## 八、联动机制设计

### 8.1 联动原理

联动不是"通知机制"，而是**共享状态空间**。同一联动组内的场景读写同一份底层状态对象，一个场景的操作自然影响其他场景的仿真结果。

```
┌─────────────┐     ┌─────────────────────┐     ┌─────────────┐
│ PoW挖矿场景  │────▶│   共享状态空间         │◀────│ 51%攻击场景  │
│ 读写: nodes, │     │ nodes: {...}         │     │ 读写: nodes, │
│ blockchain   │     │ blockchain: {...}    │     │ blockchain   │
└─────────────┘     │ network: {...}       │     └─────────────┘
                    │ globalParams: {...}  │
┌─────────────┐     │                      │     ┌─────────────┐
│ 区块链分叉场景│────▶│                      │◀────│ 网络传播场景  │
│ 读: blockchain│    └─────────────────────┘     │ 读写: network│
└─────────────┘                                  └─────────────┘
```

### 8.2 联动组定义（9 组）

| # | 联动组 | 编码 | 包含场景 | 共享状态 | 联动效果 |
|---|--------|------|----------|----------|----------|
| 1 | PoW 攻击组 | pow-attack-group | CON-01 + ATK-01 + DAT-01 + NET-05 | nodes, blockchain, network | 调整攻击者算力→挖矿竞争变化→链分叉→网络传播延迟 |
| 2 | PBFT 攻击组 | pbft-attack-group | CON-03 + ATK-03 + NET-03 | nodes, messages, view | 注入拜占庭节点→投票异常→视图切换→网络消息变化 |
| 3 | Raft 容错组 | raft-fault-group | CON-04 + NET-03 + NET-05 | nodes, terms, logs | 宕掉 Leader→重新选举→网络分区→脑裂场景 |
| 4 | 密码学验证组 | crypto-verify-group | CRY-01 + CRY-03 + CRY-05 | hashes, keys, tree | 修改数据→哈希变化→签名失效→Merkle 路径变化 |
| 5 | 区块链完整性组 | blockchain-integrity-group | DAT-01 + TXN-01 + CRY-05 + DAT-02 | blocks, transactions, merkle_root | 发起交易→打包入块→Merkle 树更新→链结构增长 |
| 6 | 合约安全组 | contract-security-group | ATK-04 + ATK-05 + SC-01 + SC-02 | contract_state, call_stack, balances | 发起攻击→调用栈展开→状态异常变化→余额被盗 |
| 7 | PoS 经济组 | pos-economy-group | CON-02 + ECO-01 + ECO-03 + ECO-02 | stakes, token_supply, proposals | 质押变化→验证者权重变化→投票权变化→治理结果变化 |
| 8 | 交易处理组 | tx-processing-group | TXN-01 + TXN-02 + TXN-03 + TXN-04 | mempool, gas, balances | 交易入池→Gas 计算→排序打包→余额变更 |
| 9 | 网络基础组 | network-base-group | NET-01 + NET-02 + NET-06 | topology, routing_table, load | 节点发现→消息传播→负载均衡 |

### 8.3 字段 Owner 模型（每字段唯一 writer）

每个 LinkGroup 的 `shared_state_schema` 中，**每个字段必须显式指定唯一 owner 场景（即 writer）**：

```json
// 例：pbft-attack-group 的 schema
{
  "code": "pbft-attack-group",
  "shared_state_schema": {
    "view_number":      { "type": "int",      "owner": "pbft-consensus" },
    "byzantine_nodes":  { "type": "string[]", "owner": "pbft-byzantine" },
    "partition_active": { "type": "bool",     "owner": "network-partition" }
  }
}
```

**约束**：

- 每字段全平台**唯一 writer**，其他成员场景只能 read，不能 write
- 平台静态校验：CI 启动时扫描所有场景 catalog，发现违例直接拒绝注册
- 运行时校验：Core 接收 SyncSharedState 时校验，违例拒收并记录审计日志
- ActionDef 协议层 `link_owner_fields[]` 必须 ⊆ 该场景在对应 LinkGroup 的 owner 字段集
- 不允许"按注册顺序最后写者胜出"等妥协逻辑

### 8.4 N:M 多组归属

一个场景可同时属于多个 LinkGroup（如 `network-partition` 同属 pbft-attack-group + raft-fault-group）：

- **字段 owner 在不同组里独立判定**：同一场景在 A 组里 own 字段 X，在 B 组里 own 字段 Y，但**不能在两组同时 own 同名字段**（schema 设计时由平台审核避免）
- **`LinkTriggers` 协议字段按 link_group 分组**：每事件必填 `link_group` 字段，接收方按 group 分别处理
- **画布上多个 link_indicator 徽章**：场景画布角落为每个所属组放一个徽章，并排或堆叠显示
- **N:M Fan-out**：任一成员写 SharedState → Core fan-out 到该组**所有其他成员**；事件只在源组内传播，不跨组扩散

### 8.5 联动可视化机制

教学价值的关键：**学生必须能看到跨场景因果**，否则联动只是后台数据流。

#### 8.5.1 单场景画布内（两件套原语）

| 原语 | 用途 |
|---|---|
| `link_indicator`（layer=overlay） | 画布角落徽章，显示当前联动组与最近一次外部事件 |
| `external_event_marker`（layer=overlay） | 临时浮窗 / 边缘闪光，标注"刚刚因联动场景 X 的事件而更新"，自动 fade out |

#### 8.5.2 协议层 `LinkTriggers`

接收方场景的 `RenderEnvelope.link_triggers[]` 携带跨场景因果事件（详 §6.2 LinkTrigger 结构）。场景容器在 `Step` 时根据该数组：

- **方式 A 插入临时 MicroStep**（推荐，最显式，固定 600-1000ms）
- **方式 B 直接更新原语参数**（隐式，适合连续型联动如 hashrate 数值变化）
- **方式 C 触发 link_indicator 徽章闪烁**（最弱，仅作背景提示）

#### 8.5.3 跨画布因果弧线（M8，多场景对照启用）

详见 §九 多场景对照机制。

### 8.6 教师扩展联动组（接入或新建）

**接入既有 9 组**：教师上传场景时在 meta 声明 `link_groups: ["pos-economy-group"]`：

1. 教师代码声明本场景在该组里 read 哪些字段、是否申请成为某字段的 owner
2. 平台审核：字段 owner 是否冲突 / `SyncSharedState` 实现是否合规 / 字段类型是否匹配
3. 审核通过：写入 `sim_link_group_scenes` 关联表
4. 失败：返回字段冲突报告

**新建联动组**：教师发现既有组都不合适时：

1. 教师提交新组申请（包含 code / name / description / shared_state_schema 含每字段 owner / 成员场景列表）
2. 平台审核：每字段 owner 唯一 / 是否与既有组语义重叠（建议优先扩展既有）/ 成员实现一致
3. 审核通过：插入 `sim_link_groups` + `sim_link_group_scenes`

### 8.7 schema 不可演进

LinkGroup 的 `shared_state_schema` 一旦定义并启用，**完全不可变**。落库由 `sim_link_groups.code + version` 双唯一索引强制（详 [`02-数据库设计.md`](./02-数据库设计.md) §2.9）：

- 新增字段 / 删除字段 / 重命名字段 / 改类型 / 改 owner —— **全部禁止**
- 如需变 → 走"新增版本"路径：
  1. 在 `sim_link_groups` 表新增 row：相同 `code`，新 `version`（如 `1.0.0` → `2.0.0`）
  2. 原成员场景 catalog 决定是否加入新版本组
  3. 旧版本 row `status` 改为 `deprecated`，不再分发到新建实验
  4. 既有运行实例不受影响（`template_sim_scenes.link_group_version` 已锁定旧版本）
- 这样保证联动组的"语义稳定性"，避免渐进式 schema 腐化

### 8.8 联动与时间控制兼容性

| 组合 | 行为 |
|---|---|
| process + continuous 联动 | 以 continuous 场景的固定周期为基准时钟，process 场景按阶段对齐到周期边界 |
| reactive 场景参与联动 | reactive 的 `HandleAction` 触发的状态变更立即同步到 SharedState，下一 tick 起对联动方可见 |
| 联动暂停 | 任一场景暂停（教师暂停或学生暂停）→ 整个联动组暂停 |
| 联动重置 | 联动组重置 = 所有成员场景重置 + SharedState 重置 |
| 联动单步回退 | **禁用**（详 §5.2 ⏮ 限制范围 C） |

### 8.9 联动可选

- 教师在实验编排器中选择仿真场景
- 如果选择了同一联动组内的多个场景，平台**自动提示**"检测到可联动场景，是否启用联动？"
- 教师可选择 **启用联动** 或 **独立运行**
- 单场景始终可以独立运行，不强制联动
- 学生端不感知联动配置，只看到多个场景面板自然联动或独立运行

## 九、多场景对照机制

### 9.1 痛点与目标

学生 / 教师在同一实验内并排查看多个场景（如 PoW + 51% 攻击 + Block-Sync）时，跨画布的因果可视化是教学最大价值。本章定义同实验并排展示与跨画布因果弧线（**机制 M8**）的协议与实现。

### 9.2 实验配置层（教师编排）

在 `template_sim_scenes` 关联表配置每个场景在该实验中的展示参数：

| 字段 | 取值 | 说明 |
|---|---|---|
| `scene_id` | 场景 ID | 实验包含的场景 |
| `layout_role` | `primary` / `secondary` / `auxiliary` | 布局角色 |
| `display_mode` | `single` / `split-2` / `split-3` / `grid-4` | 该场景在该实验中的展示模式 |
| `link_to_primary` | bool | 是否需要与主场景时间轴同步 |
| `default_visible` | bool | 实验启动时是否默认展开 |

学生在实验详情页可手动切换展示模式（教师未禁用时）。

### 9.3 时间控制策略

| 场景关系 | 时间控制 |
|---|---|
| 同 LinkGroup 内多场景（如 PoW + 51% 攻击） | **强制时钟同步**，全局 Play / Pause / Step / Speed 作用于全部成员 |
| 不同 LinkGroup 但同实验 | 教师配置：默认共享时钟，可改独立时钟 |
| 非联动场景同实验 | 默认独立时钟，每场景自己的播放控制 |

实现：SimEngine Core 维护"会话时间轴 SessionClock"，强制同步的场景共享同一 SessionClock 实例。

### 9.4 跨画布因果弧线（机制 M8）

单场景画布角落的 `link_indicator` 徽章在多场景并排时不够直观，必须有跨画布的事件传播线。

**实现方式**（不是新原语，是渲染器层之上的一层 overlay）：

1. **多场景容器组件**：前端实现 `MultiSceneContainer`，在所有场景画布之上叠加全局 SVG `<svg class="cross-canvas-overlay">`
2. **数据来源**：`link_trigger` 事件到达接收方场景时，如果检测到 `source_scene` 也在当前并排视图中，触发跨画布绘制
3. **绘制内容**：
   - 起点：source_scene 画布上的某锚点（由 `link_trigger.source_anchor_id` 指定，如 attacker 矿工 bar）
   - 终点：本场景画布上的某锚点（由 `link_trigger.target_anchor_id` 指定，如 attacker chain 节点）
   - 形态：贝塞尔弧线 + 头部箭头 + 短文字标签（如 "↗ hashrate 0.55"）
   - 持续：1.5-2s 后 fade out
4. **协议要求**：`LinkTrigger.source_anchor_id` 与 `target_anchor_id` 必填（详 §6.2）
5. **降级**：source_scene 不在当前并排视图时，退化到画布内的 `external_event_marker` + `link_indicator`（即 §8.5.1 单场景机制）

### 9.5 SharedStatePanel 聚合面板

多场景对照时学生可能想看"全联动组所有共享字段当前值 + owner 标签"：

- 顶部 [📊] 按钮触发右侧抽屉（width 360px）
- 抽屉内分组显示：每个 LinkGroup 一个手风琴 section
- 每个字段一行：字段名 + 当前值 + owner 场景 + 字段值变化 1 秒高亮
- 多组归属（N:M）抽屉内每组独立 section

教师默认抽屉可见，学生默认隐藏（教师可配置开放给学生）。

### 9.6 联动状态传播流程（更新版）

```
场景A执行 Step / HandleAction
    │
    ▼
场景容器 SyncSharedState（仅写自己 owner 的字段）
    │
    ▼
LinkEngine 将 diff 合并到共享状态空间，校验字段 owner
    │
    ▼
LinkEngine fan-out 到同组其他成员场景，构造 LinkTrigger 事件
（含 source_anchor_id / target_anchor_id）
    │
    ▼
其他场景下一次 Step 接收 link_triggers[]，按 §8.5.2 三方式之一可视化
    │
    ▼
RenderEnvelope 推送给前端 → 单场景画布内 link_indicator + 跨画布弧线 M8
```

## 十、混合实验数据采集

### 10.1 采集架构

混合实验中，真实链节点运行在实验环境容器中，SimEngine 通过 Collector Agent（sidecar 容器）采集实时数据进行可视化。

```
┌──────────────────────────────────────────────────┐
│              实验环境容器组                         │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌────────────────┐  │
│  │ Geth节点1 │  │ Geth节点2 │  │ Collector Agent│  │
│  │          │  │          │  │ (sidecar容器)   │  │
│  └────┬─────┘  └────┬─────┘  └───────┬────────┘  │
│       │              │                │            │
│       └──────────────┴────────────────┘            │
│              容器内部网络                            │
└──────────────────────────┬───────────────────────┘
                           │ WebSocket
┌──────────────────────────┴───────────────────────┐
│  SimEngine Core → Collector 模块                   │
│  接收采集数据 → 转换为 RenderState → 推送前端渲染    │
└──────────────────────────────────────────────────┘
```

### 10.2 Collector Agent 标准化接口

Collector Agent 是一个轻量 sidecar 容器，负责从同 Pod 内的链节点采集数据并转换为标准格式。

```json
// Collector Agent 输出的标准化数据格式
{
  "source": "ethereum",
  "timestamp": 1712500000000,
  "data_type": "block | transaction | peer | consensus | contract_event",
  "payload": {
    // 根据 data_type 不同而不同
  }
}
```

### 10.3 链生态采集适配器

| 链生态 | 采集方式 | 采集内容 |
|--------|---------|---------|
| Ethereum (Geth) | JSON-RPC + WebSocket 订阅 | 区块、交易、节点列表、合约事件 |
| Hyperledger Fabric | gRPC EventHub | 区块、交易、链码事件、通道状态 |
| ChainMaker | SDK 订阅 | 区块、交易、合约事件 |
| FISCO BCOS | JSON-RPC + AMOP | 区块、交易、共识状态 |

平台内置以上四种链生态的 Collector Agent 镜像，教师可通过 SDK 开发自定义采集适配器。

## 十一、数据展示规范

### 11.1 设计原则

数据融入可视化本身，而非旁边挂一个独立数据面板。

### 11.2 数据展示方式

| 方式 | 依赖原语 | 说明 | 适用场景 |
|---|---|---|---|
| **内嵌指标卡片** | `label`（带 style）/ `progress_bar` | 画布内嵌关键指标（如"当前轮次: 3"） | 所有场景 |
| **悬停详情** | `Primitive.HoverTooltip` / `tooltip` 原语 | 鼠标悬停节点 / 连线 / 区块显示 tooltip | 所有场景 |
| **状态变化高亮** | `glow` + `MicroStep.highlight_ids` | 变化字段闪烁 / 变色 | 所有场景 |
| **事件时间线** | `phase_progress` + `MicroStep.parent_phase` | 画布下方简洁事件时间线，可点击 tick 跳转 | process 模式 |
| **对比面板** | `horizontal_lane` × N（多泳道并排） | A/B 对比视图 | 密码学雪崩效应 / 攻防对比 |
| **错误 / 故障指示** | `error_overlay` | 容器故障 / Collector 采集失败 / gRPC 超时 / 真链容器 down；不自动 fade，需故障恢复或手动 dismiss | 混合实验 / 所有依赖外部资源场景 |
| **数学公式渲染** | `math_formula`（KaTeX） | LaTeX 语法公式渲染 | 密码学 6 场景（SHA-256 / Keccak / ECDSA / RSA / Merkle / ZKP） |
| **联动徽章** | `link_indicator` / `external_event_marker` | 画布角落联动状态 + 临时事件浮窗 | 所有联动场景 |

### 11.3 教师课堂标注能力（`annotation` 原语扩展）

教师可在可视化上实时添加标注（箭头 / 圆圈 / 方框 / 自由笔 / 文字），学生同屏可见。`annotation` 原语参数：

```go
type AnnotationParams struct {

    OwnerRole  string         `json:"owner_role"`  // teacher（仅教师可创建）

    Text       string         `json:"text,omitempty"`
}
```

指令通过教师干预面板项目下发，全班学生同步可见。

## 十四、扩展规范（SimScenario SDK）

### 14.1 自定义场景开发流程

```
教师开发场景算法（任意语言，实现 gRPC SimScenario 接口）
    │
    ▼

    │
    ▼

    │
    ▼

    │
    ▼
审核通过 → 上架到场景库，教师可在编排时选用
```

### 14.2 场景算法容器基础镜像

平台提供多语言基础镜像模板，内置 gRPC 服务框架：

| 语言 | 基础镜像 | 体积 |
|------|---------|------|
| Go | sim-base-go:latest | ~15MB |
| Python | sim-base-python:latest | ~40MB |
| Rust | sim-base-rust:latest | ~10MB |
| TypeScript/Node | sim-base-node:latest | ~35MB |

### 14.3 L3 自定义渲染器（< 1% 场景）

仅在场景视觉范式完全不属 8 类目、且 47 原语无法表达时走 L3 路径（详 §3.7）。

```typescript
interface CustomRenderer {
  domain: string;           // 仅允许为 `generic`，不拓展新 category
  name: string;             // 渲染器名称
  render(envelope: RenderEnvelope, ctx: CanvasRenderingContext2D): void;
  handlePrimitiveClick(primitive: Primitive, event: MouseEvent): { action_code: string, params: any } | null;
}
```

**上传路径**：

1. 教师开发 npm 包实现 `CustomRenderer` 接口
2. 上传至平台 npm 私有仓库（名空间 `@chainmirror/renderer-<school_id>__<package_name>`）
3. 管理员审核：安全扫描（无恶意代码） + 沙箱测试（可运行） + 原语覆盖评估（是否应走 L1）
4. 审核通过 → 注册到 `DomainRendererRegistry`
5. 教师在场景 `meta.custom_renderer_package` 字段引用包名

**约束**：

- `sim_scenarios.category` 必须为 `generic`，不允许拓展 9 选 1 之外的 category（避免渲染器碎片化）
- L3 场景仍需遵守 `RenderEnvelope` 协议与 ActionDef schema（仅渲染实现可自定义）
- L3 场景默认不允许加入联动组（如需加入，走 §8.6 教师扩展联动组审核路径）

### 14.4 教师自定义 action（命名空间 + 审核）

教师可为自定义场景声明新的 ActionDef，以避免与平台内置 action_code 冲突：

- **命名空间**：`teacher_<school_id>__<custom_code>`（两个下划线隔开，`<custom_code>` 遵平台 action_code 命名规范）
- **范围**：仅在本场景生效，不会注册到全局 action_code 表
- **审核项**：
  - `category` 必须是 4 类之一（param_tune / attack_inject / primary / observe）
  - `roles` 不得含 `super_admin`
  - `triggers_link_groups` / `link_owner_fields` 必须通过字段 owner 静态校验
  - `hybrid_channel = container` 时，`container_cmd` 模板必须通过平台沙箱白名单（仅允许预定义容器命令前缀，如 `geth` / `solana-cli` / `fabric-peer`；禁用 shell 元字符 `;` `&&` `|` `\`` `$()`）
  - 学生输入参数 `{{xxx}}` 插值时强制 shell-escape

### 14.5 教师跨学生干预 action（独立面板，不进 InteractionDefinition）

教师课堂上可能需要跨全班干预（广播 / 强制暂停 / 帮学生跨卡点 / 调试联动）。这类 action 不在场景 InteractionDefinition 中声明（学生看不到），走独立接口 `POST /api/v1/teacher/experiments/:id/intervene`：

| action_code | 影响范围 | 用途 | category |
|---|---|---|---|
| `broadcast_message` | 全班学生 | 推文字提示 | primary |
| `pause_all` | 全班学生 × 全部场景 | 集中讲解时暂停 | primary |
| `resume_all` | 全班学生 × 全部场景 | 恢复 | primary |
| `force_step` | 指定学生 + 指定场景 | 帮学生跨过卡点 | primary |
| `force_reset` | 指定学生 + 指定场景 | 强制重做 | primary |
| `force_link_state` | 指定 LinkGroup + 字段 | 调试联动（紧急后门） | param_tune |
| `unlock_link_sync` | 指定 LinkGroup | 临时解除联动时钟同步·应急 | primary |
| `kick_student` | 指定学生 | 踢出实验（违规） | primary |

## 十五、与现有编排流程的集成

教师在实验模板创建向导（P-21）的步骤三"工具与可视化"中：

1. 选择实验类型（纯仿真 / 真实环境 / 混合）

学生端全自动：启动实验 → 后端初始化仿真会话 + 启动场景算法容器 → 前端自动加载领域渲染器 → 开始交互。

### 17.2 联动场景五锚点联动

联动场景必须满足 **B5 / D3c / E1 / E4 / E7** 五锚点自洽：

- **B5** 自然语言写"学生在 X 场景做 Y 操作 → 本场景画布发生什么具体视觉变化"
- **D3c** 列出"事件 source_scene + source_action → 高亮 primitives ID + MicroStep label"映射表
- **E1** 必须包含至少一个 `link_indicator` 原语
- **E4** 必须包含至少一个 `is_link_trigger=true` 的临时 MicroStep 模板
- **E7** 必须明示哪些字段是本场景 owner（write）/ 哪些 read（owner 在其他成员场景）
