# 实验环境模块 — 实验类型与环境配置

> 模块状态：✅ 已确认
> 文档版本：v1.0

## 一、文档概述

本文档是实验环境模块的补充设计文档，聚焦于实验类型体系、镜像配置模板、环境编排机制、配置验证策略等核心设计决策。与 `01-功能需求说明.md` 中的功能列表互补，本文档回答"如何确保教师自由编排的实验环境真正可用"这一核心问题。

### 核心设计原则

- **平台提供零件库，教师组装实验**：平台不预设实验模板，提供完整的镜像库、工具库、仿真场景库，教师根据教学需要自行编排
- **配置即可用**：教师通过编排器配置的实验环境必须能真正运行，不能只是一堆工具的堆砌
- **快速启动**：官方镜像全量预拉取，学生启动实验无需等待镜像下载
- **安全防护**：5层配置验证机制，防止无效配置进入生产环境

## 二、三种实验类型

### 2.1 类型定义

| 类型 | 编码 | 启动时间 | 资源消耗 | 适用场景 |
|------|------|----------|----------|----------|
| 纯仿真实验 | type=1 | < 1s | 极低（轻量场景算法容器 10-50MB） | 原理认知、算法理解、过程可视化 |
| 真实环境实验 | type=2 | 30s-2min | 中-高（K8s容器集群） | 开发实践、部署运维、性能测试 |
| 混合实验 | type=3 | 30s-2min | 中-高 | 综合实战、数据采集驱动可视化 |

### 2.2 类型与配置项关系

| 配置项 | 纯仿真(1) | 真实环境(2) | 混合(3) |
|--------|-----------|-------------|---------|
| 仿真场景选配 | ✅ 必选 | ❌ 不可选 | ✅ 可选 |
| 容器镜像选配 | ❌ 不可选 | ✅ 必选 | ✅ 必选 |
| 拓扑模式选择 | ❌ 不需要 | ✅ 必选 | ✅ 必选 |
| 工具选配 | ❌ 不需要 | ✅ 可选 | ✅ 可选 |
| 初始化脚本 | ❌ 不需要 | ✅ 可选 | ✅ 可选 |
| 资源限制配置 | ❌ 不需要 | ✅ 必选 | ✅ 必选 |
| 联动组配置 | ✅ 可选 | ❌ 不可选 | ✅ 可选 |
| 数据采集通道 | ❌ 不需要 | ❌ 不需要 | ✅ 可选 |
| K8s编排配置 | ❌ 不生成 | ✅ 自动生成 | ✅ 自动生成 |
| 检查点（脚本） | ❌ | ✅ | ✅ |
| 检查点（SimEngine状态断言） | ✅ | ❌ | ✅ |
| 手动评分项 | ✅ | ✅ | ✅ |

### 2.3 纯仿真实验

- 完全由 SimEngine 驱动，无需 K8s 容器
- 学生点击启动后，SimEngine Core 初始化会话，按需启动场景算法容器（10-50MB 轻量镜像）
- 即开即用（< 1s），资源消耗极低
- 适用于：共识算法原理、密码学运算过程、数据结构可视化、攻击原理演示等
- 检查点通过 SimEngine 状态断言验证（如"PBFT是否完成三阶段共识"）

### 2.4 真实环境实验

- 完全由 K8s 容器驱动，教师从镜像库选择镜像组合
- 学生启动后创建独立 K8s Namespace，容器按依赖顺序启动
- 启动时间 30s-2min（镜像已预拉取）
- 适用于：智能合约开发部署、链节点搭建运维、DApp开发、性能测试等
- 检查点通过在容器内执行脚本验证

### 2.5 混合实验

- 容器环境 + SimEngine 联动，两者通过 Collector Agent 桥接
- 容器中运行真实链节点，Collector Agent sidecar 实时采集链上数据
- 采集数据推送到 SimEngine 前端渲染层，驱动可视化面板
- 适用于：在真实环境中操作，同时通过可视化观察底层原理
- 检查点支持脚本验证和 SimEngine 状态断言两种方式

## 四、镜像配置模板机制

### 4.1 设计目标

每个官方镜像携带预定义的配置模板，包含默认端口、环境变量、依赖关系、资源建议、典型搭配等元数据。教师选择镜像时，平台自动填充默认配置，教师可在此基础上调整。

### 4.2 配置模板数据结构

镜像主表（images）扩展以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| default_ports | JSONB | 默认端口映射 |
| default_env_vars | JSONB | 默认环境变量（支持条件变量） |
| default_volumes | JSONB | 默认挂载卷 |
| typical_companions | JSONB | 典型搭配镜像列表 |
| required_dependencies | JSONB | 必要依赖镜像列表 |
| resource_recommendation | JSONB | 建议资源配置 |
| documentation_url | VARCHAR(500) | 详细文档链接 |

### 4.3 典型搭配（typical_companions）

描述镜像的常见搭配关系，分三个级别：

```json
{
  "required": [
    {"image": "zookeeper", "reason": "Kafka 必须依赖 ZooKeeper 进行集群协调"}
  ],
  "recommended": [
    {"image": "couchdb", "reason": "推荐使用 CouchDB 作为 Fabric 富查询状态数据库"}
  ],
  "optional": [
    {"image": "fabric-explorer", "reason": "可选：提供区块链浏览器方便查看链上数据"}
  ]
}
```

- **required**：缺少则无法正常运行，验证时强制检查
- **recommended**：推荐搭配，验证时给出建议提示
- **optional**：可选增强，仅在镜像文档中展示

### 4.4 条件环境变量

环境变量支持根据实验中其他容器的存在动态调整：

```json
[
  {
    "key": "CORE_LEDGER_STATE_STATEDATABASE",
    "default_value": "goleveldb",
    "conditions": [
      {
        "when": "container_exists:couchdb",
        "value": "CouchDB",
        "inject_vars": [
          {"key": "CORE_LEDGER_STATE_COUCHDBCONFIG_COUCHDBADDRESS", "value": "${COUCHDB_HOST}:5984"}
        ]
      }
    ],
    "description": "状态数据库类型，检测到 CouchDB 容器时自动切换"
  }
]
```

## 五、K8s Namespace 隔离与服务发现

### 5.1 Namespace 隔离

每个实验实例创建独立的 K8s Namespace，命名规则：

```
exp-{instance_id}
```

**隔离效果：**
- 不同学生的实验实例在不同 Namespace 中，端口完全独立，不存在冲突
- 同一学生的同一实验中，多个容器在同一 Namespace 内，可通过 Service 名称互访
- 多人协作实验中，每个组员仍拥有独立的实验实例与 Namespace；平台通过组标签与 Network Policy 放通同组实例之间的网络访问
- 不同组的实例在不同 Namespace 中，通过 Network Policy 完全隔离

**端口冲突解决：**
- 100 个学生同时做 geth 实验，每人的 geth 都监听 8545 端口
- 因为在不同 Namespace 中，端口互不影响
- 学生通过平台 Web 终端/IDE 访问自己 Namespace 内的服务，无需关心端口映射

### 5.2 服务发现机制

平台在创建容器时自动注入服务发现环境变量，使容器间可以互相访问：

```bash
# 自动注入的环境变量示例（geth 实验）
GETH_NODE_1_HOST=geth-node-1          # 容器名即 Service 名
GETH_NODE_1_PORT_8545=8545            # RPC 端口

# Fabric 多节点实验
PEER0_ORG1_HOST=peer0-org1
PEER0_ORG1_PORT_7051=7051
ORDERER_0_HOST=orderer-0
ORDERER_0_PORT_7050=7050
FABRIC_CA_HOST=fabric-ca-0
FABRIC_CA_PORT_7054=7054
COUCHDB_HOST=couchdb-0
COUCHDB_PORT_5984=5984
```

**命名规则：**
- `{CONTAINER_NAME}_HOST`：容器的 Service 名称
- `{CONTAINER_NAME}_PORT_{端口号}`：容器暴露的端口

**条件注入：**
- 环境变量仅在对应容器存在时注入
- 如实验中没有 CouchDB 容器，则不会注入 `COUCHDB_HOST` 变量
- 镜像配置模板中的条件环境变量根据实际容器列表动态生效

**多人协作组网说明：**
- 角色专属容器只部署到被分配该角色的组员实例中
- `template_containers.role_id = NULL` 表示该容器配置对所有角色共享；在实例独立 Namespace 模型下，平台会为各角色实例复制部署该共享容器配置，使所有组员获得一致的依赖能力
- 同组跨实例访问通过组级网络放通和服务发现地址注入实现，不依赖“同组共用 Namespace”

### 5.3 跨链实验支持

多种链节点可以放在同一 Namespace 中，天然可以互相访问：

```
Namespace: exp-{instance_id}
├── geth-node-1 (EVM)
├── fabric-peer-0 (Fabric)
├── chainmaker-node-0 (ChainMaker)
└── cross-chain-bridge (跨链桥接服务)
```

教师在编排器中自由选择不同链生态的镜像，平台不做限制。容器间通过 Service 名称互访，跨链桥接逻辑由教师通过初始化脚本或学生手动配置。

## 六、四种拓扑模式详解

### 6.1 单人单节点（topology_mode=1）

```
Namespace: exp-{instance_id}
└── 单个容器（如 geth + solidity-dev）
```

- 最简单的拓扑，一个学生一个容器（或少量辅助容器）
- 适用于：智能合约开发、单节点实验

### 6.2 单人多节点（topology_mode=2）

```
Namespace: exp-{instance_id}
├── peer0-org1
├── peer0-org2
├── orderer-0
├── fabric-ca-0
├── couchdb-0
└── fabric-tools
```

- 一个学生独享多个容器，模拟完整的分布式网络
- 适用于：Fabric 多组织网络搭建、多节点共识实验

### 6.3 多人协作组网（topology_mode=3）

```
Group: {group_id}
├── Namespace: exp-{instance_id_student_1}
│   ├── [角色A: 学生1] peer0-org1 + couchdb-org1
│   └── [共享配置复制] fabric-ca-0
├── Namespace: exp-{instance_id_student_2}
│   ├── [角色B: 学生2] peer0-org2 + couchdb-org2
│   └── [共享配置复制] fabric-ca-0
└── Namespace: exp-{instance_id_student_3}
    └── [角色C: 学生3] orderer-0
```

- 多个学生各自扮演不同角色，角色专属容器部署在各自实例内
- 同组学生通过组级 Network Policy 与服务发现实现跨 Namespace 互通
- `role_id = NULL` 的共享容器配置会复制到各角色实例中，作为公共依赖使用
- 不同组在不同 Namespace 中，完全隔离
- 适用于：多组织 Fabric 网络、分布式系统协作

### 6.4 共享基础设施（topology_mode=4）

> 当前阶段说明：`topology_mode=4` 在模块04后端中先作为模板拓扑语义保留，用于教师编排时表达“全班共享基础设施”的目标形态。  
> 本轮验收范围仍以单人单节点、单人多节点、多人协作组网三类已落地运行时模式为主；共享基础设施的“共享容器生命周期管理、共享 Namespace 编排、学生开发环境按需挂接”将在后续共享环境编排能力中统一实现，避免在当前实例生命周期与分组链路中混入不完整实现。

```
Namespace: exp-shared-{template_id}-{lesson_id}
├── [共享] geth-node-1 (全班共享的链节点)
├── [共享] geth-node-2
├── [共享] blockscout (共享区块链浏览器)
│
├── Namespace: exp-student-{student_id_1}
│   └── solidity-dev (学生1的个人开发环境)
├── Namespace: exp-student-{student_id_2}
│   └── solidity-dev (学生2的个人开发环境)
└── ...
```

- 全班共享一套链网络基础设施，每个学生有独立的开发环境
- 共享容器由教师/平台管理，学生容器按需创建
- 学生的开发环境可访问共享链节点（通过 Network Policy 允许跨 Namespace 访问）
- 适用于：全班共用一条链进行 DApp 开发、智能合约部署
- 与多人协作的区别：多人协作是每组独立环境，共享基础设施是全班共用一套

## 七、配置验证机制（5 层）

### 7.1 验证层级

教师在编排器中配置实验模板时，平台提供 5 层配置验证，确保配置的实验环境可以正常运行：

| 层级 | 名称 | 触发时机 | 严重程度 |
|------|------|----------|----------|
| L1 | 依赖完整性检查 | 添加/移除容器时实时触发 | 错误（阻断发布） |
| L2 | 端口冲突检测 | 添加/修改容器端口时实时触发 | 错误（阻断发布） |
| L3 | 资源合理性检查 | 保存/发布时触发 | 警告（不阻断） |
| L4 | 生态一致性提示 | 添加工具镜像时实时触发 | 提示（不阻断） |
| L5 | 连通性预检 | 发布前可选触发 | 信息（不阻断） |

### 7.2 L1：依赖完整性检查

检查镜像的 `required_dependencies` 是否满足：

```
规则：如果选择了 kafka，必须同时选择 zookeeper
规则：如果选择了 fabric-peer，必须同时选择 fabric-orderer
规则：如果选择了 grafana，必须同时选择 prometheus
规则：如果选择了 webase-web，必须同时选择 webase-front
```

**行为：**
- 教师添加 kafka 时，如果没有 zookeeper，编排器顶部显示红色错误提示："kafka 需要 zookeeper 作为依赖，请添加 zookeeper 容器"
- 同时在 zookeeper 镜像卡片上显示"推荐添加"标记
- 发布时如果仍有未满足的必要依赖，阻断发布

### 7.3 L2：端口冲突检测

检查同一 Namespace 内是否有端口冲突：

```
规则：同一实验模板中，不同容器不可暴露相同端口号
例外：K8s Namespace 隔离保证不同实验实例间无冲突，此处仅检查同一实例内
```

**行为：**
- 如果两个容器都暴露 8080 端口，编排器显示红色错误提示
- 教师可修改其中一个容器的端口映射来解决冲突
- 发布时如果仍有端口冲突，阻断发布

### 7.4 L3：资源合理性检查

检查资源配置是否合理：

```
规则：单容器资源不低于镜像的最低要求（image_versions.min_cpu/min_memory）
规则：所有容器资源总和不超过实验模板的资源限制
规则：所有容器资源总和不超过课程级配额的单实验上限
```

**行为：**
- 黄色警告提示，不阻断发布
- 如"当前配置总 CPU 需求 4 核，超过课程配额单实验上限 2 核，可能导致启动失败"

### 7.5 L4：生态一致性提示

检查工具镜像与链节点镜像的生态匹配：

```
规则：EVM 专属工具（remix-ide, blockscout, slither, mythril, echidna）仅在存在 EVM 链节点时推荐
规则：Fabric 专属工具（fabric-explorer, fabric-tools）仅在存在 Fabric 链节点时推荐
规则：FISCO 专属工具（webase-front, webase-web）仅在存在 FISCO 链节点时推荐
规则：ChainMaker 专属工具（chainmaker-explorer）仅在存在 ChainMaker 链节点时推荐
规则：通用工具（code-server, jupyter-notebook 等）不受限制
```

**行为：**
- 镜像选择面板中，不兼容的工具镜像灰显，鼠标悬停显示原因
- 教师仍可强制选择（点击"仍然添加"），但会显示提示
- 不阻断发布

### 7.6 L5：连通性预检

发布前可选的预检功能，模拟验证容器间的连通性：

```
检查项：
- 所有容器的依赖关系是否形成有向无环图（无循环依赖）
- 启动顺序是否合理（依赖的容器先启动）
- 环境变量中引用的服务是否存在
```

**行为：**
- 教师点击"预检"按钮触发
- 返回检查报告，列出潜在问题
- 不阻断发布，仅供参考

## 九、镜像预拉取策略

### 9.1 策略设计

| 镜像类型 | 预拉取时机 | 说明 |
|----------|-----------|------|
| 官方镜像（全部版本） | 平台部署时 | 部署脚本自动拉取所有官方镜像到所有 K8s 节点 |
| 教师自定义镜像 | 审核通过时 | 审核通过后自动拉取到所有节点 |
| 镜像版本更新 | 版本发布时 | 新版本发布后自动拉取，旧版本保留 |

### 9.2 实现机制

- 使用 K8s DaemonSet 在所有节点上运行镜像预拉取 Job
- 平台维护镜像清单（image manifest），部署时对比清单与节点本地镜像列表
- 缺失的镜像自动拉取，多余的旧版本按策略清理（保留最近 3 个版本）
- 预拉取状态可在管理后台查看

### 9.3 启动时间保障

| 场景 | 预期启动时间 | 前提条件 |
|------|-------------|----------|
| 官方镜像实验 | ≤ 30s（单节点） | 镜像已预拉取 |
| 自定义镜像实验 | ≤ 30s | 镜像已在审核通过时预拉取 |
| 冷启动（镜像未缓存） | ≤ 180s | 首次使用某镜像版本 |

## 十、Collector Agent 数据采集（混合实验）

### 10.1 架构

混合实验中，Collector Agent 作为 sidecar 容器自动注入，负责从实验容器实时采集链上数据并推送到 SimEngine 前端渲染层：

```
实验容器（geth/fabric-peer/...）
    ↕ 本地网络
Collector Agent sidecar
    ↕ WebSocket
SimEngine Core
    ↕ WebSocket
前端仿真面板（领域渲染器）
```

### 10.2 链生态适配器

| 适配器 | 采集方式 | 采集数据 |
|--------|----------|----------|
| ethereum | JSON-RPC (eth_subscribe) | 新区块、交易、日志、Pending交易 |
| fabric | Event Hub (gRPC) | 区块事件、链码事件、交易验证结果 |
| chainmaker | Subscribe API | 区块、交易、合约事件 |
| fisco | Event Push | 区块、交易回执、合约事件 |

### 10.3 配置方式

教师在混合实验编排时配置数据采集通道：

1. 选择需要采集数据的容器
2. 平台根据容器镜像的 `ecosystem` 自动匹配适配器，适配器编码统一使用 `ethereum` / `fabric` / `chainmaker` / `fisco`
3. 教师选择采集的数据类型（区块/交易/事件等）
4. 配置数据推送到哪个仿真场景面板

### 10.4 双模式数据源

混合实验中，仿真面板可同时接收两种数据源：

- **仿真模式**：场景算法容器计算的仿真数据
- **采集模式**：Collector Agent 从真实容器采集的实时数据

教师可为每个仿真场景配置数据源模式：仿真 / 采集 / 双模式（同时显示）。
