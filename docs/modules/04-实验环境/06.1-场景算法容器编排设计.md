## 一、设计概述

本文档是 06-可视化仿真引擎设计的实施细化文档，聚焦 **SceneManager 如何按需启停场景算法容器**。06 文档定义了"算法容器化 + 按需启停 + 维护 gRPC 连接池"的设计原则，本文档给出具体的协议契约、接口定义、容器编排规则、配置项与运维约定。

### 核心设计原则

- **数据驱动镜像**：场景对应的镜像地址存于 `sim_scenarios.container_image_url`，backend 在 CreateSession 时随 sceneCode 一起透传给 SimEngine；新增场景或换镜像无需修改任何配置
- **按需启停**：会话开始时由 SceneManager 通过 K8s API 创建场景 Pod / Service，会话销毁时立即清理，避免空闲资源占用
- **共享运行时镜像**：43 个内置场景共享同一个二进制 `registry.lianjing.com/scenarios/runtime:v1.0.0`，启动时通过 `SCENE_CODE` 环境变量分发；教师自定义场景独立打包（基于 `scenario-base` 基础镜像）
- **唯一接口契约**：SceneManager 只依赖 `scene.Orchestrator` 接口，K8s 实现是唯一生产实现，无静态映射等替代路径
- **集群内外网络兼容**：使用 Service ClusterIP 直接拨号（生产 K8s 内 + 本地 docker-desktop 桥接网络均可达），不依赖 `*.svc.cluster.local` DNS（后者在集群外 docker 容器中不可解析）
- **故障可重启**：场景 Pod 异常崩溃后，下一次 `Manager.Restart` 调用会驱逐旧 Pod 并起新 Pod，恢复会话继续推进
- **空闲回收**：周期扫描 namespace 内由 sim-engine 创建但已不在连接池中的场景 Pod，超过 `IdleTTL` 后清理，避免会话异常丢失导致的 Pod 泄漏

## 二、整体架构

```
┌──────────────────────────────────────────────────────────────────┐
│                    backend（CreateSimSession）                    │
│  从 sim_scenarios.container_image_url 读取镜像地址，随 SceneConfig │
│  proto 字段 container_image_url / resource_request_* 一起透传      │
└──────────────────────────┬───────────────────────────────────────┘
                           │ gRPC CreateSession
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│                       SimEngine Core                              │
│                                                                  │
│  ┌────────────────────┐                                          │
│  │  SessionManager    │                                          │
│  └─────────┬──────────┘                                          │
│            │  StartScene(config)                                 │
│            ▼                                                     │
│  ┌────────────────────────────────────────────┐                  │
│  │  scene.Manager                             │                  │
│  │   ↓ 委托接口调用                             │                  │
│  │  scene.Orchestrator                         │                  │
│  │    StartScene / EvictScene / DestroySession │                  │
│  └─────────┬──────────────────────────────────┘                  │
│            │  生产实现                                              │
│            ▼                                                     │
│  ┌────────────────────────────────────────────┐                  │
│  │  K8sOrchestrator                            │                  │
│  │   - 连接池 (sessionID, sceneCode) → conn    │                  │
│  │   - 主动 gRPC HealthCheck 拨号确认 Ready    │                  │
│  │   - StartIdleReaper 后台空闲回收            │                  │
│  └─────────┬──────────────────────────────────┘                  │
└────────────┼─────────────────────────────────────────────────────┘
             │ K8s API（in-cluster ServiceAccount 或 kubeconfig）
             ▼
┌──────────────────────────────────────────────────────────────────┐
│                    Kubernetes（lenschain ns）                     │
│                                                                  │
│  ┌──────────────────────────┐    ┌──────────────────────────┐    │
│  │  Service (ClusterIP)     │←──→│  Pod (scenario container) │    │
│  │  scn-{session}-{scene}   │    │  image=container_image_url│    │
│  │  port: 50100/grpc        │    │  env: SCENE_CODE=<code>   │    │
│  └──────────────────────────┘    └──────────────────────────┘    │
│                                                                  │
│  按需创建，会话销毁后立即清理；孤儿 Pod 由 IdleTTL 兜底回收            │
└──────────────────────────────────────────────────────────────────┘
```

### 关键约定

| 约定 | 取值 | 说明 |
|------|------|------|
| Pod / Service 命名空间 | `lenschain` | 与 SimEngine Core 同 ns，复用现有 RBAC |
| Pod / Service 命名 | `scn-{session_id_short}-{scene_code_normalized}` | session_id 取后 12 位，scene_code 转小写并把 `_` 替换为 `-` 截断到 30 字符；总长 ≤ 47，DNS-1123 合规 |
| 容器内 gRPC 端口 | `50100` | 与 `scenario-base.Dockerfile` EXPOSE / `SCENARIO_LISTEN_ADDR` 一致 |
| Service 类型 | `ClusterIP` | 不暴露 NodePort；sim-engine 通过 ClusterIP 直拨 |
| 镜像拉取策略 | `IfNotPresent` | 与实验环境容器一致；冷启动通过 `enqueueApprovedScenarioPrePull` 预拉 |
| 标签 | `app.kubernetes.io/managed-by=sim-engine` `app.kubernetes.io/component=scenario` `lenschain.io/session=<sessionID>` `lenschain.io/scene-code=<code>` | 用于审计、清理、Reaper 扫描 |
| Pod Ready 判定 | sim-engine 主动 gRPC `HealthCheck` 拨号 + 重试 | 不依赖 K8s readinessProbe，避免镜像刚 pull 完但 grpc 未起的窗口期 |
| 资源默认值 | requests `100m / 128Mi`，limits `500m / 256Mi` | 对齐 06 文档 §2.2 "场景算法容器 CPU 少量 + 内存少量"；可被 `sim_scenarios` 数据库字段或 Orchestrator 配置覆盖 |
| 启动总超时 | `10s`（默认） | 对齐 06 文档 §6.4 性能指标"场景算法容器启动超时 10 秒" |
| 空闲回收周期 | `1min` 扫描 + `IdleTTL=10min` 默认 | 仅清理"不在连接池且 Pod 创建超过 IdleTTL"的孤儿 Pod |

## 三、协议契约

### 3.1 backend ↔ SimEngine（gRPC）

`SceneConfig` 在 `sim-engine/proto/lenschain/sim_engine/v1/sim_engine.proto` 中定义；CreateSession 时由 backend 填充：

```protobuf
message SceneConfig {
  string scene_code              = 1;   // 场景编码
  string scenario_id             = 2;
  string scenario_version        = 3;   // 显式锁定的场景版本
  string link_group_id           = 4;
  string link_group_code         = 5;
  string link_group_version      = 6;   // 显式锁定的联动组版本
  string layout_role             = 7;   // primary | secondary | auxiliary
  string display_mode            = 8;   // single | split-2 | split-3 | grid-4
  bool   link_to_primary         = 9;
  bool   default_visible         = 10;
  bytes  params_json             = 11;
  bytes  shared_state_json       = 12;
  lenschain.sim_scenario.v1.DataSourceMode data_source_mode = 13;
  bytes  data_source_config_json = 14;
  bytes  layout_position_json    = 15;
  // ↓ 场景编排扩展字段
  string container_image_url     = 16;  // 必填，来自 sim_scenarios.container_image_url
  string resource_request_cpu    = 17;  // 可选，K8s 格式如 "100m"；空走 Orchestrator 默认值
  string resource_request_memory = 18;  // 可选，K8s 格式如 "128Mi"；空走 Orchestrator 默认值
}
```

backend 在 `instance_service.go::buildSimSessionRequest` 读取 `scenario.ContainerImageURL`，缺失时立即返回错误（不进入 SimEngine 调用）。

### 3.2 SceneManager ↔ Orchestrator（Go 接口）

```go
// scene.Orchestrator —— SceneManager 唯一依赖的场景算法容器编排能力契约。
type Orchestrator interface {
    // StartScene 按需启动一个场景算法容器并返回可用的 gRPC 客户端。
    StartScene(ctx context.Context, config Config) (ScenarioClient, error)

    // EvictScene 在场景重启前驱逐旧 Pod / Service 与连接池条目。
    EvictScene(ctx context.Context, sessionID, sceneCode string) error

    // DestroySession 销毁会话对应的所有场景 Pod / Service。
    DestroySession(ctx context.Context, sessionID string) error
}
```

唯一生产实现：`K8sOrchestrator`（`sim-engine/core/internal/scene/k8s_orchestrator.go`）。

### 3.3 SceneManager ↔ 场景算法容器（gRPC）

容器内运行的 `scenario` 二进制读取以下环境变量并启动 `SimScenarioService` gRPC 服务：

| 环境变量 | 值 | 说明 |
|---------|----|------|
| `SCENARIO_LISTEN_ADDR` | `:50100` | 容器内 gRPC 监听地址，由 `scenario-base.Dockerfile` 内置默认 |
| `SCENE_CODE` | `<sceneCode>` | 由 K8sOrchestrator 在 Pod spec 注入；共享运行时镜像据此选择具体场景定义 |

## 四、K8s 编排实现

### 4.1 启动流程（StartScene）

```
1. 校验 config.ContainerImageURL 非空
2. 构造 resourceName = scn-{sessionID 后 12 位}-{sanitized sceneCode}
3. 检查连接池命中 → 命中直接复用
4. 创建 ClusterIP Service（端口 50100），获取分配到的 ClusterIP
5. 创建 Pod（image, env, resources, securityContext）
6. dialAndWaitReady：
     target = clusterIP:50100
     循环主动调 SimScenarioService.HealthCheck，500ms 间隔
     总超时 = ReadyTimeout（默认 10s）
7. 入连接池，返回 ScenarioClient
   异常路径：任一步骤失败均回滚已创建的 Pod / Service / gRPC 连接
```

### 4.2 销毁流程（DestroySession）

```
1. 找出连接池中所有以 sessionID:: 开头的条目
2. 关闭 gRPC 连接
3. 删除 Pod 和 Service
4. 移出连接池
   错误处理：仅记录第一个错误并继续删除其余资源（最大化清理）
```

### 4.3 重启流程（EvictScene + StartScene）

`Manager.Restart` 调用顺序：

```
1. 关闭旧 ScenarioClient 的 gRPC 连接
2. 调用 Orchestrator.EvictScene：
     从连接池删除该 (sessionID, sceneCode) 条目
     按 resourceName 删除 Pod 与 Service（即使池中找不到也按命名兜底删除）
3. 调用 Orchestrator.StartScene：
     完整执行 4.1 流程，得到全新的 Pod 和 gRPC 连接
4. 用最近的 State 重新 Init（保持仿真状态连续性）
```

### 4.4 空闲回收（StartIdleReaper）

后台 goroutine，由 main.go 在 Engine 启动时调用：

```
每 1 分钟扫描一次 namespace 内符合 managed-by=sim-engine, component=scenario 标签的 Pod
对每个 Pod：
  - 若该 Pod 仍在连接池中（属于活跃会话），跳过
  - 若 Pod 已存在超过 IdleTTL（默认 10 分钟），按 Pod 名删除 Pod 与同名 Service
  - 否则跳过（避免误杀刚创建尚未入池的 Pod）
```

### 4.5 进程退出（Shutdown）

main.go 通过 `defer orchestrator.Shutdown(ctx)` 触发：

```
1. 清空连接池，关闭所有 gRPC 连接
2. 删除连接池中记录的所有 Pod 和 Service
3. 兜底清理：按 managed-by=sim-engine, component=scenario 标签列出 namespace 内所有相关
   Pod / Service 并删除（覆盖池外残留）
```

### 4.6 Pod 安全上下文

```yaml
securityContext:
  allowPrivilegeEscalation: false
  privileged: false
  runAsNonRoot: true             # 镜像 USER=scenario (UID 1001) 已满足
  readOnlyRootFilesystem: true   # 场景算法只读文件系统，加固
podSpec.securityContext:
  runAsNonRoot: true
restartPolicy: OnFailure
```

## 五、镜像设计

### 5.1 镜像清单

| 镜像 | 用途 | Dockerfile | 体积目标 |
|------|------|-----------|---------|
| `registry.lianjing.com/lenschain/scenario-base:v1.0.0` | 场景算法容器基础镜像（alpine + ca-certs + tzdata + 非 root scenario UID 1001 + EXPOSE 50100） | `deploy/docker/scenario-base.Dockerfile` | ≤ 30MB |
| `registry.lianjing.com/scenarios/runtime:v1.0.0` | 平台 43 个内置场景的共享运行时（FROM scenario-base + 单一 Go 二进制） | `deploy/docker/scenario.Dockerfile` | ≤ 50MB |
| 教师自定义场景 | 教师自带，FROM `scenario-base` | 教师自定义 | ≤ 50MB |

### 5.2 共享运行时设计

`sim-engine/scenarios/cmd/scenario/main.go` 是唯一入口：

```go
sceneCode := strings.TrimSpace(os.Getenv("SCENE_CODE"))
launcher.MustRunByCode(ctx, sceneCode)
```

- 43 个场景的算法定义统一注册在 `scenarios/internal/catalog`，由 `framework + sdkscenario` 提供 gRPC 服务框架
- 编译产物大小不变：因为 `catalog.NewRegistry()` 静态引用所有场景代码，单独 main 与共享 main 的二进制内容完全一致；共享 main 消除了 43 个 main 包导致的 `go build ./...` 链接资源爆炸问题

### 5.3 教师自定义场景接入

教师在 SimScenario SDK 之上开发自定义场景，打包独立镜像：

```dockerfile
FROM golang:1.25-alpine AS builder
# ... 教师的算法构建 ...
RUN go build -o /out/run .

FROM registry.lianjing.com/lenschain/scenario-base:v1.0.0
COPY --from=builder /out/run /scenario/run
```

审核通过后，`enqueueApprovedScenarioPrePull` 触发节点级预拉取；教师场景的 `container_image_url` 写入 `sim_scenarios` 表后，K8sOrchestrator 与内置场景**走完全相同的调度路径**，无需任何代码区分。

## 六、配置项

`OrchestratorConfig`（位于 `sim-engine/core/internal/config/config.go`，对应 `config.yaml::orchestrator`）：

| 字段 | 默认值 | 说明 |
|------|--------|------|
| `in_cluster` | `false` | true 走集群内 ServiceAccount；false 走 kubeconfig 文件 |
| `kubeconfig_path` | `""` | 集群外 kubeconfig 路径；空时走 `~/.kube/config` |
| `namespace` | `lenschain` | 场景 Pod / Service 所在命名空间 |
| `image_pull_secret_name` | `registry-pull-secret` | 镜像拉取凭据 Secret 名称；本地开发可留空 |
| `ready_timeout` | `10s` | 启动 + 拨号总超时（对齐 06 文档 §6.4） |
| `idle_ttl` | `10m` | 孤儿 Pod 回收前的最小留存时间 |
| `default_cpu` / `default_memory` | `100m` / `128Mi` | 场景未提供资源请求时的默认值 |
| `limit_cpu` / `limit_memory` | `500m` / `256Mi` | 资源上限 |

**敏感字段**（如 JWT secret、MinIO 凭据、快照加密密钥）不入 ConfigMap，通过 `LENSCHAIN_SIM_*` 环境变量从 K8s Secret 注入覆盖。

## 七、覆盖的实验类型与拓扑

本设计**仅**承载 SimEngine SceneManager 编排路径，与实验环境容器编排正交。

| 实验类型（07 文档） | 是否走本设计 | 说明 |
|---|---|---|
| **type=1 纯仿真** | ✅ 是 | 完全由 SceneManager 编排场景 Pod |
| **type=2 真实环境** | ❌ 否 | 不会创建 SimEngine 会话；走 backend 自身 K8sService 编排实验环境容器 |
| **type=3 混合实验** | ✅ 部分 | 场景容器走本设计；实验环境容器与 Collector Agent sidecar 走 backend 现有路径，两条路径独立 |

| topology_mode（07 文档） | 是否影响场景 Pod | 说明 |
|---|---|---|
| 1 / 2 / 3 / 4 | ❌ 否 | 这是实验环境容器的拓扑，与场景算法容器无关 |

**会话粒度**：SimEngine Session 按 `experiment_instance` 创建（每个学生独立 Session）。即使在 `topology_mode=3` 多人协作组下，每个组员仍有独立的场景 Pod，组内联动通过 LinkEngine 共享状态空间中转，**不依赖**场景 Pod 之间的网络互通。

## 八、运维要点

### 8.1 RBAC

`deploy/k8s/base/rbac/sim-engine-clusterrole.yaml` 已为 `lenschain-sim-engine` ServiceAccount 授权：

```yaml
rules:
  - apiGroups: [""]
    resources: ["pods", "services", "configmaps"]
    verbs: ["*"]
```

涵盖本设计需要的全部 K8s 操作，无需新增 RBAC 资源。

### 8.2 ConfigMap + Secret

- 非敏感配置：`deploy/k8s/base/sim-engine/configmap.yaml` 挂到 `/app/configs/config.yaml`
- 敏感配置覆盖：`deployment.yaml` 通过 `LENSCHAIN_SIM_AUTH_WS_JWT_SECRET` 等环境变量从 `backend-secret` / `minio-secret` 注入

### 8.3 本地开发

要求：
1. 启用 Docker Desktop Kubernetes（Kubeadm 模式）
2. 通过 `deploy/scripts/{bash,powershell}/build-all-images.{sh,ps1}` 一次性构建：
   - 阶段 0：预拉公共基础镜像
   - 阶段 1：实验环境镜像（`deploy/images/`）
   - 阶段 2：平台服务镜像 + `scenario-base`
   - 阶段 3：场景共享运行时（`scenarios/runtime:v1.0.0`）
3. SimEngine Core 在 docker-compose 启动时挂载 `~/.kube/config` 到容器，`in_cluster=false` 模式访问 docker-desktop K8s
4. 场景 Pod 用 ClusterIP 直拨，docker-compose 桥接网络与 K8s ClusterIP 段连通

### 8.4 故障排查

| 现象 | 排查方向 |
|------|---------|
| `场景 ${code} 缺少容器镜像 URL` | 检查 `sim_scenarios.container_image_url` 是否正确填充 |
| `场景 gRPC 拨号重试超时` | 1) 镜像是否存在节点缓存（`enqueueApprovedScenarioPrePull` 是否触发）2) Pod Events 看 ImagePullBackOff / CrashLoopBackOff 3) 容器日志 `kubectl logs <pod>` 看二进制启动错误 4) `SCENE_CODE` 是否注入正确 |
| 旧 Pod 残留 | `kubectl get pods -n lenschain -l app.kubernetes.io/managed-by=sim-engine`；`StartIdleReaper` 应在 `IdleTTL+1min` 内自动清理 |
| RBAC 拒绝 | sim-engine Pod 的 ServiceAccount 必须是 `lenschain-sim-engine`，已绑定 `lenschain-sim-engine-operator` ClusterRole |

## 九、验收标准

| 编号 | 验收点 | 验证方法 |
|------|--------|---------|
| AC-01 | **数据驱动**：更新 `sim_scenarios.container_image_url` 后下次会话自动使用新镜像 | `UPDATE sim_scenarios SET container_image_url='...' WHERE code='pow-mining'`，重新启动实验，看 Pod 镜像是否更新 |
| AC-02 | **按需启停**：无活跃会话时 namespace 中无场景 Pod | `kubectl get pods -n lenschain -l app.kubernetes.io/managed-by=sim-engine` 应为空 |
| AC-03 | **秒级启动**：CreateSession 到 Pod Ready ≤ 10 秒 | 实验启动至前端开始接收 WebSocket 消息的耗时 ≤ 10 秒（镜像已缓存） |
| AC-04 | **会话清理**：DestroySession 后场景 Pod 立即消失 | 销毁实验后 `kubectl get pods` 看不到对应 `scn-*-*` Pod |
| AC-05 | **空闲回收**：异常残留 Pod 在 `IdleTTL+1min` 内自动清理 | 模拟会话异常（如直接重启 sim-engine）后等待 11 分钟，看 Pod 是否被回收 |
| AC-06 | **故障重启**：场景 Pod 异常退出后 `Manager.Restart` 触发新建 Pod | `kubectl delete pod scn-xxx`，触发 Restart，确认产生新 Pod 且会话状态连续 |
| AC-07 | **生产部署无配置变更**：上线新场景仅需更新数据库 | 不修改任何 ConfigMap / Secret / Deployment 即可使用新场景（前提：镜像已通过 CI 推送到 Harbor） |
