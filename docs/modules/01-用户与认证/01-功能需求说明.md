> 模块状态：✅ 已确认

## 一、模块概述

### 1.1 模块职责

用户与认证模块是整个平台的基础模块，负责：

- 用户身份管理（创建、导入、编辑、禁用、归档、删除）
- 身份认证（手机号+密码登录、SSO登录）
- 会话管理（双Token机制、单设备登录、踢下线）
- 权限控制（RBAC角色权限体系）
- 安全策略（密码复杂度、登录锁定、操作日志）

### 1.2 模块边界

| 本模块负责 | 本模块不负责 |
|-----------|-------------|
| 用户CRUD、认证、授权 | 学校/租户的创建与管理（→ 模块02） |
| 登录/登出/Token管理 | 课程与学生的关联关系（→ 模块03） |
| 用户导入/导出 | 实验环境权限（→ 模块04） |
| 操作日志记录 | 竞赛报名与参赛资格（→ 模块05） |
| 个人中心基础功能 | 学习数据详情（→ 模块06） |

### 1.3 依赖关系

```
本模块被以下模块依赖：
  ← 02-学校与租户管理（学校管理员是用户的一种角色）
  ← 03-课程与教学（教师、学生身份）
  ← 04-实验环境（用户身份鉴权）
  ← 05-CTF竞赛（参赛者身份）
  ← 06-评测与成绩（成绩归属用户）
  ← 07-通知与消息（消息接收者）
  ← 08-系统管理与监控（操作审计）

本模块依赖：
  → 无（基础模块，无外部依赖）
```

### 2.2 用户故事

#### US-01：学校管理员批量导入学生

```
作为 学校管理员
我想要 通过上传Excel文件批量导入学生信息
以便于 快速为新学期的学生创建平台账号

验收条件：
1. 可下载标准导入模板（含字段说明和示例数据）
2. 上传文件后先进入预览页面，显示解析出的数据和校验结果
3. 手机号已存在时标记为冲突，管理员可选择跳过或覆盖
4. 确认导入后显示结果统计（成功N条、失败N条、跳过N条）
5. 可下载失败记录的Excel文件，包含失败原因
6. 导入文件中的每条学生记录都需要填写初始密码，并满足当前密码复杂度要求
```

#### US-02：学生首次登录

```
作为 新导入的学生
我想要 使用初始密码首次登录平台
以便于 开始使用平台学习

验收条件：
1. 使用手机号+初始密码成功登录
2. 系统检测到首次登录，强制跳转到修改密码页面
3. 新密码必须满足复杂度要求（至少8位，含大小写字母和数字）
4. 新密码不能与初始密码相同
5. 修改成功后自动登录进入平台首页
```

#### US-03：SSO登录

```
作为 已导入的用户
我想要 通过学校统一认证系统登录平台
以便于 无需记忆额外的密码

验收条件：
1. 登录页面显示"学校SSO登录"入口
2. 点击后跳转到对应学校的SSO认证页面
3. SSO认证成功后回调平台，通过「学校ID+学号/工号」匹配用户
4. 匹配成功则签发Token并登录
5. 匹配失败（未导入）则显示"账号未开通，请联系管理员"
6. 首次通过SSO登录的用户如果还未修改初始密码，同样强制修改
```

#### US-04：单设备登录

```
作为 平台
我想要 限制同一账号只能在一个设备上登录
以便于 防止账号共享和保障安全

验收条件：
1. 用户在设备A登录后，在设备B登录同一账号
2. 设备B登录成功，设备A的Token立即失效
3. 设备A下次请求时收到401响应，提示"账号已在其他设备登录"
4. 设备A前端收到该响应后跳转到登录页
```

#### US-05：登录失败锁定

```
作为 平台
我想要 在用户连续登录失败时锁定账号
以便于 防止暴力破解密码

验收条件：
1. 管理员可配置：最大失败次数、锁定时长
2. 连续失败达到阈值后，账号被锁定
3. 锁定期间尝试登录提示"账号已锁定，请N分钟后重试"
4. 锁定时间到期后自动解锁
5. 登录成功后重置失败计数
6. 管理员可手动解锁被锁定的账号
```

## 三、业务规则与约束

### 3.1 账号规则

| 规则编号 | 规则描述 |
|---------|---------|
| BR-01 | 手机号全局唯一；除超级管理员外，一人一校严格限制 |
| BR-02 | 不开放注册，所有账号由管理员导入或手动创建 |
| BR-03 | 导入时每条记录必须提供初始密码，手动创建账号时也必须显式设置初始密码 |
| BR-04 | 首次登录必须修改初始密码 |
| BR-05 | 密码至少8位，必须包含大写字母、小写字母和数字 |
| BR-06 | 修改密码时新密码不能与当前密码相同 |

### 3.2 认证规则

| 规则编号 | 规则描述 |
|---------|---------|
| BR-07 | 登录方式：手机号+密码 或 学校SSO |
| BR-08 | SSO通过「学校ID + 学号/工号」匹配用户，未匹配则拒绝登录 |
| BR-09 | 同一账号同一时间只能在一个设备登录（单设备策略） |
| BR-10 | 新设备登录时旧设备Token立即失效 |
| BR-11 | Access Token有效期30分钟，Refresh Token有效期7天 |
| BR-12 | Refresh Token刷新时同时签发新的Access Token和Refresh Token |
| BR-13 | 登录失败锁定策略可由超级管理员配置 |
| BR-13A | 账号重新登录、被禁用、被归档、被重置密码时，系统通过 `token_valid_after` 基线使历史Access Token全部失效 |

### 3.3 角色规则

| 规则编号 | 规则描述 |
|---------|---------|
| BR-14 | 系统角色：超级管理员、学校管理员、教师、学生 |
| BR-15 | 仅教师可兼任学校管理员，其他角色不允许兼任 |
| BR-16 | 学校管理员只能管理本校用户数据 |
| BR-17 | 超级管理员可跨学校管理所有数据 |
| BR-17A | 超级管理员为平台级账号，不隶属任何学校，`school_id` 固定为 `0` |

### 3.4 账号状态规则

| 规则编号 | 规则描述 |
|---------|---------|
| BR-18 | 账号状态：正常、禁用、归档、已删除（软删除） |
| BR-19 | 禁用账号无法登录，已有会话立即失效 |
| BR-20 | 归档账号无法登录，保留所有历史数据 |
| BR-21 | 软删除账号可由管理员恢复 |
| BR-22 | 学籍变更（转学院/转专业）不影响账号和历史数据 |
| BR-22A | “已有会话立即失效”不仅删除 Refresh Session，还必须拒绝旧 Access Token 后续访问 |

### 4.2 登录认证流转

```
用户输入手机号+密码
        │
        ▼
   账号是否存在？ ──否──▶ 返回"手机号或密码错误"
        │是
        ▼
   账号状态正常？ ──否──▶ 返回对应提示（禁用/归档/锁定）
        │是
        ▼
   密码是否正确？ ──否──▶ 失败计数+1 ──▶ 达到阈值？──是──▶ 锁定账号
        │是                                    │否
        ▼                                      ▼
   重置失败计数                           返回"密码错误，还剩N次"
        │
        ▼
   是否首次登录？ ──是──▶ 强制修改密码 ──▶ 修改成功
        │否                                    │
        ▼                                      ▼
   踢掉旧设备会话                         踢掉旧设备会话
        │                                      │
        ▼                                      ▼
   签发双Token ◄──────────────────────────签发双Token
        │
        ▼
   登录成功，进入首页
```
