## 一、模块概述

### 1.1 模块定位

系统管理与监控模块是**平台运维管理层**，负责统一审计、全局配置、告警引擎、运维仪表盘和平台使用统计。

**核心职责：**
- 统一审计中心（聚合各模块操作日志）
- 全局平台配置管理
- 告警规则引擎
- 运维仪表盘（平台健康监控）
- 平台使用统计
- 数据备份管理
- 系统公告管理（通过模块07发送）

### 1.2 模块边界

| 本模块负责 | 不属于本模块（归属说明） |
|-----------|----------------------|
| 统一审计中心（聚合查询） | 登录日志记录（→ 模块01 `login_logs`） |
| 全局平台配置 | 操作日志记录（→ 模块01 `operation_logs`） |
| 告警规则管理 | 实验操作日志记录（→ 模块04 `instance_operation_logs`） |
| 运维仪表盘 | 实验环境资源监控（→ 模块04 自有监控面板） |
| 平台使用统计 | 竞赛监控面板（→ 模块05 自有监控面板） |
| 数据备份管理 | 安全策略配置（→ 模块01 `security_policies`） |
| 系统公告（通过模块07） | 学校管理（→ 模块02） |

> **重要说明：** 统一审计中心采用**聚合查询模式**——不新建统一日志表，而是跨模块01的 `login_logs`/`operation_logs`、模块04的 `instance_operation_logs` 等表进行联合查询。各模块继续向自己的日志表写入数据。

### 1.3 与其他模块的联动关系

| 关联模块 | 联动方式 | 数据流向 | 具体说明 |
|---------|---------|---------|---------|
| 模块01 用户与认证 | 读取日志 | 01 → 08 | 聚合查询 `login_logs`（登录日志）和 `operation_logs`（操作日志） |
| 模块01 用户与认证 | 读取配置 | 01 → 08 | 读取 `security_policies` 展示在全局配置页 |
| 模块02 学校与租户管理 | 读取统计 | 02 → 08 | 读取 `schools` 表统计学校数量和状态 |
| 模块03 课程与教学 | 读取统计 | 03 → 08 | 读取 `courses` 表统计课程数量 |
| 模块04 实验环境 | 读取日志+统计 | 04 → 08 | 聚合查询 `instance_operation_logs`；读取实验实例统计数据 |
| 模块05 CTF竞赛 | 读取统计 | 05 → 08 | 读取竞赛数量和状态统计 |
| 模块06 评测与成绩 | 读取统计 | 06 → 08 | 读取成绩统计概览数据 |
| 模块07 通知与消息 | 发送通知 | 08 → 07 | 告警触发时通过模块07发送站内信通知管理员；系统公告通过模块07发布 |

### 1.4 依赖关系

```
模块08 依赖：
  → 01-用户与认证（日志数据、用户数据）
  → 02-学校与租户管理（学校统计）
  → 03-课程与教学（课程统计）
  → 04-实验环境（操作日志、实验统计）
  → 05-CTF竞赛（竞赛统计）
  → 06-评测与成绩（成绩统计）
  → 07-通知与消息（告警通知发送）

模块08 被依赖：
  无（模块08是最顶层的聚合模块）
```

## 三、用户故事

### US-01：超级管理员查看统一审计日志

```
作为 超级管理员
我想要 在一个统一的界面中搜索所有模块的操作日志
以便于 快速定位问题和审计操作

验收条件：
1. 可搜索登录日志（模块01 login_logs）
2. 可搜索操作日志（模块01 operation_logs）
3. 可搜索实验操作日志（模块04 instance_operation_logs）
4. 支持按时间范围、操作人、操作类型、日志来源筛选
5. 支持关键词搜索
6. 结果按时间倒序统一展示
```

### US-02：超级管理员管理全局配置

```
作为 超级管理员
我想要 在一个集中的页面管理平台全局配置
以便于 统一控制平台行为

验收条件：
1. 可配置平台基本信息（名称、Logo、备案号、版权信息）
2. 可配置默认存储配额（每个学校的默认存储空间）
3. 可配置全局安全策略（密码复杂度、会话超时等）
4. 所有配置变更自动记录历史（变更前后值、操作人、时间）
5. 配置修改后立即生效
```

### US-03：超级管理员配置告警规则

```
作为 超级管理员
我想要 配置系统告警规则
以便于 在系统异常时及时收到通知

验收条件：
1. 支持阈值告警（如CPU使用率>80%持续5分钟）
2. 支持事件告警（如连续登录失败>10次）
3. 支持服务状态告警（如某服务不可达）
4. 告警触发后通过站内信通知管理员（通过模块07）
5. 同一规则的告警在未处理前不重复通知（静默期可配置）
6. 告警分级：信息、警告、严重、紧急
```

## 四、业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-01 | 审计聚合查询 | 不新建统一日志表，跨 `login_logs`、`operation_logs`、`instance_operation_logs` 联合查询 |
| BR-02 | 审计日志不可删除 | 所有审计日志不可删除、不可篡改 |
| BR-03 | 配置变更记录 | 所有全局配置变更自动记录到 `config_change_logs`，包含变更前后值 |
| BR-04 | 告警级别 | 信息(1)、警告(2)、严重(3)、紧急(4)，不同级别不同通知策略 |
| BR-05 | 告警静默期 | 同一规则触发的告警在静默期内不重复通知，静默期可配置（默认30分钟） |
| BR-06 | 告警去重 | 同一规则同一目标的未处理告警不重复创建 |
| BR-07 | 告警通知渠道 | 告警通过模块07的站内信通知超级管理员 |
| BR-08 | 运维数据刷新 | 运维仪表盘数据每30秒自动刷新 |
| BR-09 | 统计数据聚合 | 平台统计按日聚合，保留365天历史数据 |
| BR-10 | 备份存储 | 备份文件存储到MinIO，支持PostgreSQL全量备份 |
| BR-11 | 备份保留策略 | 默认保留最近30份备份，超出自动删除最旧的 |
| BR-12 | 系统公告 | 通过模块07的系统公告接口发布，模块08不重复实现公告功能 |

## 五、统一审计中心 — 日志源说明

### 5.1 聚合的日志表

| 日志表 | 所属模块 | 记录内容 | 关键字段 |
|--------|---------|---------|---------|
| `login_logs` | 模块01 | 登录成功/失败/登出/踢下线/锁定 | user_id, action, ip, user_agent, created_at |
| `operation_logs` | 模块01 | CRUD操作审计（用户管理、密码重置等） | operator_id, action, target_type, target_id, detail(JSONB), ip, created_at |
| `instance_operation_logs` | 模块04 | 实验操作（启动/暂停/终端命令/仿真交互） | instance_id, student_id, action, command, created_at |

### 5.2 聚合查询设计

审计中心的查询通过**统一查询接口**实现，后端根据筛选条件分别查询各日志表，然后合并排序返回：

```
用户请求 → 审计查询接口
              │
              ├── 查询 login_logs（按条件筛选）
              ├── 查询 operation_logs（按条件筛选）
              └── 查询 instance_operation_logs（按条件筛选）
              │
              └── 合并结果，按 created_at 排序，分页返回
```

> **性能考虑：** 当数据量大时，可通过指定日志来源（source）缩小查询范围，避免全表扫描。

## 六、告警规则引擎

### 6.1 告警类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 阈值告警 | 监控指标超过阈值持续一段时间 | CPU > 80% 持续5分钟 |
| 事件告警 | 特定事件发生次数超过阈值 | 同一IP连续登录失败 > 10次 |
| 服务状态告警 | 服务健康检查失败 | PostgreSQL连接失败 |

### 6.2 告警规则配置结构

```json
{
  "name": "CPU使用率过高",
  "type": "threshold",
  "level": 3,
  "condition": {
    "metric": "cpu_usage",
    "operator": ">",
    "value": 80,
    "duration": 300
  },
  "silence_period": 1800,
  "is_enabled": true
}
```

```json
{
  "name": "异常登录检测",
  "type": "event",
  "level": 2,
  "condition": {
    "event_source": "login_logs",
    "event_filter": { "action": 2 },
    "group_by": "ip",
    "count_threshold": 10,
    "time_window": 600
  },
  "silence_period": 3600,
  "is_enabled": true
}
```

### 6.3 告警处理流程

```
规则检测 → 条件满足 → 检查静默期 → 创建告警事件
                                        │
                                   通过模块07发送站内信
                                        │
                                   管理员查看 → 处理/忽略
```

### 6.4 与模块04/05已有告警的关系

- 模块04的实验环境异常告警（Pod异常、资源不足）：由模块04自行检测，通过模块07发送通知，同时在模块04的监控面板展示
- 模块05的竞赛监控告警：由模块05自行检测，在模块05的监控面板展示
- 模块08的告警规则引擎：负责**平台基础设施级别**的告警（CPU/内存/磁盘/服务状态/异常登录），不重复模块04/05的业务级告警
- 模块08的运维仪表盘可以展示模块04/05的告警汇总数据（只读）
