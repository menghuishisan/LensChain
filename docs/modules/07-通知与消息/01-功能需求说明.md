# 通知与消息模块 — 功能需求说明

> 模块状态：✅ 已确认
> 文档版本：v1.0

---

## 一、模块概述

### 1.1 模块定位

通知与消息模块是**统一消息中心**，负责将平台各模块产生的通知事件汇聚到站内信系统，为用户提供统一的消息查看和管理体验。

**核心职责：**
- 站内信收件箱（统一消息中心）
- 跨模块事件通知总线（各模块通过统一接口发送通知）
- 系统公告管理
- 消息模板管理
- 用户通知偏好配置
- 定时通知（截止提醒、开始提醒等）

### 1.2 模块边界

| 本模块负责 | 不属于本模块（归属说明） |
|-----------|----------------------|
| 站内信收发与管理 | 课程公告（→ 模块03 `course_announcements`，课程内部公告） |
| 系统公告（平台级） | 竞赛公告（→ 模块05，竞赛内WebSocket推送） |
| 消息模板管理 | 实验分组消息（→ 模块04 `group_messages`，实验内即时通讯） |
| 用户通知偏好 | 竞赛排行榜实时推送（→ 模块05 WebSocket） |
| 定时通知触发 | 实验环境状态推送（→ 模块04 WebSocket） |
| 跨模块事件通知总线 | 短信发送（当前版本不实现外部渠道） |

> **重要说明：** 各模块已有的WebSocket实时推送（模块04实验状态、模块05竞赛排行榜/回合状态）保持不变，由各模块自行管理。模块07负责的是"非实时"的站内信通知和系统公告。

### 1.3 与其他模块的联动关系

| 关联模块 | 联动方式 | 数据流向 | 具体说明 |
|---------|---------|---------|---------|
| 模块01 用户与认证 | 读取 + 接收事件 | 01 → 07 | 读取 `users` 获取接收者信息；接收账号创建、密码重置等事件 |
| 模块02 学校与租户管理 | 接收事件 | 02 → 07 | 接收学校授权到期等**平台内部用户**通知事件；申请联系人短信仍由模块02自行处理 |
| 模块03 课程与教学 | 接收事件 | 03 → 07 | 接收课程发布、作业发布、作业截止提醒、批改完成等事件 |
| 模块04 实验环境 | 接收事件 | 04 → 07 | 接收实验发布、实验即将超时、评分完成等事件 |
| 模块05 CTF竞赛 | 接收事件 | 05 → 07 | 接收竞赛发布、竞赛即将开始、报名截止提醒等事件 |
| 模块06 评测与成绩 | 接收事件 | 06 → 07 | 接收成绩审核通过/驳回、申诉处理、学业预警等事件 |
| 模块08 系统管理与监控 | 接收事件 | 08 → 07 | 接收系统维护通知、告警通知等事件 |

> **联动机制：** 各模块通过调用模块07提供的**内部通知接口**发送事件，模块07根据消息模板生成站内信并投递到接收者的收件箱。各模块不直接操作 `notifications` 表。

### 1.4 依赖关系

```
模块07 依赖：
  → 01-用户与认证（用户数据、接收者信息）

模块07 被依赖（接收事件）：
  ← 01-用户与认证（账号事件）
  ← 02-学校与租户管理（学校审核事件）
  ← 03-课程与教学（课程/作业事件）
  ← 04-实验环境（实验事件）
  ← 05-CTF竞赛（竞赛事件）
  ← 06-评测与成绩（成绩/预警事件）
  ← 08-系统管理与监控（系统事件）
```

## 三、用户故事

### US-01：用户查看站内信

```
作为 平台用户
我想要 在一个统一的收件箱中查看所有通知
以便于 不遗漏任何重要信息

验收条件：
1. 收件箱按时间倒序显示所有站内信
2. 支持按分类筛选：系统通知、课程通知、实验通知、竞赛通知、成绩通知
3. 支持已读/未读筛选
4. 未读消息有视觉区分（加粗/蓝色圆点）
5. 点击消息可查看详情，自动标记为已读
```

### US-02：未读消息实时提醒

```
作为 平台用户
我想要 在有新消息时立即看到提醒
以便于 及时处理重要通知

验收条件：
1. 顶部导航栏铃铛图标显示未读消息数
2. 新消息到达时未读数实时更新（WebSocket推送）
3. 点击铃铛展开消息预览面板（最近5条）
4. 点击"查看全部"跳转到完整收件箱
```

### US-03：超级管理员发布系统公告

```
作为 超级管理员
我想要 发布平台级系统公告
以便于 通知所有用户重要信息（如维护、更新）

验收条件：
1. 填写公告标题和内容（富文本）
2. 可选择立即发布或定时发布
3. 发布后所有用户的收件箱收到该公告
4. 系统公告置顶显示，不可被用户删除
5. 可下架已发布的公告
```

### US-04：用户配置通知偏好

```
作为 平台用户
我想要 配置我接收哪些类型的通知
以便于 减少不必要的消息干扰

验收条件：
1. 按分类显示通知开关（系统通知、课程通知、实验通知、竞赛通知、成绩通知）
2. 系统通知和成绩通知不可关闭（强制接收）
3. 关闭某类通知后，该类事件不再生成站内信
4. 配置立即生效
```

### US-05：管理员配置消息模板

```
作为 超级管理员
我想要 配置各类通知的消息模板
以便于 统一通知的格式和内容

验收条件：
1. 每种事件类型对应一个模板（标题模板 + 内容模板）
2. 模板支持变量占位符：{user_name}、{course_name}、{deadline}、{score} 等
3. 修改模板后，后续新生成的通知使用新模板
4. 提供预览功能，可用示例数据渲染模板
```

### US-06：定时通知

```
作为 系统
我想要 在特定时间自动发送提醒通知
以便于 用户不会错过重要截止时间

验收条件：
1. 作业截止前24小时自动提醒未提交的学生
2. 竞赛报名截止前48小时自动提醒未报名的学生
3. 竞赛开始前1小时自动提醒已报名的参赛者
4. 定时通知通过定时任务扫描触发
```

## 五、业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-01 | 站内信保留180天 | 超过180天的站内信自动归档清理 |
| BR-02 | 系统公告不可删除 | 用户不可删除系统公告，只能标记已读 |
| BR-03 | 系统公告置顶 | 系统公告在收件箱中置顶显示 |
| BR-04 | 强制通知分类 | 系统通知和成绩通知不可关闭，用户必须接收 |
| BR-05 | 通知偏好过滤 | 用户关闭某类通知后，该类事件不生成站内信 |
| BR-06 | 未读计数缓存 | 未读消息计数通过Redis缓存，新消息到达时通过WebSocket推送更新 |
| BR-07 | 统一通知接口 | 各模块通过内部接口发送**平台内部用户**通知事件，不直接操作 notifications 表 |
| BR-08 | 消息模板渲染 | 通知内容根据消息模板 + 事件参数动态渲染 |
| BR-09 | 批量通知异步 | 发给多人的通知（如全班/全校）通过NATS消息队列异步处理 |
| BR-10 | 定时通知扫描 | 定时任务每小时扫描即将到期的作业/竞赛，触发提醒通知 |
| BR-11 | 通知去重 | 同一事件对同一用户不重复发送（如作业截止提醒只发一次） |
| BR-12 | 消息队列选型 | 使用NATS作为内部消息队列，轻量级，Go原生支持 |

## 六、消息队列架构说明

### 6.1 NATS 使用方式

```
各模块 ──publish──→ NATS Subject ──subscribe──→ 模块07 通知服务
                                                    │
                                              ┌─────┴─────┐
                                              │ 模板渲染   │
                                              │ 偏好过滤   │
                                              │ 写入DB     │
                                              │ 推送WebSocket│
                                              └───────────┘
```

**NATS Subject 设计：**
| Subject | 说明 |
|---------|------|
| `notification.user.*` | 用户相关事件（账号创建、密码重置） |
| `notification.school.*` | 学校相关站内信事件（如授权到期提醒） |
| `notification.course.*` | 课程相关事件（发布、作业、批改） |
| `notification.experiment.*` | 实验相关事件（发布、超时、评分） |
| `notification.competition.*` | 竞赛相关事件（发布、开始、报名） |
| `notification.grade.*` | 成绩相关事件（审核、申诉、预警） |
| `notification.system.*` | 系统相关事件（维护、告警） |

### 6.2 各模块接入方式

各模块只需调用一个统一的通知发送函数：

```go
// 各模块调用示例（Go伪代码）
notificationService.Send(ctx, notification.Event{
    Type:       "assignment.published",
    ReceiverIDs: studentIDs,          // 接收者ID列表
    Params: map[string]string{
        "course_name":     "区块链原理",
        "assignment_name": "作业1",
    },
})
```

通知服务内部流程：
1. 发布事件到 NATS
2. 通知服务订阅并消费
3. 查询消息模板，渲染通知内容
4. 检查每个接收者的通知偏好
5. 写入 `notifications` 表
6. 更新 Redis 未读计数
7. 通过 WebSocket 推送未读数更新
