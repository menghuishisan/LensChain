## 一、文档目的

本文档是部署与运维模块的镜像与容器设计文档，聚焦于 **每个镜像的 Dockerfile 规范、manifest.yaml 元数据规范、构建策略、标签策略、签名与安全** 等细节。与 `01-功能需求说明.md` 的镜像清单互补，本文档回答"每个镜像该怎么写 Dockerfile、怎么打标签、怎么构建"。

### 设计原则

- **所有镜像都有 Dockerfile + manifest.yaml**：即使是直接引用官方的镜像，也要有薄 Dockerfile 声明 EXPOSE / ENV / 标签，便于统一管理
- **薄封装优先**：官方镜像尽量 `FROM 官方:版本` 不做侵入改造，只补充平台约定（EXPOSE、HEALTHCHECK、非 root、标签）
- **多阶段构建**：平台自研镜像必须多阶段构建，runtime 阶段仅保留最小运行时
- **非 root**：所有镜像运行时必须以非 root 用户启动（除 fabric-peer 等上游强制 root 的少数例外）
- **最小体积**：runtime 基础镜像优先 `alpine` / `distroless`；避免 `ubuntu:latest` 这类全量镜像
- **可重复构建**：所有版本号显式固定（不使用 `latest` 作为 FROM），依赖校验和锁定

## 二、manifest.yaml 完整规范

### 2.1 字段定义

每个镜像目录下必须有一份 `manifest.yaml`，字段严格对应模块04 `images` 和 `image_versions` 表：

```yaml
# manifest.yaml 完整字段示例
name: geth                              # 镜像名，对应 images.name
category: chain-nodes                   # 类别：chain-nodes / middleware / tools / base
ecosystem: ethereum                     # 生态编码：ethereum / fabric / chainmaker / fisco / solana / polkadot / cosmos / move / bitcoin / general
description: Go-Ethereum 全功能客户端   # 中文描述
maintainer: lenschain-platform          # 维护者
license: GPL-3.0                        # 许可协议
official_source: ethereum/client-go     # 上游镜像来源（Docker Hub path）
registry_project: chain-nodes           # 推送到 Registry 的 Project

# 多版本
versions:
  - version: "1.14"                     # 展示版本
    tag: "v1.14.0"                      # Registry 标签
    upstream_tag: "v1.14.0"             # 上游镜像标签
    min_cpu: "0.25"                     # 最低 CPU
    min_memory: "512Mi"                 # 最低内存
    recommended_cpu: "0.5"              # 推荐 CPU
    recommended_memory: "1Gi"           # 推荐内存
    is_default: true                    # 是否为默认版本
    released_at: "2026-04-01"           # 发布日期（可选）
  - version: "1.13"
    tag: "v1.13.15"
    upstream_tag: "v1.13.15"
    min_cpu: "0.25"
    min_memory: "512Mi"
    is_default: false

# 默认端口映射（BR-54）
default_ports:
  - port: 8545
    protocol: tcp
    name: HTTP-RPC
    description: HTTP JSON-RPC API
  - port: 8546
    protocol: tcp
    name: WebSocket-RPC
    description: WebSocket JSON-RPC API
  - port: 30303
    protocol: tcp
    name: P2P
    description: P2P 发现与通信

# 默认环境变量
default_env_vars:
  - key: GETH_NETWORK
    default_value: dev
    description: 运行网络（dev / mainnet / testnet）
    required: false
  - key: GETH_DATADIR
    default_value: /root/.ethereum
    description: 链数据目录
    required: false

# 条件环境变量（BR-56）
conditional_env_vars: []

# 默认挂载卷
default_volumes:
  - mount_path: /root/.ethereum
    purpose: chain_data
    size: 10Gi
    description: 链数据持久化

# 典型搭配（BR-55）
typical_companions:
  required: []
  recommended:
    - image: blockscout
      reason: 本地区块链浏览器，方便查看交易和区块
  optional:
    - image: remix-ide
      reason: Solidity 在线 IDE，可直接连接本地节点
    - image: prometheus
      reason: 采集 geth 指标数据
    - image: grafana
      reason: 可视化监控仪表盘

# 依赖关系（BR-59）
required_dependencies: []

# 生态兼容
supported_tools:
  - remix-ide
  - blockscout
  - slither
  - mythril
  - echidna

# 健康检查
healthcheck:
  test: ["CMD-SHELL", "geth attach --exec 'eth.blockNumber' http://localhost:8545"]
  interval: 30s
  timeout: 10s
  retries: 3
  start_period: 20s

# 文档
documentation_url: /docs/images/geth

# 预拉取策略
prepull_phase: 1                        # Phase 1 / 2 / 3（BR-63）
prepull_node_tier: light                # light / heavy（对应 lenschain.io/exp-tier 节点标签）

# 安全扫描策略
security_scan:
  enabled: true
  allow_critical: false                 # 是否允许 Critical 漏洞（平台自研镜像为 false，CTF 漏洞镜像为 true）
  allow_high: true
```

### 2.2 字段说明

| 字段 | 必填 | 对应数据库表 | 用途 |
|------|------|--------------|------|
| `name` | ✅ | `images.name` | 镜像名 |
| `category` | ✅ | `images.category` | 分类（四选一） |
| `ecosystem` | ✅ | `images.ecosystem` | 生态编码，用于工具过滤（BR-68） |

| `default_env_vars` | 按需 | `images.default_env_vars` | JSONB |
| `conditional_env_vars` | 按需 | `images.default_env_vars`（合并） | 条件变量 |

| `documentation_url` | ✅ | `images.documentation_url` | 文档链接 |
| `prepull_phase` | ✅ | 仅 manifest | 预拉取阶段 |
| `security_scan` | ✅ | 仅 manifest | Harbor 扫描策略 |

## 三、Dockerfile 通用规范

### 3.1 必填项

所有 Dockerfile **必须**包含以下元素：

1. **中文注释头**：说明镜像用途、基础镜像、维护者
2. **基础镜像显式版本**：禁止使用 `FROM xxx:latest` 或无 tag 的 `FROM xxx`
3. **LABEL 元数据**：
   ```dockerfile
   LABEL org.opencontainers.image.title="geth"
   LABEL org.opencontainers.image.description="Go-Ethereum 全功能客户端"
   LABEL org.opencontainers.image.version="1.14.0"
   LABEL org.opencontainers.image.vendor="LensChain"
   LABEL org.opencontainers.image.source="https://github.com/ethereum/go-ethereum"
   LABEL lenschain.io/ecosystem="ethereum"
   LABEL lenschain.io/category="chain-nodes"
   ```
4. **时区统一**：`ENV TZ=UTC`
5. **非 root 运行**（平台自研与可改造镜像）：
   ```dockerfile
   RUN addgroup -S lenschain && adduser -S lenschain -G lenschain
   USER lenschain
   ```
6. **HEALTHCHECK**：对外提供服务的镜像必须配置
7. **EXPOSE**：显式声明端口
8. **CMD / ENTRYPOINT**：明确启动命令

### 3.2 禁止项

- 禁止 `FROM xxx:latest` 或无 tag
- 禁止在 Dockerfile 中硬编码密钥 / 密码 / Token
- 禁止使用 `ADD` 从 URL 下载文件（校验和不可控），使用 `RUN curl` + 校验和
- 禁止 `RUN curl | sh` 等不可审计的安装方式
- 禁止以 root 运行的平台自研镜像
- 禁止在镜像中包含 `.git` / `.env` / SSH 私钥等敏感文件（使用 `.dockerignore`）

### 3.3 多阶段构建要求

平台自研镜像（`backend` / `frontend` / `sim-engine-core` / `scenario-base` / `collector-agent-*` / `judge-service` / `patch-verifier` / CTF 5 个基础开发镜像）**必须**使用多阶段构建：

```dockerfile
# build 阶段（包含完整工具链）
FROM golang:1.22-alpine AS builder
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o /out/app ./cmd/server

# runtime 阶段（最小运行时）
FROM alpine:3.19
RUN apk add --no-cache ca-certificates tzdata && \
    addgroup -S lenschain && adduser -S lenschain -G lenschain
COPY --from=builder --chown=lenschain:lenschain /out/app /app/server
USER lenschain
EXPOSE 8080
HEALTHCHECK CMD wget -q -O- http://localhost:8080/healthz || exit 1
ENTRYPOINT ["/app/server"]
```

### 3.4 镜像体积上限

| 类型 | 上限 |
|------|------|
| 平台服务 runtime 镜像（backend / sim-engine-core） | ≤ 100MB |
| 平台服务 frontend 镜像 | ≤ 250MB |
| scenario-base | ≤ 30MB |
| collector-agent-* | ≤ 50MB |
| 场景算法容器（每个） | ≤ 50MB |
| CTF 基础开发镜像 | ≤ 1.5GB |
| 链节点镜像（薄封装） | 基本等同上游 |

超限需在 PR 描述中说明原因，并在 manifest 中标记。

## 四、平台服务镜像 Dockerfile 设计

### 4.1 `backend.Dockerfile`

```dockerfile
# backend.Dockerfile
# 链镜平台后端 Go API 服务镜像
# 多阶段构建：golang:1.22-alpine 构建 → alpine:3.19 运行

# ============================
# 构建阶段
# ============================
FROM golang:1.22-alpine AS builder

ARG VERSION=dev
ARG COMMIT_SHA=unknown

WORKDIR /src

# 依赖层（利用 Docker cache）
COPY go.mod go.sum ./
RUN go mod download

# 源码层
COPY . .

RUN CGO_ENABLED=0 GOOS=linux go build \
    -ldflags="-s -w -X main.Version=${VERSION} -X main.Commit=${COMMIT_SHA}" \
    -o /out/lenschain-backend \
    ./cmd/server

# ============================
# 运行阶段
# ============================
FROM alpine:3.19

ARG VERSION=dev
ARG COMMIT_SHA=unknown

LABEL org.opencontainers.image.title="lenschain-backend"
LABEL org.opencontainers.image.description="链镜平台后端 Go API 服务"
LABEL org.opencontainers.image.version="${VERSION}"
LABEL org.opencontainers.image.revision="${COMMIT_SHA}"
LABEL org.opencontainers.image.vendor="LensChain"
LABEL lenschain.io/service="backend"

ENV TZ=UTC

RUN apk add --no-cache ca-certificates tzdata curl && \
    addgroup -S -g 1001 lenschain && \
    adduser -S -u 1001 -G lenschain lenschain && \
    mkdir -p /app/configs /app/logs && \
    chown -R lenschain:lenschain /app

WORKDIR /app

COPY --from=builder --chown=lenschain:lenschain /out/lenschain-backend /app/lenschain-backend

USER lenschain

EXPOSE 8080

HEALTHCHECK --interval=30s --timeout=5s --retries=3 --start-period=20s \
    CMD curl -fsS http://localhost:8080/healthz || exit 1

ENTRYPOINT ["/app/lenschain-backend"]
CMD ["--config", "/app/configs/config.yaml"]
```

### 4.2 `frontend.Dockerfile`

```dockerfile
# frontend.Dockerfile
# 链镜平台前端 Next.js 应用镜像（standalone 模式）
# 多阶段构建：node:20-alpine 构建 → node:20-alpine 运行

# ============================
# 依赖阶段
# ============================
FROM node:20-alpine AS deps

WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --no-audit --no-fund

# ============================
# 构建阶段
# ============================
FROM node:20-alpine AS builder

ARG NEXT_PUBLIC_API_BASE_URL
ARG NEXT_PUBLIC_WS_BASE_URL

WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .

ENV NEXT_TELEMETRY_DISABLED=1 \
    NEXT_PUBLIC_API_BASE_URL=${NEXT_PUBLIC_API_BASE_URL} \
    NEXT_PUBLIC_WS_BASE_URL=${NEXT_PUBLIC_WS_BASE_URL}

RUN npm run build

# ============================
# 运行阶段
# ============================
FROM node:20-alpine

ARG VERSION=dev

LABEL org.opencontainers.image.title="lenschain-frontend"
LABEL org.opencontainers.image.version="${VERSION}"
LABEL org.opencontainers.image.vendor="LensChain"
LABEL lenschain.io/service="frontend"

ENV TZ=UTC \
    NODE_ENV=production \
    NEXT_TELEMETRY_DISABLED=1 \
    PORT=3000

RUN apk add --no-cache curl tzdata && \
    addgroup -S -g 1001 lenschain && \
    adduser -S -u 1001 -G lenschain lenschain

WORKDIR /app

# Next.js standalone 产物
COPY --from=builder --chown=lenschain:lenschain /app/.next/standalone ./
COPY --from=builder --chown=lenschain:lenschain /app/.next/static ./.next/static
COPY --from=builder --chown=lenschain:lenschain /app/public ./public

USER lenschain

EXPOSE 3000

HEALTHCHECK --interval=30s --timeout=5s --retries=3 --start-period=15s \
    CMD curl -fsS http://localhost:3000 || exit 1

CMD ["node", "server.js"]
```

> **构建时变量**：`NEXT_PUBLIC_*` 在构建阶段注入（Next.js 编译到客户端 Bundle），CI 中按环境传入不同值；运行时无法修改。

## 五、实验环境镜像 Dockerfile 设计

### 5.1 链节点镜像：薄封装模板

链节点镜像一律 `FROM 官方镜像:固定版本`，附加平台约定的 LABEL / EXPOSE / HEALTHCHECK / 非 root 改造（能改则改），不修改上游行为。

#### 5.1.1 `chain-nodes/geth/Dockerfile`

```dockerfile
# chain-nodes/geth/Dockerfile
# Go-Ethereum 全功能客户端（基于官方镜像薄封装）
# 构建参数：--build-arg VERSION=1.14.0

ARG VERSION=1.14.0
FROM ethereum/client-go:v${VERSION}

ARG VERSION
LABEL org.opencontainers.image.title="geth"
LABEL org.opencontainers.image.description="Go-Ethereum 客户端"
LABEL org.opencontainers.image.version="${VERSION}"
LABEL org.opencontainers.image.source="https://github.com/ethereum/go-ethereum"
LABEL org.opencontainers.image.vendor="LensChain"
LABEL lenschain.io/ecosystem="ethereum"
LABEL lenschain.io/category="chain-nodes"

ENV TZ=UTC

EXPOSE 8545 8546 30303 30303/udp

HEALTHCHECK --interval=30s --timeout=10s --retries=3 --start-period=20s \
    CMD wget -qO- http://localhost:8545 \
        --post-data='{"jsonrpc":"2.0","id":1,"method":"eth_blockNumber","params":[]}' \
        --header='Content-Type: application/json' || exit 1

# 默认启动命令（开发模式）
CMD ["--dev", "--http", "--http.addr", "0.0.0.0", "--http.api", "eth,net,web3,personal,debug", "--http.corsdomain", "*", "--ws", "--ws.addr", "0.0.0.0", "--ws.api", "eth,net,web3"]
```

#### 5.1.2 `chain-nodes/fabric-peer/Dockerfile`（Fabric 薄封装）

```dockerfile
# chain-nodes/fabric-peer/Dockerfile
```

#### 5.1.3 其余 13 个链节点

其他链节点镜像（besu / ganache / hardhat-node / op-node / fabric-orderer / fabric-ca / chainmaker-node / fisco-bcos / solana-validator / substrate-node / cosmos-node / aptos-node / bitcoind）**全部采用相同结构**：`FROM 官方:版本` + `LABEL` + `ENV TZ=UTC` + `EXPOSE` + `HEALTHCHECK` + `manifest.yaml`。

## 六、镜像清单映射到 images 表

### 6.1 `seed-images.sh` 脚本行为

```bash
#!/bin/bash
# scripts/seed-images.sh
# 扫描 deploy/images/**/manifest.yaml，写入 images 和 image_versions 表

set -euo pipefail

for manifest in deploy/images/**/manifest.yaml; do
    echo "Processing $manifest"
    # 调用后端 admin API：POST /api/v1/admin/images/sync
    # body: manifest.yaml 内容
    curl -X POST "${BACKEND_URL}/api/v1/admin/images/sync" \
        -H "Authorization: Bearer ${ADMIN_TOKEN}" \
        -H "Content-Type: application/yaml" \
        --data-binary @"$manifest"
done
```

### 6.2 同步规则

| manifest 变化 | 同步行为 |
|--------------|----------|
| 新增镜像 | 插入 `images` 表 + 关联版本 |
| 新增版本 | 插入 `image_versions` 表 |
| 修改元数据（端口 / 搭配等） | 更新 `images` 表 |
| 删除镜像 | **不自动删除**，由管理员在后台标记 deprecated |
| 版本下架 | 标记 `image_versions.deprecated=true`，已有实验继续运行 |

### 6.3 与业务代码接口

后端需实现管理员接口：

- `POST /api/v1/admin/images/sync` — 接收 manifest YAML，同步到数据库（内部使用，需 super_admin）
- `GET /api/v1/admin/images/manifest-diff` — 对比 Registry 实际存在的镜像与数据库记录

这两个接口属于模块04 范围，本模块仅定义 manifest 格式和脚本调用方式。

## 七、构建流水线

### 7.1 CI 触发规则

| 触发条件 | 构建内容 |
|----------|----------|
| `backend/**` 变化 | 构建 `lenschain/backend` 镜像 |
| `frontend/**` 变化 | 构建 `lenschain/frontend` 镜像 |
| `sim-engine/core/**` 变化 | 构建 `lenschain/sim-engine-core` 镜像 |
| `sim-engine/scenarios/**` 变化 | 仅构建变化的场景镜像 |
| `deploy/images/<category>/<name>/**` 变化 | 仅构建对应实验镜像 |
| `deploy/docker/**` 变化 | 重新构建受影响平台镜像 |

### 7.2 构建步骤（统一模板）

```yaml
steps:
  - checkout
  - setup-buildx
  - login-to-registry:
      registry: registry.lianjing.com
      username: ${{ secrets.REGISTRY_USER }}
      password: ${{ secrets.REGISTRY_PASSWORD }}
  - extract-metadata:
      tags: |
        v${{ github.event.release.tag_name }}
        sha-${{ github.sha | slice(0, 8) }}
        ${{ github.ref == 'refs/heads/main' && 'staging' || '' }}
  - build-and-push:
      platforms: linux/amd64
      cache-from: type=registry,ref=${REGISTRY}/${PROJECT}/${NAME}:cache
      cache-to: type=registry,ref=${REGISTRY}/${PROJECT}/${NAME}:cache,mode=max
      build-args: |
        VERSION=${{ env.VERSION }}
        COMMIT_SHA=${{ github.sha }}
  - cosign-sign  # 仅 main / release 分支
  - trivy-scan   # 阻断 Critical 漏洞（除非 manifest 允许）
```

### 7.3 并行构建

- 不同镜像之间**完全独立**，CI 并行构建
- 同一镜像多版本按版本矩阵并行构建
- Phase 1 全量构建预计 ≤ 60 分钟；增量构建 ≤ 15 分钟

### 7.4 镜像推送前检查清单

CI 必须执行以下检查，任何一项失败阻断推送：

1. ✅ Dockerfile 通过 hadolint（Dockerfile 静态检查）
2. ✅ 镜像体积不超过 manifest 规定上限
3. ✅ 运行时非 root（平台自研镜像）
4. ✅ Trivy 扫描通过（按 `security_scan.allow_*` 规则）
5. ✅ Cosign 签名（生产分支）
6. ✅ manifest.yaml 通过 schema 校验（字段完整、JSON 格式正确）
7. ✅ 标签符合规范（禁止覆盖已发布标签）

## 八、版本管理

### 8.1 平台服务镜像版本

- 跟随平台语义化版本：`v{MAJOR}.{MINOR}.{PATCH}`
- 主版本升级：不兼容改动（数据库 schema 破坏、API 不兼容）
- 次版本升级：新功能，兼容
- 补丁版本：Bug 修复

### 8.2 实验环境镜像版本

- **上游版本号**作为主版本标签：`geth:1.14` / `fabric-peer:2.5`
- **平台内部小版本**作为后缀：`geth:1.14-lensch1` / `geth:1.14-lensch2`（当 manifest 修改、但上游未升级时使用）
- 默认版本在 manifest 的 `versions[].is_default` 标记

### 8.3 版本保留策略（BR-65）

- 每张镜像保留最近 **3 个版本**在 Registry
- CronJob 定时扫描，清理第 4 个及更旧版本
- 已有实验引用的版本**不得清理**（通过 `image_versions.ref_count > 0` 判断）

### 8.4 离线镜像包

学校私有化部署场景：

- 运维脚本 `scripts/build-offline-bundle.sh` 执行：
  1. 从 Registry 拉取所有 Phase 1 镜像
  2. `docker save -o lenschain-phase1-{version}.tar` 打包（约 50-80GB）
  3. 生成 SHA256 校验文件
- 学校运维使用 `scripts/import-offline-bundle.sh`：
  1. 校验 SHA256
  2. `docker load` 导入到本地 Docker
  3. `docker push` 到学校私有 Registry

## 九、Collector Agent 特殊设计

### 9.1 Sidecar 注入机制

混合实验启动时，后端业务代码为每个被采集的容器自动注入 Collector Agent sidecar（模块04 §10）：

```yaml
# Pod 规范示例（业务代码生成）
spec:
  containers:
    - name: geth
      image: registry.lianjing.com/chain-nodes/geth:v1.14.0
      ports:
        - containerPort: 8545
    - name: collector
      image: registry.lianjing.com/lenschain/collector-agent-ethereum:v1.0.0
      env:
        - name: ADAPTER
          value: ethereum
        - name: TARGET_URL
          value: ws://localhost:8546
        - name: SIM_ENGINE_URL
          value: ws://sim-engine-core.lenschain.svc.cluster.local:50052
        - name: SESSION_ID
          value: {session_id}
```

### 9.2 `collector_image_template` 配置

`backend/configs/config.example.yaml`:

```yaml
k8s:
  collector_image_template: "registry.lianjing.com/lenschain/collector-agent-%s:v1.0.0"
```

`%s` 由业务代码填入 `ethereum` / `fabric` / `chainmaker` / `fisco`。

### 9.3 版本升级策略

Collector Agent 升级时：

1. 构建新版本并推送
2. 更新 `config.yaml` 的 `collector_image_template` 版本号
3. 后端滚动重启（新创建的实验实例使用新版 Collector）
4. 存量实验实例 Collector **不自动升级**（避免中断采集）
