## 一、模块概述

### 1.1 模块定位

部署与运维模块是链镜平台的**基础设施交付层**，为平台自身服务（后端、前端、仿真引擎）和实验环境（60+ 区块链节点/中间件/工具镜像）提供统一的容器化构建、镜像仓库、编排部署、镜像预拉取和运维能力。

**核心职责：**

- 平台服务镜像构建（`deploy/docker/`）
- 实验环境镜像体系（`deploy/images/`）
- 本地开发环境编排（`deploy/docker-compose/`）
- 生产环境 K8s 编排（`deploy/k8s/`）
- CI/CD 流水线（`deploy/ci/`）
- 镜像仓库与预拉取策略
- 运维脚本（数据库初始化、镜像清单种子化、MinIO 初始化）

### 1.2 模块边界

| 本模块负责 | 不属于本模块（归属说明） |
|-----------|----------------------|
| 平台服务的 Dockerfile、K8s manifest、compose 文件 | 后端业务代码（→ `backend/`） |
| 实验环境镜像的 Dockerfile、镜像清单元数据文件 | 前端业务代码（→ `frontend/`） |
| 镜像预拉取 DaemonSet、实验 Namespace RBAC、Network Policy | 仿真引擎与业务模块的运行时逻辑实现（→ `sim-engine/` / `backend/`） |
| 镜像仓库（Registry）部署配置 | 镜像数据库记录的写入（→ 模块04 `images` 表的服务端实现） |
| Registry 镜像命名与标签规范 | 实验实例的 Namespace 动态创建调用（→ 模块04 service 层） |
| 本地 / 测试 / 生产三套部署环境的编排 | Collector Agent 的链适配器实现（→ 模块04 sim-engine/Collector 代码） |
| 数据库迁移脚本入口、MinIO Bucket 初始化脚本 | 业务数据迁移 SQL（→ `backend/migrations/`） |
| CI/CD 流水线配置 | 运行时监控告警规则（→ 模块08） |

> **重要说明：** 本模块**只提供基础设施与交付物**，不包含业务逻辑。平台服务的业务行为由 `backend/` `frontend/` `sim-engine/` 代码决定；模块04、05 中涉及的 "Namespace 动态创建"、"容器生命周期管理"、"Collector sidecar 注入" 等业务动作由业务代码调用 K8s API 实现，本模块只提供对应的 RBAC 权限、Network Policy、镜像清单等**前置条件**。

### 1.3 与其他模块的联动关系

| 关联模块 | 联动方式 | 具体说明 |
|---------|---------|---------|
| 模块01 用户与认证 | 提供基础设施 | 后端服务依赖 PostgreSQL / Redis，部署阶段提供这些中间件的 Deployment / Service |
| 模块02 学校与租户管理 | 提供基础设施 | 同上，依赖共享的 PostgreSQL 实例和多租户逻辑库规划 |
| 模块03 课程与教学 | 提供基础设施 | 依赖 PostgreSQL + MinIO（文件上传） |
| 模块04 实验环境 | **强依赖** | 提供全部 60+ 实验环境镜像；提供实验 Namespace 动态创建所需的 RBAC ClusterRole；提供 Network Policy 模板；提供镜像预拉取 DaemonSet；提供 Collector Agent sidecar 镜像 |
| 模块05 CTF竞赛 | **强依赖** | 复用模块04 全部镜像；额外提供 CTF 专用镜像（`ctf-web`/`ctf-crypto`/`ctf-blockchain`/`ctf-reverse`/`ctf-pwn`/`foundry`/`ghidra-server`/`sagemath` 等）；提供 `judge-service`、`patch-verifier` 镜像；提供攻防赛多链拓扑的 Namespace 规划、RBAC 与 Network Policy（每队独立队伍链 Namespace + 裁判链 Namespace + `judge-service` 跨 Namespace 代理攻击） |
| 模块06 评测与成绩 | 提供基础设施 | 依赖 PostgreSQL |
| 模块07 通知与消息 | 提供基础设施 | 依赖 PostgreSQL / Redis / NATS |
| 模块08 系统管理与监控 | 提供基础设施 | 依赖 PostgreSQL / Redis；提供备份脚本与数据库转储工具 |

### 1.4 依赖方向

```
部署与运维（模块09）
    ↓ 提供基础设施
模块01 ~ 模块08（全部业务模块）
```

模块09 位于**所有业务模块之下**，是平台运行的前置条件，但**不直接被业务代码 import**。业务代码通过容器内的服务地址（环境变量）、K8s API、Registry URL 等间接使用本模块的产出。

## 二、部署场景

### 2.1 四类部署环境

| 环境 | 用途 | 形态 | 位置 |
|------|------|------|------|
| `local-dev` | 开发者本地开发调试 | docker-compose + 本地 Go/Next.js 进程 | `deploy/docker-compose/docker-compose.dev.yml` |
| `staging` | 集成测试环境 | K8s 单节点（minikube / k3s） | `deploy/k8s/overlays/staging/` |
| `production-public` | 公网统一运营 | K8s 多节点集群 | `deploy/k8s/overlays/prod/` |
| `production-school` | 学校私有化部署 | K8s 多节点集群（可按需裁剪） | `deploy/k8s/overlays/prod/`（同 prod 叠加学校差异） |

### 2.2 不同环境的差异

| 维度 | local-dev | staging | production |
|------|-----------|---------|-----------|
| 中间件（PG/Redis/MinIO/NATS） | docker-compose 启动（宿主机端口可按本机环境避让） | K8s StatefulSet（单副本） | K8s StatefulSet（多副本 / 外接托管） |
| 平台服务 | 本地 `go run` / `npm run dev` | K8s Deployment | K8s Deployment（HPA + 多副本） |
| SimEngine Core | 本地进程 | K8s Deployment（单副本） | K8s Deployment（多副本） |
| 实验环境镜像 | 不预拉取（开发一般不起实验） | 仅预拉取 Phase 1 主线镜像 | 全量预拉取到所有节点 |
| Registry | 不需要（直接用 Docker Hub 公共镜像） | 私有 Registry（`registry.lianjing.com`） | 私有 Registry |
| TLS | 不启用 | 自签证书 | 正式证书 |
| 配置秘钥 | `.env.local` / `config.yaml` | K8s Secret（明文） | K8s Secret + 外部 KMS |
| 资源规模 | 单机 4C8G 即可 | 单节点 8C16G | 3+ 节点，每节点 ≥ 16C32G |

### 3.2 目录职责边界

严格遵守 `CLAUDE.md` §3.4 和 §7.3 的规定：

- `deploy/docker/` **仅**存放平台自身服务与平台运维运行时镜像的 Dockerfile
- `deploy/images/` **仅**存放实验环境相关镜像（链节点 / 中间件 / 工具 / 基础开发环境）
- 两者**禁止混放**。如果某个镜像既被平台使用又被实验使用（如 `redis`、`postgres`、`minio`），放在 `deploy/images/middleware/` 下，平台服务的 K8s manifest 通过镜像 Tag 引用，不重复写 Dockerfile。

> `deploy/docker/` 当前维护 8 份 Dockerfile。其中 Collector Agent 通过 `collector-agent.Dockerfile` 按 `ADAPTER` 构建参数产出 4 个镜像标签，因此最终平台与平台运维运行时镜像产物超过 10 个。

### 4.2 构建要求

- **多阶段构建：** build 阶段和 runtime 阶段分离，减小最终镜像体积
- **非 root 运行：** runtime 阶段必须创建 `lenschain` 用户，以非 root 身份启动
- **健康检查：** Dockerfile 中声明 `HEALTHCHECK`，供 Docker 和 K8s 探针使用
- **镜像体积：** backend / sim-engine-core ≤ 100MB；frontend ≤ 250MB；scenario-base ≤ 30MB；collector-agent ≤ 50MB
- **镜像标签：** `${registry}/lenschain/${service}:${version}` + `latest`；版本号使用 Git Tag / Commit SHA
- **签名：** 生产构建必须通过 CI 签名（cosign），运行时校验

### 5.2 镜像交付策略

**策略：全部镜像都在 `deploy/images/` 下有 Dockerfile 和清单元数据文件。**

- **官方镜像**：Dockerfile 使用 `FROM 官方镜像:版本` + 平台预设配置（默认端口 EXPOSE、环境变量、标准挂载点、健康检查），不做侵入性改造。这样做的理由：
  1. 模块04 BR-54 要求"每个官方镜像携带预定义配置模板"，配置模板由镜像元数据描述，Dockerfile 层面保证 EXPOSE / ENV 与模板一致
  2. 所有镜像统一推到私有 Registry，避免生产环境直接依赖 Docker Hub 速率限制
  3. 版本可控，学校私有化部署时 Registry 离线分发可行
- **平台自研镜像**（`judge-service`、`patch-verifier`、5 个 CTF 基础开发镜像、`hardhat-node`）：完整自建
- **同一镜像多版本**：Dockerfile 支持 `--build-arg VERSION=1.14` 构建多版本，版本列表由清单元数据管理

### 5.3 镜像清单元数据

每个镜像目录下包含 `manifest.yaml`，描述该镜像的元数据（对应模块04 BR-54 "镜像配置模板" 和 `images` 表字段）：

### 5.4 镜像优先级（Phase 规划）

> **重要说明：** Phase 不是“只做一小部分就交付”。**首次上线必须完整交付 Phase 1 全部镜像**，覆盖平台核心功能所需的全部链生态与工具；Phase 2 / 3 是在 Phase 1 稳定上线后按需扩展新兴链生态与前沿技术，不做也不影响平台主业务可用。

对应模块04 §3.5：

| Phase | 范围 | 是否首次上线必须 | 说明 |
|-------|------|------------------|------|
| **Phase 1** | 11 个主流链节点（EVM 全部 + Fabric 3 件 + ChainMaker + FISCO + Bitcoin）+ 全部 10 个中间件 + 全部 25 个工具（17 通用 + 8 CTF 专用）+ 全部 8 个平台自研镜像（含 4 个 Collector Agent）+ 6 个基础开发镜像（solidity-dev / go-dev / java-dev / python-dev / dapp-dev + 全部 5 个 CTF 基础开发镜像）| **✅ 必须** | 覆盖 EVM / Fabric / FISCO / ChainMaker / Bitcoin 主流链教学场景 + CTF 竞赛核心 |
| **Phase 2** | `solana-validator` + `substrate-node` + `cosmos-node` + `solana-dev` + `substrate-dev` + `cosmos-dev` | ❌ 可选 | Phase 1 上线后扩展新兴链生态 |
| **Phase 3** | `aptos-node` + `op-node` + `move-dev` + `circom-dev` | ❌ 可选 | 继续扩展前沿技术 |

**交付节奏：**

- Phase 1 内部所有镜像按 CI 并行构建推送，全部通过验收后才算完成首次上线准备
- Phase 2 / 3 作为后续迭代，允许单独发布，不阻塞 Phase 1 的完整交付
- 学校私有化部署默认随 Phase 1 全量分发；Phase 2 / 3 按学校需求按需拉取

## 七、镜像预拉取策略（BR-63 / BR-64 / BR-65）

### 7.1 预拉取时机

| 镜像类型 | 时机 |
|----------|------|
| 官方镜像 | 平台部署时，所有 K8s 节点预拉取 |
| 教师自定义镜像 | 管理员审核通过时触发全节点预拉取 |
| 版本更新 | 新版本发布时自动拉取，旧版本保留最近 3 个 |

### 7.2 实现机制

- **DaemonSet `image-prepuller`** 在所有 K8s 节点上运行，读取镜像清单 ConfigMap，执行 `docker pull` / `crictl pull`
- **清单同步**：部署脚本 `preload-images.sh` 根据 `deploy/images/**/manifest.yaml` 生成并刷新清单 ConfigMap
- **状态上报**：节点拉取结果的数据库写入与管理后台展示由模块04/05业务层实现，本模块只提供 DaemonSet 与清单前置条件
- **清理策略**：定时 CronJob 提供版本清理入口；是否结合业务引用关系保护旧版本，依赖后端镜像使用记录能力

### 7.3 启动时间保障（BR-63）

| 场景 | 目标 |
|------|------|
| 官方镜像实验 | ≤ 30s |
| 自定义镜像实验（审核通过后） | ≤ 30s |
| 冷启动（镜像未缓存） | ≤ 180s |

## 八、K8s 部署架构

### 8.1 Namespace 规划

| Namespace | 用途 | 生命周期 |
|-----------|------|----------|
| `lenschain` | 平台自身服务（后端 / 前端 / sim-engine / 中间件） | 永久 |
| `lenschain-system` | 平台运维（Registry / image-prepuller / backup Job） | 永久 |

| `ctf-{competition_id}-{team_id}` | CTF 攻防赛单支队伍 | 竞赛期间 |
| `ctf-{competition_id}-judge` | CTF 攻防赛裁判链 | 竞赛期间 |

### 8.2 RBAC 设计（BR-67 前置条件）

后端服务需要动态创建 / 删除 `exp-*` 和 `ctf-*` Namespace，本模块提供 ClusterRole：

```yaml
# rbac/backend-clusterrole.yaml（示例）
rules:
  - apiGroups: [""]
    resources: ["namespaces", "pods", "services", "configmaps", "secrets", "persistentvolumeclaims"]
    verbs: ["*"]
  - apiGroups: ["apps"]
    resources: ["deployments", "statefulsets"]
    verbs: ["*"]
  - apiGroups: ["networking.k8s.io"]
    resources: ["networkpolicies", "ingresses"]
    verbs: ["*"]
  - apiGroups: [""]
    resources: ["pods/exec", "pods/log"]
    verbs: ["create", "get"]
```

### 8.3 Network Policy

**实验 Namespace 隔离**（模块04 BR-67）：
- 默认：`exp-*` Namespace 间**禁止互访**
- 允许：`exp-*` → `lenschain`（访问后端 API、MinIO、数据库代理）
- 允许：`exp-*` → 外网（镜像拉取、依赖下载，可按学校策略收紧）

**CTF 竞赛隔离**（模块05 §11.2）：
- `ctf-*-team-*` Namespace 间**禁止互访**
- `ctf-*-team-*` → `ctf-*-judge` 允许（提交攻击证据给裁判）
- `ctf-*-judge` → `ctf-*-team-*` 允许（下发题目、结算）

### 8.4 Kustomize 分层

```
base/     # 完整的资源定义（默认值）
overlays/
├── dev/       # 资源最小化，单副本，关闭 HPA，关闭 TLS
├── staging/   # 单节点，自签证书，部分预拉取
└── prod/      # 多副本 + HPA + PDB + 正式证书 + 监控
```

每个 overlay 通过 `kustomization.yaml` 覆盖副本数、镜像标签、ConfigMap、Secret。

### 8.5 docker-compose（本地开发）

`deploy/docker-compose/docker-compose.dev.yml` 只起**中间件**和**可选的 sim-engine-core**，后端和前端由开发者本地 `go run` / `npm run dev` 启动：

**必含服务：**
- postgres（5432）
- redis（6379）
- minio（9000 / 9001）
- nats（4222 / 8222）
- sim-engine-core（50051 / 50052，可选）

**不含：**
- 后端 / 前端（开发者本地跑）
- K8s（开发一般不起实验）
- 实验环境镜像（按需 `docker pull`）

**本地宿主机端口映射约定：**
- 为避免与开发者机器上其他项目冲突，`docker-compose.dev.yml` 可调整宿主机映射端口
- 当前默认偏移为 `+10`：
  - PostgreSQL `5442 -> 5432`
  - Redis `6389 -> 6379`
  - MinIO API `9010 -> 9000`
  - MinIO Console `9011 -> 9001`
  - NATS Client `4232 -> 4222`
  - NATS Monitoring `8232 -> 8222`
  - SimEngine gRPC `50061 -> 50051`
  - SimEngine WS `50062 -> 50052`
- 容器内标准端口、K8s Service 端口、镜像默认端口和 manifest 不因本地映射变化而修改

## 十一、业务规则

### 11.1 交付规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-01 | 平台镜像与实验镜像分离 | `deploy/docker/` 仅放平台镜像，`deploy/images/` 仅放实验镜像，不得混放 |
| BR-02 | 所有镜像都有 Dockerfile | 即使是官方镜像，也必须有薄 Dockerfile 声明 EXPOSE / ENV / 标签 |
| BR-03 | 所有镜像都有 manifest.yaml | 描述元数据，供 `seed-images.sh` 写入 `images` 表 |
| BR-04 | 镜像统一推送到私有 Registry | 不直接依赖 Docker Hub |
| BR-05 | 非 root 运行 | 平台服务镜像 runtime 必须以非 root 身份启动 |
| BR-06 | 多阶段构建 | 平台服务镜像必须使用多阶段构建，runtime 体积符合上限 |

### 11.2 Registry 规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-07 | 命名空间划分严格 | 按 `lenschain/` `chain-nodes/` `middleware/` `tools/` `base/` `scenarios/` 分类推送 |
| BR-08 | 版本号显式 | 除 Phase 1 主线镜像外，禁止只用 `latest` 标签 |
| BR-09 | 双标签策略 | CI 构建时同时打语义化版本标签和 Git Commit SHA 标签 |
| BR-10 | 离线分发 | 学校私有化部署支持离线镜像包，通过 `docker save/load` 导入 |

### 11.3 K8s 部署规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-11 | Namespace 隔离 | 平台服务、运维、实验、CTF 各自独立 Namespace |
| BR-12 | 实验 Namespace 动态创建 | 业务代码通过 K8s API 调用创建 `exp-{instance_id}`，本模块只提供 RBAC |
| BR-13 | Network Policy 默认拒绝 | 实验 / CTF Namespace 默认禁止跨 Namespace 通信，按需开白名单 |
| BR-14 | 资源限制强制 | 所有 Pod 必须声明 `resources.requests` 和 `resources.limits` |
| BR-15 | 探针必填 | 平台服务 Deployment 必须配置 `livenessProbe` 和 `readinessProbe` |
| BR-16 | Kustomize 分层 | base 保持中性，overlay 承担差异化，不允许 base 写死环境值 |

### 11.4 镜像预拉取规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-17 | 官方镜像全量预拉取 | 平台部署时拉取所有官方镜像到所有节点（对应模块04 BR-63） |
| BR-18 | 审核触发预拉取 | 教师自定义镜像审核通过后由业务代码触发预拉取（对应 BR-64） |
| BR-19 | 版本保留 3 个 | 每张镜像保留最近 3 个版本（对应 BR-65），CronJob 定时清理 |
| BR-20 | 预拉取状态可查 | 本模块提供预拉取前置能力，具体状态聚合与管理后台展示由模块04/05业务层实现 |

### 11.5 CI/CD 规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-21 | 路径触发 | CI 按变更路径触发对应流水线（后端改动只跑后端 CI） |
| BR-22 | 测试必须通过 | lint / unit test 不通过禁止构建镜像 |
| BR-23 | 镜像签名 | 生产环境镜像必须通过 cosign 签名 |
| BR-24 | Secret 管理 | CI 使用 GitHub Secrets / 外部 KMS，不允许明文写入代码 |
| BR-25 | staging 自动部署 | main 合并后自动部署到 staging；prod 需人工审批 |

## 十二、非功能要求

### 12.1 性能

| 项 | 目标 |
|----|------|
| 实验启动（官方镜像，已预拉取） | ≤ 30s |
| 实验启动（冷启动） | ≤ 180s |
| 平台服务启动（K8s Pod 就绪） | ≤ 30s |
| Registry 镜像拉取（千兆内网） | ≥ 100MB/s |
| 镜像预拉取（全量 Phase 1） | ≤ 30min |

### 12.2 可用性

| 项 | 目标 |
|----|------|
| 平台服务可用性（生产） | 99.9% |
| 滚动升级 | 零停机（RollingUpdate + PDB） |
| Registry 可用性 | 99.95%（高可用部署） |

### 12.3 安全

| 项 | 要求 |
|----|------|
| 容器非 root | 平台服务镜像必须 |
| 镜像签名 | 生产镜像必须 cosign 签名 |
| TLS | 生产环境全链路 TLS，包括内部 gRPC |
| Secret 加密 | 生产环境使用 SealedSecret / 外部 KMS，禁止明文 |
| 最小权限 RBAC | ServiceAccount 按最小权限分配 |

### 12.4 可维护性

| 项 | 要求 |
|----|------|
| 所有 Dockerfile 有中文注释 | 说明基础镜像选择理由、关键步骤 |
| 所有 manifest.yaml 字段完整 | 覆盖 `images` 表全部字段 |
| K8s manifest 有 labels / annotations | `app.kubernetes.io/*` 规范标签 |
| 部署文档与 code 同步 | 修改部署逻辑必须同步更新 `docs/modules/09-*/` 和 `deploy/README.md` |
