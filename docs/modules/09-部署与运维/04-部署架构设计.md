### 2.2 base 设计原则

- **中性**：base 不含任何环境相关值（副本数、资源配额、镜像标签、域名）
- **可运行**：base 必须能独立 `kustomize build` 通过并部署到开发环境
- **最小依赖**：不依赖 overlay 即可部署（但生产一定要 overlay）

### 2.3 overlay 职责

| overlay | 主要覆盖点 |
|---------|-----------|
| `dev` | 副本数=1，资源 requests 最小，关闭 HPA/PDB，关闭 TLS，Secret 明文 |
| `staging` | 副本数=1，自签 TLS，SealedSecrets，部分镜像预拉取 |
| `prod` | 副本数≥3，HPA 生效，PDB 配置，正式 TLS，KMS Secret，全量镜像预拉取 |

**Secret 交付约定：**

- `deploy/k8s/overlays/staging/secrets/` 和 `deploy/k8s/overlays/prod/secrets/` 必须始终存在，并被各自 overlay 的 `kustomization.yaml` 引用
- `deploy/k8s/overlays/dev/secrets/` 提供本地开发可直接部署的明文 Secret
- `deploy/k8s/base/` 不再携带可直接部署的业务敏感 Secret；数据库、MinIO、后端鉴权密钥统一由 overlay 交付
- 仓库可提交占位 Secret / SealedSecret 入口文件，但**禁止**直接以 `CHANGE_ME` 部署到实际环境
- 实际交付时至少要覆盖：
  - `backend-secret`
  - `postgres-secret`
  - `minio-secret`
  - `registry-pull-secret`
- 动态实验/CTF Namespace 需要复用 `registry-pull-secret`，由后端运行时同步到目标命名空间

## 三、关键 K8s 资源示例

### 3.1 backend Deployment（base）

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend
  namespace: lenschain
  labels:
    app.kubernetes.io/name: backend
    app.kubernetes.io/component: api
    app.kubernetes.io/part-of: lenschain
    app.kubernetes.io/managed-by: kustomize
spec:
  replicas: 1                             # overlay 覆盖
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  selector:
    matchLabels:
      app.kubernetes.io/name: backend
  template:
    metadata:
      labels:
        app.kubernetes.io/name: backend
    spec:
      serviceAccountName: lenschain-backend
      securityContext:
        runAsNonRoot: true
        runAsUser: 1001
        fsGroup: 1001
      containers:
        - name: backend
          image: registry.lianjing.com/lenschain/backend:v1.0.0   # overlay 覆盖标签
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 8080
              name: http
          env:
            - name: TZ
              value: UTC
            - name: LENSCHAIN_DATABASE_HOST
              value: postgres
            - name: LENSCHAIN_DATABASE_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: backend-secret
                  key: database-password
            - name: LENSCHAIN_JWT_ACCESS_SECRET
              valueFrom:
                secretKeyRef:
                  name: backend-secret
                  key: jwt-access-secret
            - name: LENSCHAIN_JWT_REFRESH_SECRET
              valueFrom:
                secretKeyRef:
                  name: backend-secret
                  key: jwt-refresh-secret
          volumeMounts:
            - name: config
              mountPath: /app/configs
              readOnly: true
          resources:
            requests:
              cpu: 250m
              memory: 256Mi
            limits:
              cpu: 1000m
              memory: 1Gi
          livenessProbe:
            httpGet: { path: /healthz, port: 8080 }
            initialDelaySeconds: 20
            periodSeconds: 30
          readinessProbe:
            httpGet: { path: /readyz, port: 8080 }
            initialDelaySeconds: 5
            periodSeconds: 10
          startupProbe:
            httpGet: { path: /startupz, port: 8080 }
            failureThreshold: 30
            periodSeconds: 2
      volumes:
        - name: config
          configMap:
            name: backend-config
      imagePullSecrets:
        - name: registry-pull-secret
```

### 4.1 完整清单

```yaml
# deploy/docker-compose/docker-compose.dev.yml
# 本地开发环境：启动中间件，后端/前端/sim-engine 由开发者本地进程运行

services:

      - "9010:9000"

      test: ["CMD", "curl", "-fsS", "http://localhost:9000/minio/health/ready"]

  nats:
    image: nats:2.10-alpine
    container_name: lenschain-nats
    ports:
      - "4232:4222"
      - "8232:8222"
    command: ["-js", "-m", "8222"]
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost:8222/healthz"]
      interval: 10s

  sim-engine:
    image: registry.lianjing.com/lenschain/sim-engine-core:dev
    container_name: lenschain-sim-engine
    profiles: ["full"]                   # 默认不启动，需 --profile full
    ports:
      - "50061:50051"
      - "50062:50052"
    depends_on:
      postgres: { condition: service_healthy }
      redis: { condition: service_healthy }

networks:
  default:
    name: lenschain-dev
```

### 4.2 使用说明

```bash
# 只启动中间件（默认）
docker compose -f deploy/docker-compose/docker-compose.dev.yml up -d

# 启动中间件 + sim-engine
docker compose -f deploy/docker-compose/docker-compose.dev.yml --profile full up -d

# 清理所有数据
docker compose -f deploy/docker-compose/docker-compose.dev.yml down -v
rm -rf deploy/docker-compose/data/
```

### 4.3 数据持久化

- PostgreSQL / Redis / MinIO 数据挂载到 `deploy/docker-compose/data/` 目录
- `.gitignore` 必须包含 `deploy/docker-compose/data/`
- 开发重置：删除 data 目录即可

### 4.4 与 config.yaml 对齐

开发者本地运行后端时，连接参数应与 compose 的**宿主机映射端口**保持一致：

```yaml
database: { host: localhost, port: 5442, user: lenschain, password: lenschain }
redis: { host: localhost, port: 6389 }
minio: { endpoint: localhost:9010, access_key: minioadmin, secret_key: minioadmin }
nats: { url: nats://localhost:4232 }
server: { port: 8080 }
```

> 说明：本地开发允许调整宿主机映射端口以避开其他项目，但容器内标准端口和 K8s/镜像约定保持不变。

### 5.2 backend.yml 示例

```yaml
name: backend
on:
  push:
    branches: [main, develop]
    paths: ['backend/**', 'deploy/docker/backend.Dockerfile']
  pull_request:
    paths: ['backend/**']

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with: { go-version: '1.22' }
      - run: cd backend && go mod download
      - run: cd backend && golangci-lint run
      - run: cd backend && go test ./... -race -coverprofile=coverage.out

  build:
    needs: test
    if: github.ref == 'refs/heads/main' || startsWith(github.ref, 'refs/tags/')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: registry.lianjing.com
          username: ${{ secrets.REGISTRY_USER }}
          password: ${{ secrets.REGISTRY_PASSWORD }}
      - id: meta
        uses: docker/metadata-action@v5
        with:
          images: registry.lianjing.com/lenschain/backend
          tags: |
            type=ref,event=tag
            type=sha,prefix=sha-,format=short
            type=raw,value=staging,enable=${{ github.ref == 'refs/heads/main' }}
      - uses: docker/build-push-action@v5
        with:
          context: .
          file: deploy/docker/backend.Dockerfile
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          build-args: |
            VERSION=${{ github.ref_name }}
            COMMIT_SHA=${{ github.sha }}
          cache-from: type=registry,ref=registry.lianjing.com/lenschain/backend:cache
          cache-to: type=registry,ref=registry.lianjing.com/lenschain/backend:cache,mode=max
      - name: Scan
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: registry.lianjing.com/lenschain/backend:sha-${{ github.sha }}
          severity: CRITICAL
          exit-code: '1'
      - name: Sign
        if: startsWith(github.ref, 'refs/tags/')
        env:
          COSIGN_KEY: ${{ secrets.COSIGN_PRIVATE_KEY }}
          COSIGN_PASSWORD: ${{ secrets.COSIGN_PASSWORD }}
        run: cosign sign --key env://COSIGN_KEY registry.lianjing.com/lenschain/backend:${{ github.ref_name }}
```

### 5.3 images.yml 批量构建

```yaml
name: images
on:
  push:
    branches: [main]
    paths: ['deploy/images/**']

jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.detect.outputs.matrix }}
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 2 }
      - id: detect
        run: |
          CHANGED=$(git diff --name-only HEAD~1 HEAD -- deploy/images/ | \
                    awk -F/ '{print $3"/"$4}' | sort -u)
          MATRIX=$(echo "$CHANGED" | jq -R -s -c 'split("\n")[:-1] | map({path: .})')
          echo "matrix={\"include\":$MATRIX}" >> $GITHUB_OUTPUT

  build:
    needs: detect-changes
    if: fromJson(needs.detect-changes.outputs.matrix).include != null
    runs-on: ubuntu-latest
    strategy:
      matrix: ${{ fromJson(needs.detect-changes.outputs.matrix) }}
    steps:
      - uses: actions/checkout@v4
      - name: Parse manifest
        id: manifest
        run: |
          MANIFEST=deploy/images/${{ matrix.path }}/manifest.yaml
          # 解析 category / name / versions，输出到 $GITHUB_OUTPUT
      - uses: docker/login-action@v3
        with:
          registry: registry.lianjing.com
          username: ${{ secrets.REGISTRY_USER }}
          password: ${{ secrets.REGISTRY_PASSWORD }}
      - name: Build versions
        run: |
          for VERSION in $(yq '.versions[].tag' $MANIFEST); do
            docker buildx build \
              --build-arg VERSION=$VERSION \
              -t registry.lianjing.com/${{ steps.manifest.outputs.project }}/${{ steps.manifest.outputs.name }}:$VERSION \
              --push deploy/images/${{ matrix.path }}
          done
```

### 6.1 实验 PV 延迟清理

```yaml

kind: CronJob
metadata:
  name: pv-cleanup
  namespace: lenschain-system
spec:
  schedule: "0 */6 * * *"                  # 每 6 小时
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: lenschain-backup
          containers:
            - name: cleanup
              image: registry.lianjing.com/lenschain/pv-cleanup:v1.0.0
              env:
                - name: RETENTION_HOURS
                  value: "24"                # exp-* Namespace 销毁后保留 24 小时再清 PV
          restartPolicy: OnFailure
```

### 6.3 数据库备份

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: db-backup
  namespace: lenschain-system
spec:
  schedule: "0 2 * * *"                    # 每日 02:00 全量备份
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: backup
              image: registry.lianjing.com/middleware/postgres:15
              command:
                - /bin/bash
                - -c
                - |
                  pg_dump -h postgres.lenschain.svc.cluster.local \
                          -U lenschain lenschain \
                          | gzip > /backup/lenschain-$(date +%F).sql.gz
                  mc cp /backup/lenschain-$(date +%F).sql.gz minio/lenschain-backups/
              env:
                - name: PGPASSWORD
                  valueFrom: { secretKeyRef: { name: postgres-secret, key: password } }
```

增量备份（每 6 小时 WAL 归档）由 PostgreSQL 内置机制实现，配置在 postgres StatefulSet 的启动参数中。

## 九、监控与告警基线

> 当前仓库已为后续监控与告警体系预留基础设计口径，但 `deploy/` 尚未完整提交 Prometheus Operator / Grafana / Alertmanager 等全套监控部署资源。
> 因此本节内容应视为推荐架构与后续交付目标，而不是“当前全部已落地”的事实清单。

### 9.1 Prometheus ServiceMonitor

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: backend
  namespace: monitoring
spec:
  selector:
    matchLabels: { app.kubernetes.io/name: backend }
  namespaceSelector:
    matchNames: [lenschain]
  endpoints:
    - port: http
      path: /metrics
      interval: 30s
```

### 9.2 基线告警规则

| 告警 | 条件 | 严重度 |
|------|------|--------|
| 平台服务不可用 | `up == 0` 持续 2min | Critical |
| 平台服务 5xx 率 | 5xx 率 > 5% 持续 5min | High |
| 平台服务延迟 | P95 > 1s 持续 10min | Medium |
| DB 连接池耗尽 | 空闲连接 < 5% 持续 5min | High |
| Redis 命中率 | < 80% 持续 30min | Low |
| 实验 Namespace 数量 | > 配额 80% | Medium |
| 镜像预拉取失败 | 任一节点失败 | Medium |
| PV 使用率 | > 85% | High |

后续告警规则应对接模块08 告警引擎，并通过模块07 发送给管理员；当前 `deploy/` 仅提供基础部署前提与推荐口径。
