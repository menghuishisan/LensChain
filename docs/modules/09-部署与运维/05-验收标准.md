# 部署与运维模块 — 验收标准

> 模块状态：🔵 待讨论
> 文档版本：v1.0

---

## 一、文档目的

本文档定义部署与运维模块交付物的验收标准，采用 Given-When-Then 格式，覆盖镜像构建、Registry、K8s 部署、镜像预拉取、RBAC、Network Policy、CI/CD、备份、性能、安全等各维度。所有验收项逐条对应 `01-功能需求说明.md` 的功能清单（D-01 ~ D-27）、业务规则（BR-01 ~ BR-25）和模块04/05 中涉及本模块的规则（BR-63 / BR-64 / BR-65 / BR-67 等）。

## 二、镜像构建验收

### AC-01：平台服务镜像多阶段构建（D-01 / BR-06）

```
Given 开发者执行 docker build -f deploy/docker/backend.Dockerfile .
When 构建完成
Then 生成的镜像通过 docker history 可看到 2 个阶段（builder + runtime）
And runtime 阶段体积 ≤ 100MB
And docker inspect 显示 User 为非 root（1001）
And 启动容器后 /healthz 端点返回 200
```

### AC-02：所有 Dockerfile 通过 hadolint（BR-02）

```
Given CI 对 deploy/docker/**.Dockerfile 和 deploy/images/**/Dockerfile 运行 hadolint
When 检查完成
Then 没有 ERROR 级别告警
And WARNING 级别告警有白名单记录
```

### AC-03：所有镜像有 manifest.yaml（BR-03）

```
Given CI 遍历 deploy/images/**/
When 检查每个镜像目录
Then 必须存在 manifest.yaml 文件
And manifest.yaml 通过 schema 校验（字段完整、类型正确）
And name / category / ecosystem / versions / typical_companions / documentation_url 必填字段非空
```

### AC-04：镜像体积上限（BR-06）

```
Given CI 构建镜像完成
When 检查镜像体积
Then backend / sim-engine-core ≤ 100MB
And frontend ≤ 250MB
And scenario-base ≤ 30MB
And collector-agent-* ≤ 50MB
And CTF 基础开发镜像 ≤ 1.5GB
And 超限必须在 PR 描述中说明原因
```

### AC-05：非 root 运行（BR-05）

```
Given 平台自研镜像（backend / frontend / sim-engine-core / scenario-base / collector-agent / judge-service / patch-verifier）
When docker run 启动容器
Then 容器内 whoami 返回 lenschain / developer / scenario / collector / judge 等非 root 用户
And 进程 UID 非 0
```

### AC-06：HEALTHCHECK 配置（§3.1）

```
Given 对外提供服务的镜像
When 查看 Dockerfile
Then 必须包含 HEALTHCHECK 指令
And docker inspect 显示 Health 状态能正常返回 healthy
```

## 三、Registry 验收

### AC-07：Registry 命名空间划分（BR-07）

```
Given Harbor 部署完成
When 访问 Harbor UI
Then 至少存在 6 个 Project：lenschain / chain-nodes / middleware / tools / base / scenarios
And 每个 Project 有对应的镜像推送权限配置
```

### AC-08：镜像标签双标签策略（BR-09）

```
Given CI 构建并推送镜像
When 查询 Registry
Then 同一次构建推送两个标签：语义化版本（如 v1.0.0）和 sha-{前 8 位}
And 平台服务镜像 main 分支额外推送 staging 标签
```

### AC-09：镜像签名（BR-23）

```
Given 生产镜像（tag 为 v*.*.*）
When 执行 cosign verify --key cosign.pub registry.lianjing.com/lenschain/backend:v1.0.0
Then 验证通过
And 若生产环境接入镜像签名准入控制，则未签名镜像应被拒绝部署
```

### AC-10：漏洞扫描阻断（BR-05 / §7.4）

```
Given 平台自研镜像包含 Critical 漏洞
When CI 执行 Trivy 扫描
Then 构建失败，禁止推送

Given CTF 漏洞镜像（manifest.security_scan.allow_critical=true）包含 Critical 漏洞
When CI 执行 Trivy 扫描
Then 记录漏洞但不阻止推送
```

### AC-11：离线镜像包（BR-10）

```
Given 运维执行 scripts/build-offline-bundle.sh
When 执行完成
Then 生成 lenschain-phase1-{version}.tar 文件
And 同时生成 SHA256 校验文件
And 学校侧执行 scripts/import-offline-bundle.sh 后，docker images 可看到全部 Phase 1 镜像
And 推送到学校私有 Registry 成功
```

## 四、K8s 部署验收

### AC-12：Namespace 规划（§4.1）

```
Given 平台首次部署完成
When kubectl get namespace
Then 至少存在 lenschain / lenschain-system / monitoring
And 所有平台 Namespace 带 app.kubernetes.io/managed-by=lenschain 标签
```

### AC-13：平台服务 Deployment（D-09 / BR-14 / BR-15）

```
Given 执行 kubectl apply -k deploy/k8s/overlays/staging/
When 等待 10 分钟
Then kubectl -n lenschain get deployment 显示 backend / frontend / sim-engine 全部 Available
And 每个 Pod resources.requests 和 resources.limits 均已声明
And 每个 Pod 的 livenessProbe / readinessProbe / startupProbe 均已配置
And 滚动升级过程中服务不中断（HTTP 请求不返回 5xx）
```

### AC-14：中间件 StatefulSet（D-10）

```
Given 部署 staging overlay
When kubectl -n lenschain get statefulset
Then postgres / redis / minio / nats 全部 Ready=1/1
And kubectl exec 进入 postgres 容器，pg_isready 返回正常
And MinIO Console 可访问，能看到默认 Bucket
```

### AC-15：实验 Namespace 动态创建（BR-12 / 模块04 BR-67）

```
Given 学生启动一个实验
When 后端调用 K8s API 创建 Namespace
Then 创建成功，Namespace 名称为 exp-{instance_id}
And 带有 app.kubernetes.io/managed-by=lenschain 与 lenschain.io/namespace-type=experiment / instance-id / school-id 等标签
And Pod 能在该 Namespace 内成功启动
And 不同实验实例的同端口容器（如多个 8545）不冲突
```

### AC-16：Kustomize base 中性（BR-16）

```
Given 直接 kubectl apply -k deploy/k8s/base/
When 部署完成
Then 可以运行但资源配置是开发规格（副本数=1，无 TLS）
And base 文件中没有硬编码的域名、生产镜像标签、Secret 值
```

### AC-17：生产 overlay 差异化（§2.3）

```
Given 执行 kubectl apply -k deploy/k8s/overlays/prod/
When 部署完成
Then backend / frontend / sim-engine 副本数 ≥ 3
And HPA 生效（kubectl get hpa 能看到）
And PDB 生效（kubectl get pdb 能看到）
And Ingress 使用正式 TLS 证书（不是自签）
And Secret 使用 SealedSecrets 或 KMS

### AC-17A：overlay Secret 入口完整

```
Given 检查 deploy/k8s/overlays/staging/ 和 deploy/k8s/overlays/prod/
When 查看 kustomization.yaml
Then 必须显式包含 secrets/ 目录
And secrets/ 目录内至少声明 backend-secret / postgres-secret / minio-secret / registry-pull-secret 的交付入口
And 占位 Secret 不允许直接用于真实环境部署
```

## 五、RBAC 与 Network Policy 验收

### AC-18：后端 ClusterRole（§8.2）

```
Given 后端 ServiceAccount lenschain-backend
When 调用 K8s API 创建 Namespace
Then 创建成功

Given 后端 ServiceAccount
When 尝试访问 kube-system Namespace
Then 拒绝（403 Forbidden）

Given 后端 ServiceAccount
When 尝试修改 ClusterRole
Then 拒绝
```

### AC-19：实验 Namespace 入站隔离（BR-13 / 模块04 BR-67）

```
Given 两个实验实例 exp-A 和 exp-B
When exp-A 内 Pod 尝试访问 exp-B 内的 Service
Then 连接被 Network Policy 拒绝（超时）

Given lenschain Namespace 的 backend Pod
When 访问 exp-A 内的 Pod（通过 kubectl exec）
Then 访问成功
```

### AC-20：实验 Namespace 出站限制（§9.2）

```
Given exp-A 内 Pod
When 访问 Kube DNS（53）
Then 允许

Given exp-A 内 Pod
When 访问 registry.lianjing.com（拉取镜像）
Then 允许

Given exp-A 内 Pod
When 访问 lenschain Namespace 的 postgres:5432
Then 拒绝（禁止访问业务数据）

Given exp-A 内 Pod
When 访问 lenschain Namespace 的 backend:8080
Then 允许（Collector 上报通道）
```

### AC-21：CTF 队伍隔离（§9.3）

```
Given CTF 攻防赛两支队伍 ctf-1-team-A 和 ctf-1-team-B
When team-A 内 Pod 尝试访问 team-B 内链节点
Then 拒绝

Given team-A 向 ctf-1-judge 提交攻击证据
When 提交通过后端 API 中转
Then 允许
```

### AC-21A：攻防赛队伍 Namespace 之间完全隔离（§9.3）

```
Given 攻防赛进行中，team-A 和 team-B 各自队伍链运行中
When team-A 内 Pod 执行 curl http://chain.ctf-1-team-B.svc.cluster.local:8545
Then 连接超时（被 NetworkPolicy 拒绝）

Given team-A 内 Pod 尝试 nslookup ctf-1-team-B Namespace 内的 Service
When DNS 解析
Then 解析失败或返回的 IP 不可达

Given team-A 内 Pod 尝试 RPC 直接连接 team-B 队伍链发送交易
When 发起 eth_sendTransaction
Then 全部失败，无任何攻击交易能在 team-B 链上执行
```

### AC-21B：攻防赛队伍 → 裁判链禁止直连（§9.3）

```
Given 攻防赛进行中
When team-A 内 Pod 直接尝试访问 ctf-1-judge Namespace 的链 RPC（修改 Token 余额）
Then 拒绝（NetworkPolicy 阻断）

Given team-A 通过后端 API POST /api/v1/ctf/competitions/:id/attacks 提交攻击
When 后端鉴权通过后转发给 judge-service
Then 攻击被代理执行
And 队伍无法绕过后端直接操作裁判链
```

### AC-21C：judge-service 跨 Namespace 代理攻击（§9.3）

```
Given judge-service 运行在 ctf-1-judge Namespace
And team-A 提交的 exploit 目标为 team-B
When judge-service 在 team-B 队伍链 RPC 执行 exploit 交易
Then 交易成功执行
And 链状态变化被 patch-verifier 验证
And 结果记入裁判链 Token 流水

Given patch-verifier 部署 team-B 提交的补丁到 team-B 队伍链
When 部署完成
Then 替换原合约成功
And 队伍 Namespace 内的 Pod 无法干扰此过程
```

### AC-21D：跨竞赛隔离（§9.3）

```
Given 同时进行两场竞赛 ctf-1 和 ctf-2
When ctf-1-team-A 内 Pod 尝试访问 ctf-2-team-A Namespace
Then 拒绝（namespaceSelector + competition-id 标签匹配天然隔离）
```

## 六、镜像预拉取验收

### AC-22：Phase 1 全量预拉取（D-13 / BR-17 / 模块04 BR-63）

```
Given 平台部署完成并执行 scripts/preload-images.sh
When DaemonSet image-prepuller 在所有节点运行完成
Then 每个节点的 crictl images 能看到 Phase 1 全部镜像（≥ 50 个）
And 预拉取总耗时 ≤ 30 分钟（千兆内网）
And 管理后台显示所有镜像的拉取状态为 SUCCESS
```

### AC-23：教师自定义镜像审核后预拉取（BR-18 / 模块04 BR-64）

```
Given 教师上传自定义镜像并通过审核
When 业务代码触发预拉取
Then DaemonSet 在所有节点拉取该镜像
And 拉取完成后的数据库状态更新与管理后台展示由模块04/05业务层负责
```

### AC-24：版本保留策略（BR-19 / 模块04 BR-65）

```
Given 某镜像已有 5 个版本
When 每周日 CronJob image-version-gc 运行
Then Registry 中保留最近 3 个版本
And 若业务侧已接入镜像引用计数，则被引用版本不应被清理
And 若业务侧已接入审计能力，则旧版本删除事件应写入审计日志
```

### AC-25：冷启动启动时间（§12.1）

```
Given 某实验镜像未预拉取到节点
When 学生启动实验
Then 实验启动（含镜像拉取 + 容器启动）≤ 180s

Given 某实验镜像已预拉取
When 学生启动实验
Then 实验启动 ≤ 30s
```

## 七、CI/CD 验收

### AC-26：路径触发（D-17-D-20 / BR-21）

```
Given PR 只修改 backend/** 下的文件
When PR 创建
Then 只触发 backend.yml 流水线
And 不触发 frontend.yml / images.yml
```

### AC-27：测试阻断构建（BR-22）

```
Given backend 单元测试失败
When CI 运行
Then build job 不执行
And 镜像未推送到 Registry
```

### AC-28：staging 自动部署（BR-25 / D-21）

```
Given PR 合并到 main
When deploy-staging.yml 触发
Then 自动执行 kubectl apply -k deploy/k8s/overlays/staging/
And 部署完成后 smoke test 通过（/api/v1/healthz 返回 200）
```

### AC-29：prod 人工审批（BR-25）

```
Given 打 Git Tag v1.0.0
When deploy-prod.yml 触发
Then 停留在 environment=production 审批阶段
And 运维审批后才执行 kubectl apply
And 部署过程先 backend，健康检查通过后再 frontend 和 sim-engine
```

### AC-30：CI Secret 管理（BR-24）

```
Given 检查 deploy/ci/.github/workflows/ 下所有 yml
When grep -r 'password\|secret\|key' --include='*.yml'
Then 没有明文密钥，全部通过 ${{ secrets.* }} 引用
```

```

```

```
Given 5 个 K8s 节点，每节点并发 3
When 执行 scripts/preload-images.sh

```

Then 压测工具持续发送请求（100 QPS）

```

```
Given 生产环境运行一个月
When 统计平台服务可用时间

```

When 其中一个 Harbor 节点宕机
Then 拉取/推送不受影响
And 可用率 ≥ 99.95%
```

```
Given 生产环境 3 节点
When 关闭一台节点
Then 平台服务 Pod 自动迁移到其他节点
And 3 分钟内业务恢复
And 数据库主从切换成功（如有配置）

## 十二、安全验收

### AC-45：容器非 root（§12.3）

```
Given 平台自研镜像在 K8s 运行
When kubectl exec 进入容器执行 id
Then 返回 UID 为 1001（非 root）
```

### AC-46：TLS 全链路（§12.3）

```
Given 生产环境
When 访问 https://app.lianjing.com
Then TLS 证书有效且来自可信 CA
And 内部 gRPC（SimEngine）启用 mTLS（可选）
And 拒绝明文 HTTP（301 跳转或直接拒绝）
```

### AC-47：Secret 加密（§12.3）

```
Given 生产环境
When kubectl get secret -o yaml
Then data 字段经过 Sealed Secrets 加密或 KMS 加密
And Git 仓库中没有明文 Secret
```

### AC-48：RBAC 最小权限（§8.3）

```
Given 检查所有 ClusterRole
When 审计权限范围
Then 没有角色拥有 cluster-admin 或 * / *
And 每个 ServiceAccount 权限与其实际需求严格对应
```

### AC-49：Trivy 定期扫描（D-22 / §12.3）

```
Given security-scan.yml 每晚 02:00 UTC 运行
When 扫描所有 Registry 镜像
Then 至少生成并保存扫描报告
And 若后续接入通知系统，可推送到 Slack / 邮件 / Jira
```

## 十四、边界与异常场景验收

### AC-54：Namespace 销毁时 PV 保留

```
Given 实验实例销毁（Namespace 删除）
When PV 进入 Terminating
Then 延迟 24 小时后才真正删除
And 24 小时内教师可通过后端 API 下载链数据用于取证
```

### AC-55：CTF 队伍 Namespace 过期

```
Given CTF 竞赛结束
When 7 天后
Then 所有 ctf-{comp_id}-team-* Namespace 自动销毁
And 裁判 Namespace 保留 30 天
```

### AC-56：镜像拉取失败重试

```
Given Registry 短暂不可用
When DaemonSet image-prepuller 拉取失败
Then 指数退避重试 3 次
And 3 次仍失败标记为 FAIL
And 发送告警通知运维
```

### AC-57：磁盘告急

```
Given 后续接入 Prometheus 等监控系统且 PV 使用率 > 85%
When 触发告警
Then 应通过模块07 发送通知给管理员
And 管理员可通过运维仪表盘查看使用详情
```

### AC-58：超限资源申请

```
Given 实验模板声明的资源总和超过课程配额
When 学生启动实验
Then K8s 调度失败（Pod Pending）
And 后端检测到超限，返回明确错误信息（不是 500）
And 通知学生和教师
```

## 十五、灾备与数据保护验收

### AC-59：数据库备份有效性

```
Given 生产环境存在每日备份文件
When 执行恢复演练
Then 从备份恢复后，数据一致性校验通过
And 业务功能全部可用
```

### AC-60：MinIO 数据跨区域复制

```
Given 生产环境启用了跨区域复制策略
When 主区域写入对象
Then 备区域 5 分钟内可读到相同对象
```

### AC-61：离线镜像包校验

```
Given 学校运维下载离线镜像包
When 执行 sha256sum -c lenschain-phase1-v1.0.0.tar.sha256
Then 校验通过
And 任何一字节篡改都会被检测到
```

## 十六、开放问题

1. 可用性 SLO（99.9% vs 99.95%）需要和业务方对齐，不同数字对部署成本影响巨大
