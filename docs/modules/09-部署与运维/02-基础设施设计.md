## 一、文档目的

本文档是部署与运维模块的基础设施设计文档，聚焦于 **端口分配、网络拓扑、Registry 架构、Namespace 规划、存储设计、Secret 管理、RBAC 与 Network Policy** 等基础设施层决策。与 `01-功能需求说明.md` 的模块边界与功能清单互补，本文档回答"平台运行所需的基础设施长什么样"。

### 设计原则

- **端口零冲突**：平台服务、中间件、SimEngine、实验环境端口严格划分，互不干扰
- **最小权限**：所有 ServiceAccount / Network Policy 遵循最小权限原则，默认拒绝
- **环境隔离**：平台服务、运维、实验、CTF 分别独立 Namespace，生命周期清晰
- **可观测**：所有基础设施组件暴露健康探针与指标端点
- **可回滚**：镜像版本、K8s manifest、数据库迁移均可回滚到上一稳定版本

## 二、端口分配总表

### 2.1 平台服务宿主机端口（对外暴露）

| 服务 | 端口 | 协议 | 访问来源 | 说明 |
|------|------|------|----------|------|
| 前端 Next.js | 3000 | HTTP | 浏览器 / Ingress | 生产环境经 Ingress 接入 443 |
| 后端 Go API | 8080 | HTTP + WebSocket | 前端 / Ingress | REST API + 实时 WS |
| SimEngine Core gRPC | 50051 | gRPC (HTTP/2) | 后端 | 内部调用，不对外 |
| SimEngine Core WS | 50052 | WebSocket | 前端渲染器 | 仿真画面 / 交互 |

### 2.2 中间件端口

| 服务 | 端口 | 协议 | 访问来源 | 说明 |
|------|------|------|----------|------|
| PostgreSQL | 5432 | TCP | 后端 / 迁移工具 | 平台数据主库 |
| Redis | 6379 | TCP | 后端 / 通知服务 | 缓存 + 会话 + 发布订阅 |
| MinIO API | 9000 | HTTP (S3) | 后端 / 前端预签名直传 | 对象存储 API |
| MinIO Console | 9001 | HTTP | 管理员 | 运维管理界面 |
| NATS Client | 4222 | TCP | 后端 / 模块07 | 消息队列客户端端口 |
| NATS Monitoring | 8222 | HTTP | 运维 / Prometheus | NATS 健康与监控 |

### 2.3 运维组件端口

| 服务 | 端口 | 协议 | 访问来源 | 说明 |
|------|------|------|----------|------|
| Registry (Harbor Core) | 443 | HTTPS | K8s 节点 / CI | 镜像仓库 |
| Registry Notary | 4443 | HTTPS | CI | 镜像签名验证 |
| Prometheus | 9090 | HTTP | Grafana / 运维 | 指标采集（可选） |
| Grafana | 3000 (容器内) | HTTP | 运维 | 不占用宿主机 3000，经 Ingress 暴露 |

> Prometheus / Grafana 仅在生产环境部署；开发环境不启用。Grafana 容器内 3000 端口不与前端 Next.js 的宿主机 3000 冲突，因为二者部署在不同 K8s Pod 内，且 Grafana 仅经 Ingress 的独立子路径对外访问。

### 2.4 实验 Namespace 内端口

实验容器端口 **仅在实例 Namespace 内生效**（模块04 BR-67），不占用宿主机端口：

- 教师在 `default_ports` 配置的端口（如 geth 的 8545）通过 K8s Service 暴露给 **同 Namespace 内其他容器**
- 学生访问实验容器内服务通过 **后端代理**：后端使用 `kubectl exec` / `kubectl port-forward` / SPDY WebSocket 隧道对学生浏览器提供 HTTP / WS 通道
- 实验 Namespace **不创建 NodePort / LoadBalancer**，避免占用宿主机端口或暴露到公网

| 25 / 465 / 587 | SMTP |

## 三、网络拓扑

### 3.1 三层网络划分

```
┌─────────────────────────────────────────────────────────────┐
│                    公网（Internet）                          │
└──────────────────────────┬──────────────────────────────────┘
                           │ 443 / 80
                           ▼
┌─────────────────────────────────────────────────────────────┐
│              边界层（Ingress Controller）                    │
│  TLS 终止 / 路由分发 / 限流 / IP 白名单                        │
└──────────┬───────────────┬──────────────┬───────────────────┘
           │               │              │
           ▼               ▼              ▼
  ┌────────────────┐ ┌──────────────┐ ┌──────────────────┐
  │  前端 Next.js   │ │  后端 API    │ │  MinIO Console   │
  │  Namespace:    │ │  Namespace:  │ │  Namespace:      │
  │  lenschain     │ │  lenschain   │ │  lenschain       │
  └────────────────┘ └──────┬───────┘ └──────────────────┘
                            │
                            ├─ 东西向流量（ClusterIP Service）
                            ▼
┌─────────────────────────────────────────────────────────────┐
│            内部服务层（Namespace: lenschain）                 │
│  PostgreSQL / Redis / MinIO / NATS / SimEngine Core          │
└──────────────────────────┬──────────────────────────────────┘
                           │ K8s API 调用
                           ▼
┌─────────────────────────────────────────────────────────────┐
│       实验与竞赛层（Namespace: exp-* / ctf-*）                │
│  每个实验 / 竞赛队伍独立 Namespace                             │
│  默认跨 Namespace 禁止通信（Network Policy）                   │
│  允许出站：Registry（镜像拉取）+ 后端 API（Collector 上报）      │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 流量分类

| 流量类型 | 路径 | 加密 | 说明 |
|----------|------|------|------|
| 南北向-用户访问前端 | 浏览器 → Ingress → 前端 Pod | TLS (443) | Let's Encrypt / 学校证书 |
| 南北向-用户访问 API | 浏览器 → Ingress → 后端 Pod | TLS (443) | 同上，路径 `/api/v1/*` |
| 南北向-WebSocket | 浏览器 → Ingress → 后端 Pod | WSS | 复用 443，路径 `/ws/*` `/ctf/ws` 等 |
| 东西向-平台内 | 前端 Pod → 后端 Pod → DB/Redis/MinIO/NATS | 可选 mTLS | ClusterIP Service，生产环境建议启用 mTLS |
| 东西向-SimEngine | 后端 Pod → SimEngine Core Pod | gRPC（可选 TLS） | 50051 |
| 东西向-SimEngine WS | 前端 Pod（或直连） → SimEngine Core Pod | WS/WSS | 50052，生产环境经后端反代 |
| 实验出站-Registry | exp-* Namespace → Registry | HTTPS | 仅允许访问 Registry 域名 |
| 实验出站-Collector | exp-* Pod sidecar → lenschain Namespace 后端 | HTTP | 上报采集数据 |
| 实验出站-公网 | exp-* Pod → Internet | HTTP/HTTPS | 默认允许，可按学校策略关闭 |

### 3.3 Ingress 路由规则

```yaml
# 生产环境 Ingress 路由（示例）
rules:
  - host: app.lianjing.com
    paths:
      - path: /
        backend: frontend:3000
  - host: api.lianjing.com
    paths:
      - path: /
        backend: backend:8080
  - host: files.lianjing.com
    paths:
      - path: /
        backend: minio:9000                   # S3 预签名 URL 外部访问
  - host: console.lianjing.com
    paths:
      - path: /
        backend: minio-console:9001           # 仅限运维 IP 白名单
  - host: grafana.lianjing.com
    paths:
      - path: /
        backend: grafana:3000                 # 仅限运维 IP 白名单
```

学校私有化部署一般只暴露 `app.*` 和 `api.*`；`console.*` / `grafana.*` 仅对内网或跳板机开放。

## 四、Namespace 规划

### 4.1 Namespace 全表

| Namespace | 用途 | 生命周期 | 创建方式 | 谁能访问 |
|-----------|------|----------|----------|---------|
| `lenschain` | 平台自身服务 | 永久 | 部署时静态创建 | 平台服务互访 |
| `lenschain-system` | 平台运维组件 | 永久 | 部署时静态创建 | 仅集群管理员 |
| `ingress-system` | Ingress Controller | 永久 | 部署时静态创建 | 边界层 |
| `cert-manager` | 证书管理（如用 cert-manager） | 永久 | 部署时静态创建 | 系统 |
| `monitoring` | Prometheus / Grafana / Alertmanager | 永久 | 部署时静态创建 | 仅运维 |
| `exp-{instance_id}` | 单个实验实例 | 实验期间（启动 → 销毁） | 后端动态创建 | 该实例关联的学生 + 教师（只读） |
| `ctf-{competition_id}-team-{team_id}` | CTF 攻防赛单支队伍 | 竞赛期间 | 后端动态创建 | 该队伍成员 |
| `ctf-{competition_id}-judge` | CTF 攻防赛裁判链 | 竞赛期间 | 后端动态创建 | 裁判服务 |
| `ctf-{competition_id}-challenge-{challenge_id}` | CTF 解题赛题目环境 | 竞赛期间 | 后端动态创建 | 参赛队伍（只读） |

### 4.2 Namespace 命名规则

- 全部小写 + 连字符
- 使用雪花 ID 作为后缀（字符串形式，避免 JS 精度问题）
- 最大长度 63 字符（K8s 限制）
- 示例：`exp-1780000000600001`、`ctf-1780000000700001-team-1780000000800001`

### 4.3 Namespace 标签

所有平台管理的 Namespace 必须带以下标签，用于 Network Policy 匹配和生命周期管理：

```yaml
labels:
  app.kubernetes.io/managed-by: lenschain
  lenschain.io/namespace-type: experiment  # platform / system / experiment / ctf-team / ctf-judge / ctf-challenge
  lenschain.io/instance-id: "1780000000600001"
  lenschain.io/school-id: "1780000000000001"
  lenschain.io/created-at: "2026-04-24T14:00:00Z"
```

### 4.4 Namespace 销毁策略

| Namespace 类型 | 销毁时机 | 数据清理 |
|----------------|----------|----------|
| `exp-*` | 学生主动销毁 / 实验超时 / 教师强制回收 | PV 延迟 24h 清理（教师取证窗口），之后删除 |
| `ctf-*-team-*` | 竞赛结束后 7 天 | PV 立即清理（无取证需求） |
| `ctf-*-judge` | 竞赛结束后 7 天 | PV 保留 30 天（审计） |
| `ctf-*-challenge-*` | 竞赛结束后 | PV 立即清理 |

## 五、Registry 架构

### 5.1 Registry 选型

**使用 Harbor** 作为私有 Registry：

- 支持多租户（Project 划分，对应 `lenschain/` `chain-nodes/` 等命名空间）
- 内置镜像签名（Notary）、漏洞扫描（Trivy）、配额管理
- 支持离线镜像同步（学校私有化部署场景）
- 开源免费，社区活跃

### 5.2 Registry 部署拓扑

**公网统一运营：**

```
registry.lianjing.com（公网 / Harbor 集群）
├── 2 个 Harbor Core 节点（HA）
├── PostgreSQL（独立实例）
├── Redis（独立实例）
├── 对象存储后端（MinIO 集群 / 云厂商 OSS）
└── Notary 签名服务
```

**学校私有化部署：**

```
registry.{school}.internal（学校内网 / Harbor 单节点）
├── 单节点 Harbor
├── 内嵌 PostgreSQL
├── 内嵌 Redis
└── 本地磁盘 / NFS 存储

镜像来源：
  - 首次部署通过离线镜像包（docker save → load）
  - 后续通过 Harbor Replication 从公网 registry 定期同步
```

### 5.3 Project（命名空间）划分

| Project | 用途 | 访问权限 |
|---------|------|---------|
| `lenschain` | 平台服务镜像 + Collector Agent | 公开拉取 / CI 推送 |
| `chain-nodes` | 链节点镜像 | 公开拉取 / 管理员推送 |
| `middleware` | 中间件镜像 | 公开拉取 / 管理员推送 |
| `tools` | 工具镜像 | 公开拉取 / 管理员推送 |
| `base` | 开发基础镜像 | 公开拉取 / 管理员推送 |
| `scenarios` | 仿真场景容器镜像 | 公开拉取 / 教师推送（审核后） |

### 5.4 镜像命名规范

**完整格式：** `{registry}/{project}/{image_name}:{tag}`

**平台服务镜像：**
```
registry.lianjing.com/lenschain/backend:v1.0.0
registry.lianjing.com/lenschain/frontend:v1.0.0
registry.lianjing.com/lenschain/sim-engine-core:v1.0.0
registry.lianjing.com/lenschain/scenario-base:v1.0.0
registry.lianjing.com/lenschain/collector-agent-ethereum:v1.0.0
registry.lianjing.com/lenschain/collector-agent-fabric:v1.0.0
registry.lianjing.com/lenschain/collector-agent-chainmaker:v1.0.0
registry.lianjing.com/lenschain/collector-agent-fisco:v1.0.0
```

> 与 `backend/configs/config.example.yaml` 的 `collector_image_template: "registry.lianjing.com/lenschain/collector-agent-%s:v1.0.0"` 对齐：`%s` 填入链适配器编码 `ethereum` / `fabric` / `chainmaker` / `fisco`。四个采集镜像统一放在 `lenschain` Project 下，通过镜像名后缀区分生态，避免为 Collector 单独开 Project。

**实验环境镜像：**
```
registry.lianjing.com/chain-nodes/geth:1.14
registry.lianjing.com/chain-nodes/geth:1.13
registry.lianjing.com/chain-nodes/fabric-peer:2.5
registry.lianjing.com/middleware/couchdb:3.3
registry.lianjing.com/tools/remix-ide:0.35.0
registry.lianjing.com/tools/judge-service:v1.0.0
registry.lianjing.com/base/solidity-dev:v1.0.0
registry.lianjing.com/base/ctf-web:v1.0.0
registry.lianjing.com/scenarios/pow-mining:v1.0.0
```

### 5.5 标签策略

| 标签类型 | 格式 | 用途 |
|----------|------|------|
| 语义化版本 | `v{MAJOR}.{MINOR}.{PATCH}` | 平台服务镜像正式版 |
| 上游版本 | `1.14` / `2.5` / `3.3` | 实验镜像跟随上游软件版本 |
| Git Commit SHA | `sha-a1b2c3d4` | CI 每次构建 |
| 环境标签 | `staging` / `prod` | 环境专用指针（可变） |
| 不可变标签 | Git Tag | 发版 |

**禁止：**
- 平台服务镜像禁止只使用 `latest` 标签（生产环境必须指定明确版本）
- 生产镜像禁止覆盖已发布标签

**必须：**
- CI 构建时同时打语义化版本标签和 Git Commit SHA 标签（对应 BR-09）
- 每次发版打 Git Tag，触发 Release CI

### 5.6 镜像签名（Cosign）

- 平台服务生产镜像必须经 CI 使用 Cosign 签名
- K8s 部署时通过 Kyverno / Cosign admission controller 验证签名
- 签名私钥存储在 CI 的 Secret Manager，公钥随 K8s overlay 分发

### 5.7 漏洞扫描

- Harbor 内置 Trivy 对所有推送镜像自动扫描
- 平台自研镜像扫出 **Critical** 漏洞阻止推送
- 实验环境镜像扫出漏洞记录但不阻止（部分 CTF / 攻防镜像本身就包含已知漏洞）

## 六、存储设计

### 6.1 存储类型矩阵

| 数据类型 | 存储类型 | 后端 | 生命周期 | 备份策略 |
|----------|----------|------|---------|---------|
| 业务数据（PG） | PV + StatefulSet | 云盘 / Ceph RBD / 本地 SSD | 永久 | 每日全量 + 每 6 小时增量 |
| 缓存（Redis） | PV + StatefulSet | 云盘 / 本地 SSD | 永久（可丢失） | 不备份，重建即可 |
| 对象存储（MinIO） | PV + StatefulSet | 云盘 / Ceph RBD | 永久 | 每日 Bucket 级快照 |
| 消息队列（NATS） | PV + StatefulSet（JetStream） | 云盘 | 永久 | 不备份（幂等消费） |
| 平台日志 | stdout + Loki / ELK | 集群内 | 7-30 天 | 不备份 |
| 实验链数据 | PV（实验 Namespace 内） | 云盘 / 本地 SSD | 实验期间 | 不备份（教师可下载） |
| 快照数据 | MinIO Bucket | 同 MinIO | 30 天 | 同 MinIO |
| Registry 镜像 | 独立 MinIO / 云厂商 OSS | 对象存储 | 永久 | 跨区域复制 |

### 6.2 StorageClass 约定

| StorageClass | 用途 | 特性 |
|--------------|------|------|
| `lenschain-ssd` | 数据库 / Redis / MinIO | 高 IOPS，云厂商 premium SSD |
| `lenschain-standard` | 实验容器数据 | 标准性能，成本优化 |
| `lenschain-local` | NATS JetStream | 本地 SSD，低延迟 |

学校私有化部署可按实际存储后端调整（NFS / Ceph / Longhorn）。

### 6.3 PV/PVC 容量规划（参考）

| 组件 | 单副本容量 | 生产多副本 | 说明 |
|------|-----------|-----------|------|
| PostgreSQL | 100 Gi | 主 + 2 从 | 业务数据主库 |
| Redis | 10 Gi | 主从 + 哨兵 | 缓存 |
| MinIO | 500 Gi | 4 节点分布式 | 对象存储 |
| NATS JetStream | 20 Gi | 3 副本 | 消息持久化 |
| Registry（Harbor） | 2 Ti | 独立部署 | 镜像仓库 |
| 实验 Namespace PV | 10-50 Gi | N/A | 按镜像 default_volumes 自动创建 |

### 6.4 MinIO Bucket 划分

| Bucket | 用途 | 访问模式 | 生命周期 |
|--------|------|----------|---------|
| `lenschain-attachments` | 课程附件、作业附件 | 私有 + 预签名 URL | 永久 |
| `lenschain-reports` | 实验报告 / 作业报告 | 私有 + 预签名 URL | 永久 |
| `lenschain-snapshots` | 实验快照 / CTF 环境快照 | 私有 + 内部访问 | 30 天 |
| `lenschain-images` | 教师自定义镜像临时上传 | 私有 | 审核后清理 |
| `lenschain-exports` | 数据导出（成绩 / 审计日志） | 私有 + 短期预签名 | 24 小时 |
| `lenschain-backups` | 数据库备份 | 私有 + 加密 | 30 天滚动 |

## 七、Secret 与配置管理

### 7.1 Secret 分类

| Secret | 存储位置 | 轮换周期 | 备注 |
|--------|----------|----------|------|
| JWT 签名密钥（Access / Refresh） | K8s Secret | 90 天 | 对应 `config.yaml` 的 `jwt.access_secret` / `refresh_secret` |
| 数据库密码 | K8s Secret | 90 天 | `database.password` |
| Redis 密码 | K8s Secret（如启用） | 90 天 | `redis.password` |
| MinIO Access/Secret Key | K8s Secret | 180 天 | `minio.access_key` / `secret_key` |
| Registry 拉取凭证 | K8s imagePullSecret | 180 天 | 所有 Namespace 通过 ServiceAccount 挂载 |
| 短信服务 AK/SK | K8s Secret | 按供应商要求 | `sms.access_key` / `secret_key` |
| TLS 证书 | K8s Secret（tls 类型）| 自动（cert-manager） | Ingress 使用 |
| Cosign 签名公钥 | K8s ConfigMap | 发版时更新 | 公开 |
| Cosign 签名私钥 | CI Secret Manager | 90 天 | 仅 CI 使用，不进集群 |

### 7.2 Secret 加密存储

- **生产环境**：使用 **Sealed Secrets** 或云厂商 KMS
  - Sealed Secrets：加密后的 Secret 可安全提交 Git，解密密钥仅存在于集群内
  - KMS 模式：K8s 启用 EncryptionConfig，etcd 中的 Secret 由 KMS 加密
- **staging**：可使用明文 Secret + K8s 默认 base64（安全性足够）
- **开发**：`.env.local` 或 `config.yaml`，不入 Git

### 7.3 ConfigMap 使用

| ConfigMap | 内容 | 挂载方式 |
|-----------|------|---------|
| `backend-config` | 后端 `config.yaml`（非 secret 部分） | 挂载到容器 `/app/configs/config.yaml` |
| `frontend-env` | 前端 `NEXT_PUBLIC_*` | 构建时注入（不挂载） |
| `sim-engine-config` | SimEngine 配置 | 同后端 |
| `image-manifest` | 镜像清单 YAML | 被 image-prepuller DaemonSet 挂载 |
| `cors-config` | CORS 白名单 | 后端读取 |

**约定：** Secret 与 ConfigMap 严格分离，凡涉及密钥 / 密码 / 证书的字段不得放入 ConfigMap。

## 八、RBAC 设计

### 8.1 ServiceAccount 清单

| ServiceAccount | 所属 Namespace | 用途 |
|----------------|---------------|------|
| `lenschain-backend` | `lenschain` | 后端服务，需要创建 / 管理 `exp-*` 和 `ctf-*` Namespace |
| `lenschain-sim-engine` | `lenschain` | SimEngine Core，需要创建 / 管理场景算法容器 Pod |
| `lenschain-image-prepuller` | `lenschain-system` | DaemonSet，只读拉取镜像清单 |
| `lenschain-backup` | `lenschain-system` | 备份 Job，读取 PVC 数据 |
| `lenschain-monitoring` | `monitoring` | Prometheus，只读所有 Namespace 指标 |
| `default`（每个 exp-* / ctf-*） | `exp-*` / `ctf-*` | 实验容器本身，无集群权限 |

### 8.2 ClusterRole 定义

**`lenschain-experiment-operator`**（赋予 `lenschain-backend`）：

```yaml
rules:
  # Namespace 生命周期
  - apiGroups: [""]
    resources: ["namespaces"]
    verbs: ["create", "get", "list", "watch", "delete", "patch"]
  # Pod / Service / ConfigMap / Secret / PVC（仅限 exp-* / ctf-* Namespace，通过 Namespace 选择器限定）
  - apiGroups: [""]
    resources: ["pods", "services", "configmaps", "secrets", "persistentvolumeclaims", "events"]
    verbs: ["*"]
  # Deployment / StatefulSet
  - apiGroups: ["apps"]
    resources: ["deployments", "statefulsets", "replicasets"]
    verbs: ["*"]
  # Network Policy / Ingress（实验内组网）
  - apiGroups: ["networking.k8s.io"]
    resources: ["networkpolicies", "ingresses"]
    verbs: ["*"]
  # exec / log（终端功能）
  - apiGroups: [""]
    resources: ["pods/exec", "pods/log", "pods/portforward"]
    verbs: ["create", "get"]
  # 镜像拉取监控
  - apiGroups: [""]
    resources: ["nodes"]
    verbs: ["get", "list", "watch"]
```

**`lenschain-sim-engine-operator`**（赋予 `lenschain-sim-engine`）：

```yaml
rules:
  - apiGroups: [""]
    resources: ["pods", "services", "configmaps"]
    verbs: ["*"]
    # 通过 Namespace 选择器限定在 exp-* 和专用场景 Namespace
```

**`lenschain-image-prepuller-role`**（赋予 `lenschain-image-prepuller`）：

```yaml
rules:
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get", "watch"]
    resourceNames: ["image-manifest"]
```

### 8.3 最小权限原则

- 业务代码**禁止**直接使用 `cluster-admin` 或 `system:*` 角色
- 通过 Namespace 标签 + RoleBinding 限制 ServiceAccount 作用域
- ClusterRoleBinding 仅用于必须跨 Namespace 的操作（如 Namespace 创建）
- 定期审计 ClusterRole / RoleBinding，清理冗余权限

## 九、Network Policy 设计

### 9.1 默认策略：拒绝所有

生产环境所有 Namespace 默认应用 `deny-all`，再按需放行：

```yaml
# 在每个 exp-* / ctf-* Namespace 创建时应用
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
  namespace: exp-{instance_id}
spec:
  podSelector: {}
  policyTypes: ["Ingress", "Egress"]
```

### 9.2 实验 Namespace 放行规则

**入站：**
- 允许同 Namespace 内 Pod 互访（业务组网需要）
- 允许 `lenschain` Namespace 后端 Pod 访问（exec / log / port-forward）
- 其他一律拒绝

**出站：**
- 允许访问 Kube DNS（53 TCP/UDP）
- 允许访问 Registry（`lenschain.io/role=registry` 标签的 Pod / 外部 Registry IP 白名单）
- 允许访问 `lenschain` Namespace 后端 API（Collector 上报）
- 允许访问公网 HTTP/HTTPS（80 / 443），用于实验依赖下载；可按学校策略关闭
- 禁止访问其他 `exp-*` / `ctf-*` Namespace（横向移动防护）
- 禁止访问 `lenschain` Namespace 的数据库 / Redis / MinIO（隔离业务数据）

### 9.3 CTF 攻防赛专用策略（多链对抗拓扑）

**拓扑说明：** 攻防赛不是"每队独立玩"，而是"每队自己一条链作为防守阵地 + 裁判链统一结算"。队伍 A 对队伍 B 的攻击由 `judge-service` 在 B 的队伍链上代理执行，攻击结果上链裁判链；队伍之间**不直接互相访问**。

```
                    ┌─────────────────────┐
                    │   裁判链 Namespace   │  ← 中立仲裁
                    │  ctf-{comp}-judge    │    Token 总账 / 回合合约
                    │  judge-service       │    攻击代理 / 结算
                    │  patch-verifier      │    补丁验证
                    └──────────▲──────────┘
                               │
           ┌───────────────────┼───────────────────┐
           │                   │                   │
    ┌──────┴──────┐     ┌──────┴──────┐     ┌──────┴──────┐
    │ team-A 链   │     │ team-B 链   │     │ team-C 链   │
    │ ganache     │     │ ganache     │     │ ganache     │
    │ 漏洞合约    │     │ 漏洞合约    │     │ 漏洞合约    │
    └─────────────┘     └─────────────┘     └─────────────┘
           ↑                   ↑                   ↑
      A 队成员操作         B 队成员操作        C 队成员操作
      通过后端 API        通过后端 API        通过后端 API
      提交攻击 B/C        提交攻击 A/C        提交攻击 A/B
```

**队伍 Namespace 之间（ctf-{comp}-team-* ↔ ctf-{comp}-team-*）：**
- **默认完全禁止互访**：队伍 A 的 Pod 不能直接连接队伍 B 的链 RPC，不能 exec、不能 curl、不能通过 Service 名称解析
- 这保证攻击必须通过平台后端 + `judge-service` 代理，队伍之间看不到对方的 exploit 代码

**队伍 Namespace → 裁判 Namespace（ctf-{comp}-team-* → ctf-{comp}-judge）：**
- **禁止直接访问**：队伍 Pod 不能直接连裁判链 RPC（防止篡改结算逻辑）
- 所有与裁判交互（提交攻击、提交补丁、查询 Token 余额）**必须**通过 `lenschain` Namespace 的后端 API 中转
- 裁判链对队伍**只暴露只读数据**（排行榜、公告），写入路径全部经后端 API 鉴权

**裁判 Namespace → 队伍 Namespace（ctf-{comp}-judge → ctf-{comp}-team-*）：**
- **允许** `judge-service` 访问任意队伍 Namespace 的链 RPC 端口（8545 等），用于：
  - 代理执行 A 队提交的 exploit 到 B 队的队伍链
  - 执行补丁验证（部署补丁到目标队伍链，跑功能测试和漏洞断言）
  - 定期采集链状态（Token 余额、合约状态、交易记录）
- **允许** `patch-verifier` 访问队伍 Namespace 部署新合约
- 这是攻防赛唯一允许的跨 Namespace 访问路径

**队伍 Namespace → 外网：**
- 允许出站 Registry（镜像拉取）、DNS、`lenschain` 后端 API
- 禁止访问其他 CTF 竞赛的 Namespace（跨竞赛隔离）
- 公网访问按竞赛配置决定（真实漏洞题可能需要；封闭赛禁止）

**Namespace 标签约定：**

```yaml
# 队伍 Namespace
labels:
  lenschain.io/namespace-type: ctf-team
  lenschain.io/competition-id: "1780000000700001"
  lenschain.io/team-id: "1780000000800001"

# 裁判 Namespace
labels:
  lenschain.io/namespace-type: ctf-judge
  lenschain.io/competition-id: "1780000000700001"
```

Network Policy 使用 `namespaceSelector` + `lenschain.io/competition-id` 标签匹配，确保同一竞赛内的规则生效、不同竞赛间天然隔离。

### 9.4 共享基础设施模式（模块04 §6.4）
- 共享 Namespace `exp-shared-{template_id}-{lesson_id}` 由模块04运行时按“模板 + 课时”创建并复用
- 学生实例 Namespace 使用独立实例命名空间，附加以下标签：
  - `lenschain.io/namespace-type=experiment-student`
  - `lenschain.io/template-id`
  - `lenschain.io/lesson-id`
  - `lenschain.io/shared-namespace`
- 共享 Namespace 附加以下标签：
  - `lenschain.io/namespace-type=experiment-shared`
  - `lenschain.io/template-id`
  - `lenschain.io/lesson-id`
- Network Policy 要求：
  - 共享 Namespace 仅允许同 `template-id + lesson-id` 的学生实例 Namespace 访问
  - 学生实例 Namespace 允许访问所属共享 Namespace
  - 学生 Namespace 间禁止互访
  - 非所属课时/模板的实验 Namespace 不得访问共享基础设施

## 十、镜像预拉取基础设施

### 10.1 预拉取目标

| 镜像类型 | 触发条件 | 节点范围 |
|----------|----------|---------|
| 平台服务镜像 | 部署 / 升级 | 所有节点 |
| Phase 1 实验镜像 | 部署 | 所有节点 |
| Phase 2/3 实验镜像 | 按需（学校选择） | 按标签选择节点 |
| 教师自定义镜像 | 审核通过 | 所有实验节点 |
| 场景算法容器 | 场景发布 | 所有节点 |

### 10.2 DaemonSet 设计

- DaemonSet `image-prepuller` 运行在 `lenschain-system` Namespace
- 挂载 `image-manifest` ConfigMap
- 通过 `crictl pull` / `docker pull` 批量拉取
- 并发控制：默认单节点并发 3 个拉取任务，避免带宽打满
- 状态上报：每次拉取成功后向后端 `/internal/image-prepull/report` 上报
- 失败重试：指数退避，最多 3 次；3 次失败后标记为 FAIL，需要人工处理

### 10.3 节点亲和性（可选）

通过节点标签控制镜像分发范围，避免小节点被大镜像占满：

```yaml
# 给某些节点打标签
kubectl label node node-1 lenschain.io/exp-tier=heavy
kubectl label node node-2 lenschain.io/exp-tier=light

# 重型镜像（solana-validator 等）只拉取到 heavy 节点
# 轻型镜像（ganache / code-server 等）所有节点都拉取
```

## 十一、可观测性基础设施

### 11.1 Metrics

- **Prometheus** 部署在 `monitoring` Namespace
- 所有平台服务（后端 / 前端 / SimEngine）必须暴露 `/metrics` 端点（Prometheus 格式）
- 标准指标：HTTP QPS / 延迟 / 错误率、goroutine 数量、GC 时间、DB 连接池、Redis 命中率
- 业务指标：实验实例创建数、实验启动时间、CTF 提交次数等

### 11.2 Logs

- 所有容器日志输出到 stdout / stderr（JSON 格式）
- K8s 节点上通过 Fluent Bit / Promtail 采集到 Loki / ELK
- 日志保留 30 天（生产）/ 7 天（staging）/ 1 天（dev）
- 审计日志不走此通道，由业务代码直写 `operation_logs` / `login_logs` 表（模块01）

### 11.3 Tracing（可选）

- 生产环境启用 OpenTelemetry SDK
- 所有跨服务调用（前 → 后端、后端 → SimEngine、后端 → K8s API）生成 Span
- 后端发送到 Jaeger / Tempo

### 11.4 Healthcheck

所有平台服务必须实现：

| 端点 | 用途 | K8s 探针 |
|------|------|----------|
| `/healthz` | 存活探针（进程可响应） | livenessProbe |
| `/readyz` | 就绪探针（依赖就绪，可接流量） | readinessProbe |
| `/startupz` | 启动探针（适用于启动慢的服务） | startupProbe |

## 十二、时区与字符集

### 12.1 时区

- **所有容器统一 UTC**（包括数据库）
- 所有时间以 UTC 存储
- 前端按浏览器时区显示
- K8s Pod 显式设置 `TZ=UTC` 环境变量
- Dockerfile 中安装 tzdata 但运行时 TZ=UTC

### 12.2 字符集

- **UTF-8**
- PostgreSQL 创建数据库时强制 `ENCODING='UTF8' LC_COLLATE='C'`
- 所有文件编码 UTF-8
- CSV 导出时加 BOM（便于 Excel 识别中文）
